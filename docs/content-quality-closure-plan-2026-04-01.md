## 2. 最值得优先收口的 5 个问题（当前阶段核心）

1. **`published > draft` 不成立（或不可证明）**
   - 没有“发布前后文本增益”硬性验收与 Gate 阻断。

2. **seed 仅“注入”未“消费验收”**
   - 目前更像把 seed 带入 input，而不是检查正文是否自然兑现。

3. **Gate 偏结构合规，缺内容质量目标**
   - 长度、空值、蓝图粗对齐有；信息节奏、人物情绪层次、线索推进质量弱。

4. **ChangeSet 偏状态补丁，不驱动正文修订**
   - ChangeSet 提取的是 Canon patch，不是“正文改进计划”；对 publish 质量增益贡献弱。

5. **summary/derived 与正文一致性缺少反向校验**
   - summary 能产出 seed，但缺“summary 是否忠实正文、是否误导下一章”的系统检查。

## 3. 小说内容质量验收表（最小可执行版）

> 评分建议统一 1~5（1=严重不达标，3=可发布边缘，5=明显优秀）。

| 维度 | 验收目标 | 观察点 | 常见失败表现 | 评分方式（1~5） | 自动化可行性 |
|---|---|---|---|---|---|
| 跨章连续性 | 本章显式承接上一章事实与情绪后果 | 是否出现上一章关键事件后果；是否避免“当章失忆” | 重复开场铺垫；上一章冲突像没发生 | 1=断裂；3=有承接但弱；5=承接自然且推进 | **半自动+人工** |
| Seed 消费质量 | `next_chapter_seed` 被自然转化为情节推进 | seed 关键要素在本章“行动/冲突/决策”中落地 | 仅口头提到 seed；生硬复读 seed | 1=未消费；3=消费但生硬；5=自然且有增量 | **半自动+人工** |
| 信息揭露节奏 | 线索揭露有层次，不抢跑也不拖沓 | 每章新增信息量；疑点保留与回收比例 | 信息一次抖完；硬解释台词 | 1=节奏失衡；3=可读但直给；5=层次清晰 | **人工为主** |
| 人物推进与情绪真实度 | 主配角动机、反应、代价可感知 | 关键决策前后是否有心理/关系波动 | 角色功能化；情绪模板句复用 | 1=纸片人；3=有动机但薄；5=动机-行为闭环 | **人工为主** |
| 世界观/规则一致性 | 不违背既有 Canon 与规则包 | 人设、规则、时间线是否冲突 | 设定打架；硬改规则无解释 | 1=冲突明显；3=小冲突；5=一致且有解释 | **自动化优先** |

| Published 质量增益 | published 相比 draft 有可见提升 | 是否完成“修复问题清单”；可读性是否提升 | draft/published 几乎一致 | 1=无增益；3=局部润色；5=结构与表达双提升 | **自动+人工联合** |

## 4. Gate / ChangeSet / 人工验收分工建议

## 4.1 适合进入 Gate（先做可阻断项）

1. **连续性硬检查（阻断）**
   - 检查上一章 `summary + next_chapter_seed` 关键短语在本章是否被“行动化消费”（不是仅字符串出现）。

2. **世界观/规则冲突检查（阻断）**
   - 延续现有 canon/rulepack 方向，增强实体一致性（人物状态、规则冲突、时间线冲突）。

3. **summary-正文偏离检查（警告→阻断可配置）**
   - 发布前比对 summary 草案与正文关键事件覆盖率。

4. **published 增益检查（阻断）**
   - 若 draft/published 文本相似度过高 + 未解决关键 Gate issue，则禁止发布。

## 4.2 适合进入 ChangeSet（“怎么改正文”）

1. **信息揭露过直 → 变更指令：延迟揭露、拆分到动作与反应中**
2. **人物情绪单薄 → 变更指令：补一层“动机-犹豫-代价”链路**
3. **配角功能化 → 变更指令：为配角补独立诉求与反作用**
4. **线索重复铺垫 → 变更指令：压缩复述段，新增增量事实**
5. **场景无张力 → 变更指令：补目标阻力和即时后果**

> 关键点：ChangeSet 下一阶段应扩展“内容修订提案”，不仅是 Canon patch。

## 4.3 当前阶段不适合自动处理（保留人工）

- 高阶文学性（句法审美、风格独创性）
- 叙事气质偏好（“冷峻/抒情/克制”）
- 主观审美争议项（“这句是否更有味道”）

## 6. 最该先做的第一刀（只能选一个）

**第一刀：上线“Published 质量增益闸门（Draft→Published Delta Gate）”。**

### 理由

1. 它直接解决“已发布但没变好”的核心失败模式。  
2. 它是所有后续内容优化（seed 消费、信息节奏、人物补强）的落地前提：没有增益闸门，优化建议会继续停留在“可选项”。  
3. 实施粒度小、可快速闭环：先基于相似度 + 关键问题修复数做最小规则，再迭代更细指标。

### 下一轮可直接落地的小任务定义

- 在发布前新增一个 `quality_delta_check`：
  - 输入：draft 文本、候选 published 文本、Gate issue 列表
  - 规则：
    1) 文本相似度高于阈值且
    2) 关键 issue 修复数为 0  
    => 发布失败（或至少强 warning，不允许自动发布）
- 输出：`delta_report`（`similarity_score`、`fixed_issues`、`unfixed_issues`）。
