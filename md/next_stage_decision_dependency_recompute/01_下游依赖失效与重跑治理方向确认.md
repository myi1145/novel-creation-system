# 01_下游依赖失效与重跑治理方向确认

## 1. 文档目的

本文件用于对下一阶段主方向进行正式确认，并作为后续 `02 / 03 / 04` 文档的统一上游约束。

本文件是方向确认件，不是执行工单。

## 2. 结论先行

下一阶段主方向正式确认为：**下游依赖失效与重跑治理立项优先。**

一句话结论：
**先在立项层完成“失效提示 + 人工确认重跑 + 轻量重跑引导”的方向与边界收口，再决定是否进入实现，不直接跳转全链路自动重算开发。**

## 3. 为什么优先下游依赖失效与重跑治理

### 3.1 与当前系统痛点直接对齐

文本级、蓝图级、场景级人工修订闭环已建立，当前最突出风险已转为“上游改动后下游结果可能陈旧但缺乏统一治理口径”。

### 3.2 对主链稳定性影响最大

若不先定义失效治理边界，即使继续扩展编辑能力，也会放大跨层不一致与返工成本，影响主链可解释性与可审计性。

### 3.3 可分层推进、范围可控

先做方向确认与边界立项，可把复杂问题拆成“提示层、引导层、自动治理层”，避免一次性进入高风险全自动重算。

## 4. 为什么不是继续做更细一层编辑器

- 继续新增编辑入口不能直接解决“上游变更后的下游一致性”核心问题；
- 若治理机制缺失，新增编辑能力会增加状态组合复杂度与运维成本；
- 当前应优先补齐跨层治理口径，再讨论编辑能力继续下钻。

## 5. 为什么不是直接做全自动重算

- 自动重算涉及依赖图正确性、幂等保障、成本控制与人工确认策略，风险高于当前阶段承载能力；
- 在缺乏统一失效分层定义前直接自动化，容易破坏 Gate / ChangeSet / Publish 的可控边界；
- 应先完成立项层共识，再按轮次推进实现可行性。

## 6. 方向确认后的阶段含义

- 下一阶段以“下游依赖失效与重跑治理立项”作为唯一主任务口径；
- 产出重点是边界、MVP 与轮次规划，不产出执行工单；
- 仅在立项文档链稳定后，才可进入后续是否开发的决策。

## 7. 最终确认结论

### 主方向

**下游依赖失效与重跑治理立项优先。**

### 后置方向

- 更细粒度编辑能力扩展后置；
- 全链路自动重算实现后置。

### 当前不采用策略

**不采用“直接自动重算 + 并行扩编辑器”策略。**
