# 00_下游依赖失效与重跑治理候选方向与问题定义

## 1. 文档目的

本文件用于在“场景级人工修订最小闭环已签收”后，明确下一阶段候选方向、判断标准与问题定义，作为 `01~04` 文档的上游输入。

本文件是问题定义件，不是执行工单。

## 2. 当前阶段背景

已完成并签收事实：
- 文本级、蓝图级、场景级人工修订最小闭环均已形成稳定人工入口与最小审计能力；
- 章节主链“目标 -> 蓝图 -> 场景 -> 草稿 -> Gate -> ChangeSet -> Publish”可持续运行；
- 人工修订结果可回接主链，但“上游改动后的下游状态一致性治理”尚未形成统一机制。

因此，当前阶段核心不再是新增下一层编辑器，而是确定“上游修订导致下游结果失效时，系统应如何提示、如何引导重跑、如何控制范围”。

## 3. 候选方向清单

### 3.1 候选 A：下游依赖失效与重跑治理立项（推荐）

聚焦“失效识别规则 + 人工确认重跑 + 轻量重跑引导”的立项收口，先形成边界与 MVP，再决定是否进入实现。

### 3.2 候选 B：继续扩展更细粒度人工编辑能力（不推荐为当前主线）

继续下钻对象层新增编辑入口（例如更细粒度对象编辑器），暂不优先处理下游失效治理。

### 3.3 候选 C：直接启动全链路自动重算（不推荐为当前主线）

跳过立项阶段，直接开发“自动识别失效并自动重跑整个下游链路”。

## 4. 判断标准

下一阶段方向选择使用以下标准：
1. 与主链“目标 -> 蓝图 -> 场景 -> 草稿 -> Gate -> ChangeSet -> Publish”的一致性；
2. 对跨层返工风险的抑制能力；
3. 对 Canon / Story State 边界稳定性的影响；
4. 对范围可控性的支持程度（是否可先 docs-only 收口）；
5. 对后续自动治理演进的可承接性（是否可分层推进）。

## 5. 关键问题定义

本阶段需要先回答以下问题：
- 哪些上游变更会导致哪些下游对象或结果应视为“可能过期”；
- “失效提示”“人工确认重跑”“自动治理”三层能力应如何分层；
- 当前阶段最小可落地范围应聚焦到哪一层，如何避免范围膨胀；
- 如何在不改 Gate / ChangeSet / Publish 核心规则前提下建立治理闭环；
- 为什么当前不应直接进入全自动重算实现。

## 6. 初步判断

基于上述标准，候选 A（下游依赖失效与重跑治理立项）具备更高一致性与更低阶段漂移风险，适合作为下一阶段主方向进入 `01` 文档确认。

## 7. 输出约束

本文件输出仅用于方向比较与问题定义，不包含：
- 实现接口细节；
- 前后端开发任务拆解；
- 具体执行工单。
