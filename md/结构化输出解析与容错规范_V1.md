## 2. 文档目标

本文件用于正式定义系统内大模型结构化输出的解析协议、校验流程、错误分类、自动修复策略、重试机制、降级策略与审计要求。

本文件要解决的核心问题不是“怎么把 JSON 解析出来”，而是：

- 如何让 Agent 输出稳定落成系统可用对象
- 如何防止模型输出污染 Canon、对象层与工作流状态
- 如何在解析失败、字段缺失、语义偏移、枚举越界、跨对象冲突时进行可控处理
- 如何让解析与容错成为工作流正式环节，而不是零散补丁
- 如何让失败可追踪、可复盘、可统计、可优化

---

## 3. 正式定位

### 3.1 解析层不是“工具函数”

在本系统中，结构化输出解析层的正式定位是：

**模型输出到系统状态之间的安全边界层。**

它位于：
- Prompt 之后
- Agent Gateway 之内
- Workflow 编排之前
- Canon / Object 写入之前

它的职责不是美化输出，而是保证：
- 输出可解析
- 字段可落地
- 值域可校验
- 语义可接受
- 错误可分流
- 风险可审计

### 3.2 解析层的基本原则

解析层必须坚持以下原则：

1. **结构化优先于自然语言便利**
2. **解析成功不等于可入库**
3. **Schema 合法不等于语义合法**
4. **自动修复只能修格式，不能擅改业务事实**
5. **任何可能污染 Canon 的异常都必须拦截**
6. **失败必须可分级、可统计、可回放**

## 4. 输出对象分类

V1 中，系统内模型输出分为四类：

### 4.1 控制类输出

用于驱动流程，而不直接写入 Canon。

示例：
- ChapterGoalInputResult
- CandidateBlueprintList
- ScenePlanResult
- GateDecisionDraft
- RewriteInstructionSet

特点：
- 生命周期短
- 允许局部失败后重试
- 默认不直接入正史

### 4.2 创作对象类输出

用于形成对象层中的正式对象或候选对象。

示例：
- CharacterCardDraft
- RuleCardDraft
- OpenLoopCardDraft
- ChapterBlueprintDraft
- SceneCardDraft
- ReviewResultCardDraft

特点：
- 必须通过对象 Schema 校验
- 允许保留 draft / candidate 状态
- 进入正式对象前必须完成对象级校验

### 4.3 提议类输出

用于提议修改，而不直接执行修改。

示例：
- ChangeProposalDraft
- CanonPatchSuggestion
- RuleConflictSuggestion
- ObjectMergeSuggestion

特点：
- 只能进入提议链路
- 不能被自动视为最终事实
- 默认需要工作流或人工审批

### 4.4 正文类输出

用于生成章节正文、修订稿、局部重写文本。

示例：
- ChapterDraft
- SceneDraft
- RewriteDraft

特点：
- 输出主体为长文本
- 结构化部分主要体现在 meta 与 evidence 字段
- 后续需进入质量闸门与状态提取链路

## 5. 标准输出协议

### 5.1 V1 统一包装协议

所有 Agent 的结构化输出，原则上都应先包裹为统一 Envelope：

```json
{
  "request_id": "string",
  "workflow_run_id": "string",
  "agent_name": "string",
  "agent_version": "string",
  "output_type": "string",
  "status": "success | partial | failed",
  "payload": {},
  "warnings": [],
  "self_reported_risks": [],
  "source_refs": [],
  "metrics": {}
}
```

说明：
- `status` 仅表示模型自报状态，不替代解析层判定
- `payload` 才是实际业务对象
- `warnings` 与 `self_reported_risks` 用于辅助解析与审阅
- `source_refs` 用于追踪使用了哪些对象或状态片段

### 5.2 Payload 规则

`payload` 必须满足：
- 对应唯一 `output_type`
- 对应唯一 schema 版本
- 不允许混杂多个主对象根节点
- 不允许在未知字段中隐藏关键信息

### 5.3 输出必须显式空值

模型不确定时，必须输出：
- `null`
- `unknown`
- `needs_human_confirmation`

不得：
- 省略关键字段后假装完整
- 用模糊自然语言替代字段值
- 擅自编造枚举值

## 6. 解析总流程

V1 统一采用“七段式解析流程”。

### 6.1 第一步：原始响应接收

输入：
- 模型原始文本
- 请求上下文
- 目标 schema
- 当前 workflow 节点信息

输出：
- RawOutputRecord

要求：
- 原文必须原样落日志
- 不得在未留痕的情况下覆盖原始输出

### 6.2 第二步：包裹层提取

目标：
- 提取 Envelope
- 识别 `output_type`
- 识别 `status`
- 定位 `payload`

失败情形：
- 根节点不是合法对象
- 顶层多对象混杂
- 缺失 `payload`
- 缺失 `output_type`

### 6.3 第三步：语法级解析

目标：
- 解析 JSON / YAML / 指定结构化块
- 修复有限的逗号、引号、代码块包裹等格式问题

注意：
- 这一阶段只允许做格式修复
- 不允许改动业务字段含义

### 6.4 第四步：Schema 级校验

目标：
- 字段是否存在
- 类型是否正确
- 枚举是否在允许集合中
- 长度、范围、必填约束是否满足
- 嵌套结构是否完整

输出：
- SchemaValidationResult

### 6.5 第五步：语义级校验

目标：
- 字段之间是否自洽
- 对象引用是否存在
- 值域是否合理
- 与工作流当前阶段是否匹配
- 与题材配置、规则包、对象状态是否冲突

输出：
- SemanticValidationResult

### 6.6 第六步：业务准入判定

目标：
- 决定该结果是否：
  - 可直接进入下一节点
  - 可进入草稿态
  - 需自动修复后重试
  - 需 Re-ask
  - 需人工确认
  - 需直接阻断

输出：
- ParseDecision

### 6.7 第七步：标准化输出

目标：
- 生成系统内部统一对象
- 记录标准化后的字段映射
- 生成解析报告
- 进入下游 Workflow / Gate / Storage

## 7. 校验层级定义

### 7.1 L0：传输级校验

检查：
- 是否收到响应
- 是否超时
- 是否内容为空
- 是否截断

### 7.2 L1：语法级校验

检查：
- JSON / YAML 格式是否有效
- 代码块包裹是否可剥离
- 非法控制字符
- 基础编码问题

### 7.3 L2：Schema 级校验

检查：
- 必填字段
- 字段类型
- 枚举值
- 数组/对象结构
- 长度与范围

### 7.4 L3：语义级校验

检查：
- 字段间逻辑一致性
- 角色/地点/规则等引用对象是否存在
- 当前工作流阶段是否允许该输出类型
- 章位次与场景序号是否合理
- 状态迁移是否合法

### 7.5 L4：业务安全级校验

检查：
- 是否试图越权写 Canon
- 是否包含禁止直接落库的事实修改
- 是否跨越 Agent 权限边界
- 是否试图绕过 ChangeSet / 审批 / Gate

## 8. 错误分类体系

### 8.1 错误分级

采用四级分法：

- **P0 阻断级**：必须立即中断，不允许自动放行
- **P1 高风险级**：可重试或人工确认，默认不放行
- **P2 可修复级**：可自动修复后继续
- **P3 提示级**：不阻断主流程，但需记录

### 8.2 错误类型

#### E001 响应为空 / 截断
- 等级：P0
- 处理：重试；超过阈值则失败出栈

#### E002 非法结构化格式
- 等级：P1 / P2
- 处理：尝试格式修复；修复失败则 Re-ask

#### E003 缺失必填字段
- 等级：P1
- 处理：Re-ask 或使用显式空值规则

#### E004 字段类型不匹配
- 等级：P1 / P2
- 处理：有限自动转换；否则 Re-ask

#### E005 枚举越界
- 等级：P1
- 处理：不自动猜测，Re-ask

#### E006 未知字段污染
- 等级：P2 / P3
- 处理：保留原文；未知字段进入扩展区或丢弃

#### E007 对象引用不存在
- 等级：P1
- 处理：阻断对象入库；允许进入 draft 或 Re-ask

#### E008 业务语义冲突
- 等级：P1
- 处理：进入语义修复或人工确认

#### E009 越权提议 / 违规写入
- 等级：P0
- 处理：直接阻断并报警

#### E010 输出类型与工作流节点不匹配
- 等级：P1
- 处理：阻断并 Re-ask

#### E011 长文本污染结构化字段
- 等级：P2
- 处理：字段裁剪、移入备注区、必要时 Re-ask

#### E012 自相矛盾
- 等级：P1
- 处理：阻断；必要时发起澄清重试

#### E013 结构合法但事实高风险
- 等级：P1
- 处理：人工确认或进入 Gate 审查

#### E014 多对象混装
- 等级：P1
- 处理：阻断；要求单对象重答

#### E015 禁止值 / 保留字污染
- 等级：P1
- 处理：自动清洗或阻断

## 9. 自动修复边界

### 9.1 允许自动修复的内容

仅允许自动修复以下内容：
- 代码块包裹剥离
- 尾逗号、缺失引号、转义错误等轻量格式问题
- 明确可判定的数字/布尔字符串转换
- 空白裁剪
- 已知别名映射到正式枚举值前的规范化

### 9.2 禁止自动修复的内容

禁止自动修复以下内容：
- 猜测缺失的业务事实
- 猜测对象引用目标
- 猜测枚举值含义
- 擅自把自然语言改写成 Canon 事实
- 擅自推断人物关系、设定变更、时间线事件
- 擅自补全 ChangeSet patch_operations

### 9.3 自动修复的留痕要求

所有自动修复必须记录：
- 修复前内容摘要
- 修复后内容摘要
- 修复规则 ID
- 执行时间
- 是否影响业务字段

## 10. Re-ask 与重试策略

### 10.1 Re-ask 的正式定义

Re-ask 是对同一任务的受控重问，不是简单“再生成一次”。

Re-ask 必须携带：
- 原请求摘要
- 原输出问题摘要
- 允许修复的字段范围
- 不允许改变的已确认字段
- 本次必须满足的 schema 约束

### 10.2 重试等级

#### R1 格式重试
适用：
- JSON 结构错误
- 代码块污染
- 缺少包装层

特点：
- 不改业务任务
- 只强调格式与字段

#### R2 字段重试
适用：
- 缺少必填字段
- 枚举越界
- 类型错误

特点：
- 指定缺失字段
- 只要求补齐指定内容

#### R3 语义重试
适用：
- 对象引用错误
- 自相矛盾
- 工作流阶段不匹配

特点：
- 明确指出冲突点
- 限定重答范围

### 10.3 最大重试次数

V1 默认：
- 同一节点同一对象：最多 2 次自动 Re-ask
- 超过阈值后：
  - 进入人工确认
  - 或切换降级策略
  - 或标记节点失败

---

## 11. 降级策略

当结构化输出连续失败时，系统必须支持受控降级。

### 11.1 可用降级方式

1. **降级为更小对象输出**
   - 例如从完整 ChapterBlueprint 降级为仅输出 scene_goals 列表

2. **降级为半结构化输出**
   - 固定标题块 + 小字段集合

3. **降级为人工填写模板**
   - 对高风险节点进入人工补录

4. **降级为只保存原文草稿**
   - 不进入对象层，只作为参考材料

### 11.2 禁止的降级方式

禁止以下行为被视作“成功降级”：
- 直接把不可解析原文当作正式对象
- 跳过对象校验直接入库
- 把高风险提议自动写入 Canon

---

## 12. 与工作流的衔接规则

### 12.1 工作流节点必须声明目标输出类型

每个工作流节点都必须声明：
- expected_output_type
- expected_schema_version
- parse_policy
- retry_policy
- fallback_policy

### 12.2 解析结果标准状态

解析层统一输出以下状态之一：
- `accepted`
- `accepted_with_warnings`
- `needs_reask`
- `needs_human_review`
- `rejected`

### 12.3 Workflow 的处理动作

- `accepted`：进入下游节点
- `accepted_with_warnings`：进入下游，并附带 warnings
- `needs_reask`：触发受控重问
- `needs_human_review`：挂起并等待人工
- `rejected`：终止当前节点并出错误报告

---

## 13. 与对象层、状态层的边界

### 13.1 解析成功不直接写 Canon

即使输出解析成功，也只能：
- 形成候选对象
- 形成 ReviewResult
- 形成 ChangeProposal
- 形成草稿正文

只有经过对应工作流、质量闸门与审批后，才可能影响 Canon。

### 13.2 Draft / Candidate / Formal 三态要求

对于创作对象，V1 必须支持：
- `draft`
- `candidate`
- `formal`

解析层默认产出：
- draft 或 candidate

不直接产出：
- formal Canon facts

### 13.3 ChangeSet 的特殊要求

任何模型输出若试图描述 Canon 修改，必须先落为：
- ChangeProposalDraft

不得直接生成：
- 已应用 patch
- 已确认正史状态

---

## 14. 可观测性与审计要求

### 14.1 必须记录的日志

V1 必须记录：
- 原始模型输出
- 解析版本
- schema 版本
- 解析耗时
- 自动修复动作
- 重试次数
- 最终判定状态
- 错误码列表
- 关联 workflow_run_id / chapter_id / object_id

### 14.2 关键统计指标

至少统计：
- 各 Agent 的结构化成功率
- 各输出类型的首次通过率
- 平均 Re-ask 次数
- 自动修复成功率
- 人工介入率
- P0 / P1 错误分布
- 按题材模板的失败分布

### 14.3 审计原则

- 任何影响业务字段的自动修复必须可回看
- 任何被阻断的高风险输出必须可复盘
- 任何 ChangeProposal 的解析路径必须可追踪

---

## 15. V1 推荐实现策略

### 15.1 解析技术路线

V1 推荐使用：
- 统一 Envelope
- JSON Schema / Pydantic 校验
- 分层 parser
- 规则化 repair hooks
- 统一错误码体系

### 15.2 实现模块建议

建议拆分为以下模块：
- `output_parser`
- `schema_registry`
- `validation_engine`
- `repair_engine`
- `retry_coordinator`
- `parse_audit_logger`
- `decision_router`

### 15.3 与 Agent Gateway 的关系

Agent Gateway 负责：
- 发起请求
- 收取原始输出
- 调用 parser
- 收到 ParseDecision
- 回传给 Workflow

解析层不直接调度 Workflow，只返回标准结果。

---

## 16. MVP 必做范围

### 16.1 必做输出类型

MVP 至少要稳定支持以下结构化输出：
- ChapterGoalInputResult
- CandidateBlueprintList
- ScenePlanResult
- ReviewResultCardDraft
- ChangeProposalDraft
- ChapterDraftMeta

### 16.2 必做能力

MVP 必须实现：
- 统一 Envelope
- JSON 解析
- Schema 校验
- 基础语义校验
- 自动格式修复
- 受控 Re-ask
- 标准错误码
- 解析审计日志

### 16.3 可延后能力

以下可延后到 V2：
- 多格式并行解析
- 复杂语义纠偏模型
- 智能字段补全
- 高级异常聚类分析
- 跨项目解析策略动态学习

---

## 17. 非目标

V1 不追求：
- 用解析层替代质量闸门
- 用自动修复替代业务判断
- 让模型“想怎么答都能被系统吃下去”
- 将自然语言长文无损自动提纯为 Canon 事实
- 在没有对象 Schema 的情况下做宽松接入

---

## 18. 验收标准

以下条件满足时，可认为 V1 解析与容错层通过验收：

1. 目标输出类型首次结构化通过率达到可接受水平
2. 连续 5~10 章闭环中，不因解析层失控而污染 Canon
3. P0 / P1 错误均可被明确识别、记录与分流
4. 常见格式错误能够自动修复并留痕
5. 高风险业务错误不会被自动放行为正式对象或正史修改
6. Workflow 能基于解析结果完成 accepted / re-ask / human review / rejected 四类分流

---

## 19. V1 结论

在本系统中，结构化输出解析与容错层不是附属工具，而是连接大模型生成能力与系统控制能力的正式安全边界。

其根本目标不是“提高 JSON 成功率”，而是：
- 让 Agent 输出进入系统前先被标准化
- 让结构错误、语义错误、越权错误、事实高风险错误被正式识别
- 让失败成为可恢复、可分流、可审计的流程事件
- 让模型输出永远不能绕过对象层、工作流、质量闸门与 Canon 写入协议

因此，V1 正式采用：

**统一 Envelope + 七段式解析流程 + 四级错误分级 + 受控 Re-ask + 有边界自动修复 + 可观测审计**

作为系统的标准解析与容错方案。
