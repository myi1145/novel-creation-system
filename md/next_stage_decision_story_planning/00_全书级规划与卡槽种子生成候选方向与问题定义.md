# 00_全书级规划与卡槽种子生成候选方向与问题定义

## 1. 文档目的

在“基础卡槽导入导出阶段”完成签收后，重新识别系统下一阶段的核心矛盾，比较可选方向，明确为何当前阶段应转向“全书级规划与卡槽种子生成”。

## 2. 当前已完成能力回顾

基于当前仓库既有文档与阶段交接口径，系统已完成：

- 章节主链闭环：目标 -> 蓝图 -> 场景 -> 草稿 -> Gate -> ChangeSet -> Publish；
- 发布后能力：发布历史、版本差异与重发建议、已发布章节阅读与导出；
- 四类基础卡槽：角色卡、术语卡、势力卡、地点卡；
- 四类基础卡槽 JSON/CSV 导入导出、CSV 模板、导入校验报告最小闭环。

据此判断：章节级与发布后能力已具备较强工程闭环，基础卡槽维护能力已达到阶段签收条件。

## 3. 深度研究审查结论摘要

本轮采用的深度研究结论来源于用户提供的系统级重新审查意见（本仓库当前未检索到独立报告文件路径，故以任务单给出的审查结论作为阶段依据）。

审查结论摘要如下：

1. 当前系统最大结构性问题是上游缺少“全书级规划与设定资产生成”主线；
2. 章节闭环与后处理能力较强，但缺少写章节之前的全书级对象层；
3. 四类卡槽仍偏“人工维护表”，尚未成为由全书规划驱动的创作资产；
4. 下一阶段建议先做 docs-only 立项，不直接进入实现开发；
5. 当前不宜继续无序扩展导入导出、新卡型与自动抽取方向。

## 4. 当前系统结构性问题

当前结构性问题可归纳为：

1. 世界观与全书主线尚未作为“全书级对象”沉淀；
2. 分卷/阶段规划缺位，章节蓝图缺少稳定的上游依据；
3. 卡槽来源以人工维护为主，尚未形成“规划 -> 种子 -> 确认 -> 入史”的资产化链路；
4. 导入导出能力被误用为“主线能力”风险上升，偏离系统中枢设计。

## 5. 候选方向对比

### 5.1 继续扩卡槽导入导出

- 优势：工程确定性高，短期可继续提升维护效率；
- 问题：无法解决“章节生成前缺少全书级规划依据”的主矛盾；
- 结论：不应作为下一阶段主方向。

### 5.2 继续扩新卡型

- 优势：看似能补充设定维度；
- 问题：在上游规划缺位下扩卡型，会放大“维护表”负担；
- 结论：当前不优先。

### 5.3 继续做章节后处理增强

- 优势：可继续提升发布后质量控制；
- 问题：仍集中在下游，无法补足上游规划缺口；
- 结论：当前不优先。

### 5.4 只做手工全书大纲入口

- 优势：实现风险较低；
- 问题：仅有大纲入口不足以驱动卡槽资产化；
- 结论：可作为组成部分，不应作为单独方向。

### 5.5 全书级规划对象

- 优势：补齐上游对象层，改善章节蓝图输入质量；
- 问题：若不连接卡槽种子，难以形成资产反哺；
- 结论：必要但不充分。

### 5.6 全书级规划 + 卡槽种子生成

- 优势：同时补齐“上游规划对象”与“设定资产种子来源”，可直接服务章节蓝图；
- 风险：需严格守住 Canon/ChangeSet 边界，防止直接入史；
- 结论：应作为下一阶段主方向。

### 5.7 激进重构项目创建与章节主链

- 优势：理论上可一次性重排系统；
- 问题：违背 V1 增量演进原则，风险与成本过高；
- 结论：当前明确不采用。

## 6. 本阶段建议优先方向

**优先方向：全书级规划与卡槽种子生成（docs-only 立项先行）。**

核心口径：先定义全书级对象、种子对象与边界，再决定是否进入实现；不直接改写主链，不绕过 Canon/ChangeSet。

## 7. 本阶段暂不优先方向

- 继续扩基础卡槽导入导出能力；
- 继续扩新卡型（物品卡/伏笔卡/关系图谱等）；
- 自动全文抽取建卡；
- LLM 自动入 Canon；
- 激进重构章节主链。

## 8. 最终问题定义

**当前最需要解决的不是继续扩展卡槽维护工具，而是让系统在章节生成前先形成全书级规划与核心设定资产种子。**

下一阶段问题定义为：

在保持 V1 主链与 Canon/Story State/ChangeSet 中枢不变的前提下，建立“全书级规划对象 -> 卡槽候选种子 -> 作者确认 -> ChangeSet 入史”的最小设计边界，并形成后续实现的统一上游依据。
