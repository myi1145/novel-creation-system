# 00_发布前一致性验收与章节发布准入候选方向与问题定义

## 1. 文档目的

在“下游依赖失效与重跑治理闭环”签收完成后，本文件用于定义下一阶段候选方向与问题边界，回答“章节在发布前如何被统一判定为可安全 Publish”。

本文件是方向候选件，不是实现工单。

## 2. 阶段切换背景

当前系统已具备：
- 文本级 / 蓝图级 / 场景级人工修订闭环；
- 上游修订后下游 stale 标记与人工确认重跑最小治理闭环；
- Gate、ChangeSet、Publish 的既有主链与规则边界。

因此，下一阶段的主要矛盾已从“能否重跑”转向“发布前是否一致、是否可发布”。

## 3. 核心问题定义

发布前缺口不在单点能力，而在统一视图缺失：
- 作者需跨页面判断 stale、Gate、ChangeSet、Publish 前置条件；
- 系统尚未给出统一“章节发布准入状态”判定口径；
- 不一致状态缺少分层表达（提示、准入规则、阻断策略）。

## 4. 候选方向

### 方向 A：继续扩 stale / recompute 自动治理

定义：优先推进自动重算、跨章级联、批量调度能力。

问题：
- 无法直接回答“当前章节是否可发布”；
- 容易引入范围膨胀与调度复杂度；
- 与当前最短路径目标不对齐。

### 方向 B：先做发布前一致性验收与章节发布准入立项（推荐）

定义：先建立统一发布前状态汇总与准入口径分层，明确最小 MVP 与后续轮次。

优势：
- 直接响应作者“这一章能不能发布”的核心问题；
- 不破坏既有主链语义与 Canon 写入边界；
- 可为后续是否引入自动阻断与自动修复提供稳定前置。

### 方向 C：直接建设全功能发布控制台

定义：一次性并行推进聚合视图、规则引擎、阻断策略、自动修复与平台化工作台。

问题：
- 范围过大，跨层耦合高；
- 尚无稳定分层口径前直接实现，风险高；
- 不符合当前“先立项收口、再判定实现准入”的阶段策略。

## 5. 候选结论（本文件）

当前推荐方向为：

**优先进入“发布前一致性验收 / 章节发布准入”立项阶段，先完成方向确认与边界固化，再决定是否进入实现任务。**

## 6. 本文件输出约束

- 不定义接口与数据结构实现细节；
- 不定义发布准入面板代码任务；
- 不定义自动阻断发布落地方案；
- 不替代后续 01~04 文档结论。
