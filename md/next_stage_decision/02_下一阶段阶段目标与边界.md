# 02_下一阶段阶段目标与边界

## 1. 文档目的

本文件是在《01_下一阶段方向确认》基础上的继续收口文件，用于将已确认主方向进一步固化为可执行的阶段约束。

本文件的作用是正式明确下一阶段的阶段目标、实施边界、允许改动类型与后置事项，作为后续 `03 / 04 / 05` 文档与 Codex 执行任务单的统一上游约束。

本文件不是研究文档，不承担方向再论证职责；本文件也不是任务拆解文档，不承担实现排期与工单编写职责。

## 2. 阶段定位

当前项目已进入“下一阶段最小产品化闭环补齐”阶段。

本阶段核心问题不是继续铺设新的能力面，而是解决“作者无法真正接管草稿”的关键断点。

本阶段定位明确为：围绕文本级人工修订闭环，补齐系统从“能跑”到“作者可控地跑”的关键断点。

## 3. 本阶段唯一目标

**补齐文本级人工修订闭环。**

本阶段唯一目标展开如下：
- 让作者可以直接人工编辑 `ChapterDraft`。
- 让人工编辑行为具备最小可追踪性。
- 让人工编辑后的草稿能够重新进入 Gate。
- 让 Gate 之后仍可继续走 ChangeSet 与 Publish。
- 让“人工接管”成为主链中的正式一环，而非主链外的孤立动作。

## 4. 本阶段完成判定

本阶段完成判定句如下：

**作者是否已经能够在系统中人工修改 `ChapterDraft`，并将修改后的草稿继续送回 Gate → ChangeSet → Publish 主链。**

完成与否的判断标准如下：
- 不以功能数量作为完成标准。
- 不以对象种类扩充作为完成标准。
- 只以该主链是否真实成立、可稳定闭环作为完成标准。

## 5. 本阶段要解决的核心问题

### 5.1 草稿不可人工接管的问题

当前 `ChapterDraft` 虽处于工作流链路中，但作者对草稿缺少正式接管路径，导致“系统可生成”与“作者可控修订”之间存在关键断层。

### 5.2 人工接管语义已存在但产品未闭环的问题

系统已有人工接管相关语义与流程意图，但尚未形成前后端联通、状态可追踪、流程可回接的完整产品闭环。

### 5.3 人工修订后无法自然回到主链的问题

人工修改后的草稿若不能自然回到 Gate，并继续进入 ChangeSet 与 Publish，则人工修订无法成为主链内能力，仍属于链外补丁式操作。

### 5.4 修订过程需要最小可追踪的问题

人工修订过程必须具备最小审计与触发记录，以保证后续 Gate 判断、ChangeSet 审议与 Publish 追溯具备一致依据。

## 6. 本阶段目标范围

### 6.1 本阶段要做的事情

本阶段目标范围限定为以下事项：
- 为 `ChapterDraft` 增加人工编辑入口。
- 为人工编辑补齐最小审计记录。
- 在前端提供最小可用的草稿编辑页或等价入口。
- 将人工编辑结果重新接回 Gate。
- 确保 Gate 后可继续走 ChangeSet 与 Publish。
- 对该闭环补齐最小测试与最小文档说明。

### 6.2 本阶段目标的核心收敛原则

本阶段执行坚持以下收敛原则：
- 本阶段不是构建完整人工修订体系。
- 本阶段只补齐文本级人工修订最小闭环。
- 本阶段不追求一次做满，不以能力面扩张作为交付目标。

## 7. 本阶段严格边界

### 7.1 不做蓝图级人工修订

本阶段不纳入 `ChapterBlueprint` 等蓝图层人工修订能力建设。

### 7.2 不做场景级人工修订

本阶段不纳入 `SceneCard` 等场景层人工修订能力建设。

### 7.3 不扩结构化设定卡体系

本阶段不扩展结构化设定卡体系，包括但不限于：
- `LocationCard`
- `FactionCard`
- `ItemCard`
- `TerminologyCard`
- 其他世界观扩展对象

### 7.4 不重做既有页面主体

本阶段不重做以下既有页面主体：
- `WorkbenchPage`
- `GatesPage`
- `ChangesetsPage`
- `PublishedPage`
- `ObjectsPage`
- `OverviewPage`
- `WorkflowsPage`

### 7.5 不改后端主链核心业务语义

本阶段不改以下主链核心业务语义：
- `chapter-cycle`
- `chapter-sequence`
- gate 核心判定规则
- changeset 核心生成与审批逻辑
- publish 核心业务规则

### 7.6 不允许绕过 ChangeSet 直写 Canon

本阶段任何能力补齐均不得绕过 ChangeSet 机制直接写入 Canon。

### 7.7 不做大范围平台化升级

本阶段不纳入以下平台化升级事项：
- 多人协作
- 评论与批注系统
- 富文本系统
- 全站导航重做
- 大范围视觉升级
- 工作台平台化重构
- 运行中心增强版
- 更复杂的人工接管控制面

### 7.8 不做顺手重构

本阶段禁止以“顺手优化”为由引入无关重构，包括但不限于：
- 目录重整
- 命名统一式大改
- 通用 hooks 重构
- 前端状态体系重写
- 路由体系整体调整
- 无关测试清理

## 8. 本阶段允许的最小改动类型

### 8.1 后端允许的最小改动

- 在 `chapters` 相关路由下新增人工编辑 `ChapterDraft` 的接口。
- 在 `ChapterDraft` 现有 metadata 上补最小审计字段。
- 在既有状态流转记录中补充人工编辑触发信息。
- 为人工编辑闭环新增最小测试。

### 8.2 前端允许的最小改动

- 新增一个最小可用的 `DraftEditorPage`。
- 在 Workbench 或 Gate 等既有页面中补最小入口。
- 保存成功后提供回 Gate 的最小引导。
- 进行必要的 API client 补线与路由补线。

### 8.3 文档允许的最小改动

- 同步下一阶段相关 md 文档。
- 增补前端 README 中与人工修订闭环有关的最小说明。
- 记录本阶段新增能力的最短操作路径。

结构化设定卡类型扩展与深度对象化能力扩展后置。

### 9.2 蓝图级修订

蓝图层人工修订能力与对应闭环后置。

### 9.3 场景级修订

场景层人工修订能力与对应闭环后置。

涉及多角色、多权限、多状态编排的复杂人工接管控制面后置。

### 9.5 更完整的对象编辑体验

超出文本级草稿修订闭环所需范围的对象编辑体验增强后置。

## 10. 本阶段执行原则

### 10.1 单目标原则

所有实施动作均服务于“补齐文本级人工修订闭环”这一唯一目标。

### 10.2 最小闭环原则

优先实现可验证、可回接、可交付的最小闭环，不以功能面扩展替代闭环成立。

### 10.3 先审计后修改原则

所有改动应先完成现状审计与链路确认，再进行增量修改，避免破坏既有主链语义。

### 10.4 工作态优先原则

优先保证工作态中的人工接管可用、可追踪、可回链，再考虑体验层拓展。

### 10.5 诚实验收原则

验收以主链闭环真实成立为准，不以文档表述、界面数量或代码体量替代真实可用性判断。

## 12. 最终确认结论

### 阶段目标

**补齐文本级人工修订闭环，使作者能够人工接管 `ChapterDraft`，并继续推进 Gate → ChangeSet → Publish 主链。**

### 阶段边界

**本阶段只做文本级人工修订，不扩结构化设定卡，不做蓝图级修订，不做场景级修订，不改主链核心业务规则，不绕过 ChangeSet 直写 Canon。**

### 阶段方法

**以最小闭环、最小挂接、最小审计、最小回归风险为原则推进。**

## 13. 一句话口径

**这一阶段只解决一件事：让作者能手改草稿，并把改过的草稿继续送回主链。**
