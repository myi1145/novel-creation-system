# 04_任务拆解与轮次规划

## 1. 文档目的

本文件是在《01_下一阶段方向确认》《02_下一阶段阶段目标与边界》《03_下一阶段MVP范围》基础上的继续收口文件，用于将已确认的阶段目标与 MVP 范围进一步压实为可执行的任务拆解与轮次规划。

本文件作用明确如下：
- 正式明确本阶段按什么顺序推进、拆成几轮实施、每轮只解决什么问题。
- 明确审计轮、实现轮、验证轮、收口轮的职责边界与完成口径。
- 为后续 `05_Codex执行任务单_文本级人工修订闭环.md` 提供直接上游框架。

本文件不是研究文档，不承担方向再论证；本文件也不是最终执行工单，不展开代码级实现细节。

## 2. 轮次规划结论先行

下一阶段按**五轮推进**，执行顺序严格锁定为：

**先审计、后补后端、再补前端、再补验证、最后收口。**

五轮总体安排如下：
- **第 0 轮：仓库审计与挂接点确认**
- **第 1 轮：后端人工编辑接口补齐**
- **第 2 轮：前端草稿编辑页与入口补齐**
- **第 3 轮：闭环接线与最小验证补齐**
- **第 4 轮：收口、文档同步与阶段验收判断**

执行约束如下：
- 每一轮只解决一个层级的问题。
- 不并行开多条大线。
- 不在中途滑向扩卡、重构或平台化增强。

## 3. 总体执行原则

### 3.1 先审计后修改

- 必须先基于 current main 审计真实现状。
- 先确认最小挂接点、可复用结构和潜在风险。
- 禁止在没有审计结论的前提下直接进入代码修改。

### 3.2 后端先于前端

- 本阶段闭环基础是 `ChapterDraft` 的人工编辑写入口。
- 若后端接口与数据落位未定，前端页面先落地也无法形成真实闭环。

### 3.3 闭环优先于体验

- 本阶段优先保证“能编辑、能记录、能回 Gate、能继续主链”。
- 体验增强、交互优化、页面美化等事项统一后置。

### 3.4 最小可用优先于理想完备

- 本阶段目标是最小可用闭环，不是完整修订系统。
- 凡是显著扩大范围、牵动更多层级的内容，均不纳入本阶段当前轮次。

### 3.5 验收以路径成立为准

本阶段验收路径统一为：

**Workbench 生成 draft → 编辑 draft → 回 Gate → 再去 ChangeSet → 再继续 Publish**

只要该路径未成立，本阶段不判定达成。

## 4. 第 0 轮：仓库审计与挂接点确认

### 4.1 目标

确认 current main 中已有能力、缺口、可复用结构与潜在风险，锁定本阶段最小挂接点。

### 4.2 本轮定位

- 只做审计。
- 不追求实现。
- 不提前补功能。

### 4.3 需要审计的重点对象

#### 后端
- `app/db/models.py`
- `app/routers/chapters.py`
- `app/routers/changesets.py`
- `app/routers/workflows.py`
- 与 `ChapterDraft`、metadata、state-history、workflow status 相关的 schema / service / serializer

#### 测试
- `tests/test_workflow_revision_policy.py`
- 与 `ChapterDraft`、状态流转、mock provider 有关的测试
- 与 Gate / ChangeSet 主链兼容性有关的测试辅助代码

#### 前端
- `frontend/src/pages/WorkbenchPage.tsx`
- `frontend/src/pages/GatesPage.tsx`
- `frontend/src/pages/ChangesetsPage.tsx`
- `frontend/src/pages/PublishedPage.tsx`
- `frontend/src/router/AppRouter.tsx`
- `frontend/src/api/client.ts`
- 与 draft / chapter / gate / changeset 相关的 hooks / service

#### 文档
- `frontend/README.md`
- `md/status/current_stage_handoff.md`
- 已有前端阶段记录或验收记录

### 4.4 本轮必须回答的问题

1. 当前是否已有可复用的 draft 更新接口或半成品接口。
2. `ChapterDraft` 的 metadata 是否适合承载最小审计信息。
3. 是否已有 state-history 机制可直接复用。
4. 哪个页面最适合作为草稿编辑入口的前置页面。
5. 前端路由层最自然的编辑页挂载位置在哪里。
6. 哪些改动属于最小补线，哪些会引起范围膨胀。
7. 本阶段真正缺的是接口、页面，还是两者之间的接线。

### 4.5 本轮产出

- 一份简要审计结论。
- 一份最小改动点清单。
- 一份风险提示清单。
- 对下一轮实现范围的最终确认。

### 4.6 本轮完成标志

**已经明确最小挂接点、可复用结构和本阶段真正缺口。**

## 5. 第 1 轮：后端人工编辑接口补齐

### 5.1 目标

在后端建立清晰、正式、最小可用的 `ChapterDraft` 人工编辑能力，并保证人工编辑行为可追踪。

### 5.2 本轮定位

- 本阶段的基础实现轮。
- 重点是将工作态草稿人工编辑语义正式落地，而非前端可见性。

### 5.3 本轮必须完成的事项

#### 补一个正式的人工编辑接口
建议形态如下二选一：
- `PATCH /api/v1/chapters/drafts/:draftId`
- `POST /api/v1/chapters/drafts/:draftId/manual-edit`

#### 更新草稿内容
接口必须支持正式更新 `ChapterDraft.content`。

#### 写入最小审计信息
至少记录：
- `edit_reason`
- `edited_at`
- `source_type = human_edited`

条件允许时同时记录：
- `edited_by`
- `source_ref`

#### 补状态记录
若已有 `state-history` 或等价机制，应复用并追加人工编辑触发记录，例如：
- `trigger_type = human_edit`

### 5.4 本轮明确不做的事项

- 不做蓝图级编辑接口。
- 不做场景级编辑接口。
- 不改 Gate 核心规则。
- 不改 ChangeSet 核心策略。
- 不新增 Canon 直写能力。
- 不新增复杂审计系统。

### 5.5 本轮产出

- 后端人工编辑接口。
- 最小 request / response 支撑。
- metadata / state-history 的最小写入逻辑。
- 与后续前端接入兼容的接口契约。

### 5.6 本轮完成标志

**后端已经支持对 `ChapterDraft` 进行人工编辑，并保留最小审计痕迹。**

## 6. 第 2 轮：前端草稿编辑页与入口补齐

### 6.1 目标

在前端提供真实可用的草稿接管入口，使“人工编辑 draft”成为用户可操作能力。

### 6.2 本轮定位

- 本阶段核心产品接入轮。
- 重点是“让人有地方改”，不是“把编辑体验做满”。

### 6.3 本轮必须完成的事项

#### 新增最小 DraftEditorPage
建议路由：
- `/projects/:projectId/drafts/:draftId/edit`

页面最小能力：
- 读取 draft 内容。
- 大文本编辑。
- `edit_reason` 输入。
- 保存按钮。
- 成功反馈。
- 回 Gate 的引导或跳转入口。

#### 补一个正式入口
- 不允许只依赖手输 URL。
- 必须至少从一个既有页面提供正式入口，例如 Workbench 或 GatesPage。

#### API client 补线
前端必须正式调用后端人工编辑接口，并处理：
- 加载。
- 保存。
- 失败提示。
- 成功反馈。

### 6.4 本轮明确不做的事项

- 不做富文本编辑器。
- 不做差异对比编辑。
- 不做评论批注。
- 不做多人协作。
- 不重做 Workbench 主体。
- 不重做 Gate 页主体。

### 6.5 本轮产出

- DraftEditorPage。
- 路由挂接。
- 至少一个自然入口。
- 基本保存交互。

### 6.6 本轮完成标志

**前端已经提供真实可用的草稿编辑入口，用户能完成一次真实保存。**

## 7. 第 3 轮：闭环接线与最小验证补齐

### 7.1 目标

确保人工编辑后的 draft 真正接回现有主链，并补齐本阶段最小验证。

### 7.2 本轮定位

- 从单点能力到闭环成立的关键轮。
- 重点是将接口与页面接回 Gate → ChangeSet → Publish，而非新增页面。

### 7.3 本轮必须完成的事项

#### 接回 Gate
- 保存后的 draft 可继续被 Gate 使用。
- 用户明确知道下一步应回 Gate。
- 应提供明确跳转或 CTA。

#### 确认能继续 ChangeSet
必须确认人工修订后的 draft 不会导致：
- `ChangeSet` 无法生成。
- `ChangeSet` 流程断裂。
- 相关前端页无法继续处理。

#### 确认 Publish 路径未被破坏
- 本轮不新增 Publish 功能。
- 必须确认人工修订后的主路径不破坏后续发布链路。

#### 补最小测试
至少覆盖：
- draft 内容更新。
- metadata 写入。
- state-history 写入。
- mock provider 下闭环所需的最小后端验证。

#### 前端最小构建验证
至少执行：
- `cd frontend`
- `npm install`
- `npm run build`

### 7.4 本轮明确不做的事项

- 不扩为完整验收平台。
- 不追加更多修订层级。
- 不扩复杂导航体系。
- 不补无关回溯面板。

### 7.5 本轮产出

- 真正可走通的主路径。
- 最小测试。
- 最小构建验证。
- 最小人工验收结果。

### 7.6 本轮完成标志

**Workbench 生成 draft → 编辑 draft → 保存 → 回 Gate → 再去 ChangeSet → 再继续 Publish。**

只要该路径真实可走通，本阶段闭环才开始成立。

## 8. 第 4 轮：收口、文档同步与阶段验收判断

### 8.1 目标

对前面三轮结果进行收口，补齐文档，并完成阶段签收判断。

### 8.2 本轮定位

- 本阶段收口轮。
- 不再新增大功能。
- 重点是结果整理、验证归档、边界确认。

### 8.3 本轮必须完成的事项

#### 文档同步
至少同步：
- `/md/next_stage_decision/` 下相关阶段文档。
- `frontend/README.md` 的最小操作说明。
- 必要时补充本阶段新增能力的操作路径说明。

#### 输出变更清单
清楚说明：
- 改动文件清单。
- 后端改动。
- 前端改动。
- 测试改动。
- 文档改动。

#### 输出验收判断
必须诚实说明：
- 是否真正形成文本级人工修订闭环。
- 哪些仍是最小实现。
- 哪些事项进入下一阶段。
- 本阶段是否可正式签收。

#### 收口而不是扩展
- 不允许借“文档同步”继续扩功能。
- 只允许最小修正、最小补漏、最终判断。

### 8.4 本轮明确不做的事项

- 不再扩卡。
- 不再扩蓝图修订。
- 不再扩场景修订。
- 不再扩 UI 子系统。
- 不再顺手重构。

### 8.5 本轮产出

- 最终收口说明。
- 文档同步结果。
- 阶段签收判断。
- 后续阶段最小交接口径。

### 8.6 本轮完成标志

**本阶段结果已收口、文档已同步，并完成是否签收的诚实判断。**

## 9. 轮次之间的依赖关系

### 9.1 第 0 轮 → 第 1 轮

只有在审计完成、挂接点明确后，才能进入后端接口实现。

### 9.2 第 1 轮 → 第 2 轮

只有后端人工编辑接口与最小审计语义确定后，前端编辑页才有稳定接入对象。

### 9.3 第 2 轮 → 第 3 轮

只有前端已有真实编辑入口后，才有意义开展闭环接线与路径验证。

### 9.4 第 3 轮 → 第 4 轮

只有最小闭环跑通后，收口、文档同步与签收判断才成立。

### 9.5 依赖原则总结

**不允许跳轮，不建议并轮，不鼓励在前一轮未稳定时提前展开下一轮的大改。**

## 10. 每轮的完成判断口径

- **第 0 轮完成口径：**已经明确最小挂接点、可复用结构和本阶段真正缺口。
- **第 1 轮完成口径：**后端已经支持对 `ChapterDraft` 进行人工编辑，并保留最小审计痕迹。
- **第 2 轮完成口径：**前端已经提供真实可用的草稿编辑入口，用户能完成一次保存。
- **第 3 轮完成口径：**人工修订后的 draft 已真正接回 Gate → ChangeSet → Publish 主链。
- **第 4 轮完成口径：**本阶段结果已收口、文档已同步，并完成是否签收的诚实判断。

## 11. 不应出现的执行偏差

### 11.1 把第 1 轮做成后端大改

第 1 轮只补人工编辑最小接口与最小审计，不得扩展为后端架构重整。

### 11.2 把第 2 轮做成编辑器项目

第 2 轮只交付最小可用编辑入口，不得扩展为富编辑器建设。

### 11.3 把第 3 轮做成平台化整合

第 3 轮只做闭环接线与最小验证，不得扩展为平台化流程整合工程。

### 11.4 把第 4 轮做成新一轮规划

第 4 轮只做收口与签收判断，不得替代下一阶段规划工作。

## 12. 与 Codex 执行任务单的关系

- 本文件不是直接发给 Codex 的最终执行工单。
- 本文件直接决定 Codex 执行任务单的组织方式。
- 后续执行任务单建议按以下结构生成：
  - 以第 0 轮审计结论作为实施前提。
  - 以第 1 轮和第 2 轮作为核心实现任务。
  - 以第 3 轮作为验证与闭环任务。
  - 以第 4 轮作为输出与收口要求。
  - 每一轮均附带“严格边界”和“明确不做事项”。

## 13. 最终确认结论

### 13.1 执行顺序

**第 0 轮审计 → 第 1 轮后端 → 第 2 轮前端 → 第 3 轮闭环验证 → 第 4 轮收口验收**

### 13.2 规划原则

**每一轮只解决一个层级的问题，始终围绕文本级人工修订闭环推进，不并行扩卡、不并行扩修订层级、不并行做平台化升级。**

### 13.3 最终目标

**通过五轮最小推进，把 `ChapterDraft` 的人工编辑能力真正接回 Gate → ChangeSet → Publish 主链。**

## 14. 一句话口径

**这一阶段不是一口气做完整修订系统，而是分五轮，把“草稿能改、改完能继续跑”这件事稳稳做成。**
