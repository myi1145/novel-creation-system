# 03_下一阶段MVP范围

## 1. 文档目的

本文件是在《01_下一阶段方向确认》与《02_下一阶段阶段目标与边界》基础上的继续收口文件，用于将“人工修订层优先、仅做文本级人工修订闭环”的阶段共识，进一步固化为可执行、可验收、可约束后续拆解的 MVP 范围定义。

本文件定位如下：
- 本文件是“最小交付锁定件”，用于明确本阶段 MVP 的成立条件与交付边界。
- 本文件是“最小成立条件文档”，用于为 `04 / 05` 提供统一约束口径。
- 本文件不是研究文档，不承担方向再论证。
- 本文件不是任务拆解文档，不承担排期与工单展开。

本文件回答的问题只有一个：**本阶段最小做到什么样，才算“文本级人工修订闭环”成立。**

## 2. MVP结论先行

本阶段 MVP 结论如下：

**围绕 `ChapterDraft` 补齐“人工可编辑、可追踪、可重新进入 Gate、可继续走 ChangeSet 与 Publish”的最小闭环。**

具体收敛口径如下：
- 本阶段不追求完整修订体系。
- 本阶段不追求全层级编辑能力。
- 本阶段不追求对象层同步扩展。
- 本阶段只追求作者可在系统中手动修改草稿，并将修改后的草稿继续送回既有主链。

## 3. MVP完成判定

本阶段 MVP 的判定句如下：

**系统中是否已经存在一条真实可走通的路径，使作者能够对 `ChapterDraft` 进行人工编辑，保存后重新执行 Gate，并继续进入 ChangeSet 与 Publish。**

判定规则如下：
- 只要该路径未真实跑通，MVP 即不成立。
- 只要该路径已跑通，且具备最小审计与最小可追踪能力，MVP 即可判定达成。

## 4. MVP必须覆盖的能力

### 4.1 必须有 `ChapterDraft` 的人工编辑能力

系统必须提供工作态草稿的人工接管编辑能力，满足以下最小要求：
- 作者可直接修改工作态 `ChapterDraft` 文本内容。
- 编辑能力不依赖自动 revise 或重新生成。
- 作者可完成草稿定位、内容读取、内容修改、保存回工作态的完整动作。

### 4.2 必须有人工编辑的最小审计信息

人工编辑必须具备最小审计记录，至少包括：
- 记录“发生过一次人工编辑”。
- 记录本次编辑原因。
- 记录编辑时间。
- 记录来源类型为人工修订。
- 在结构允许范围内保留基础来源引用信息。

### 4.3 必须能把人工编辑后的草稿送回 Gate

人工编辑必须与主链回接绑定，满足以下要求：
- 编辑能力不得是链外孤立能力。
- 编辑完成后必须可重新进入 Gate。
- Gate 复用既有主链能力，不新造独立审查流程。

### 4.4 必须能从 Gate 继续进入 ChangeSet 与 Publish

人工编辑后的草稿必须可继续走既有后续主链，满足以下要求：
- 编辑后的草稿可被 Gate 后续流程正常承接。
- `ChangeSet` 与 `Publish` 路径不因本阶段改动而断裂。
- 全链路继续服从 Canon 仅通过 `ChangeSet` 写入的边界约束。

## 5. MVP的最小交付物

### 5.1 一个后端人工编辑接口

后端最小交付必须包含人工编辑 `ChapterDraft` 的正式接口，至少承担：
- 接收草稿标识。
- 接收新的草稿内容。
- 接收最小审计字段（至少包含编辑原因）。
- 更新 draft 内容并写回工作态。
- 写入最小审计信息。
- 不触碰 `CanonSnapshot`。
- 不绕过 `ChangeSet`。

### 5.2 一个最小前端草稿编辑页或等价编辑入口

前端最小交付必须包含可用编辑入口，至少满足：
- 展示当前草稿内容。
- 支持直接编辑文本。
- 支持填写编辑原因。
- 支持提交保存。
- 保存后提供回 Gate 的明确引导。

### 5.3 一条最小可走通的主路径

本阶段最小主路径定义如下：
1. 在 Workbench 中生成 draft。
2. 进入草稿编辑页。
3. 修改 draft 内容并填写 `edit_reason`。
4. 保存成功。
5. 回到 Gate 页面重新审查。
6. 再进入 ChangeSet。
7. 再继续 Publish。

### 5.4 一组最小测试或验证

本阶段最小验证集至少包含：
- 后端接口测试。
- draft 内容更新验证。
- metadata 或等价审计信息写入验证。
- state history 或等价状态记录验证。
- 前端构建验证。
- 主路径人工验收。

## 6. MVP不包含的内容

### 6.1 不包含蓝图级修订

不纳入 `ChapterBlueprint` 等蓝图层人工修订能力。

### 6.2 不包含场景级修订

不纳入 `SceneCard` 等场景层人工修订能力。

### 6.3 不包含新结构化对象类型扩展

不纳入结构化设定卡体系新增类型扩展与对象层扩卡工程。

### 6.4 不包含复杂编辑体验

不纳入以下复杂编辑能力：
- 富文本编辑。
- 差异对比视图。
- 多版本对照编辑。
- 评论与批注。
- 多人协作。
- 自动建议修订面板。

### 6.5 不包含主链核心规则重写

不纳入以下核心规则重写：
- Gate 核心规则。
- ChangeSet 核心规则。
- Publish 核心规则。
- `chapter-cycle` / `chapter-sequence` 主链语义。

## 7. MVP的最小数据要求

### 7.1 草稿内容可更新

`ChapterDraft` 正文内容必须可被人工更新，并可作为后续 Gate 输入继续流转。

### 7.2 最小审计字段可落位

最小审计字段至少可记录：
- `edit_reason`
- `edited_at`
- `source_type = human_edited`

在结构允许时同步记录：
- `edited_by`
- `source_ref`

### 7.3 状态记录可追踪

状态追踪要求如下：
- 若 current main 已有 `state-history` 或等价状态流转机制，本阶段优先复用。
- 不另造复杂日志系统。
- 至少可体现“发生过一次人工编辑”的可追踪事实。

## 8. MVP的最小前端要求

### 8.1 有正式入口

前端必须提供正式入口，要求如下：
- 不允许仅依赖手工输入 URL。
- 必须在至少一个既有页面提供入口。
- 入口可挂接于 Workbench、Gate 或其他当前最自然位置。

### 8.2 有最小可用编辑界面

编辑界面最小能力要求如下：
- 显示原始内容。
- 可编辑内容。
- 可填写原因。
- 可提交保存。

### 8.3 有明确的后续引导

保存成功后必须具备后续引导，至少满足：
- 明确提示下一步应回 Gate 重跑。
- 或提供回 Gate 的可点击跳转入口。

## 9. MVP的最小后端要求

### 9.1 有清晰的接口语义

人工编辑接口语义必须明确表达“人工修订”，不得以自动 revise 的别名方式模糊复用。

### 9.2 不破坏既有主链

新增逻辑不得导致以下问题：
- draft 无法继续被 Gate 使用。
- `ChangeSet` 无法继续处理相关草稿。
- `Publish` 流程异常。
- Canon 边界被破坏。

### 9.3 优先复用现有结构

实现口径如下：
- 若 current main 已存在可用 metadata、state transition、state-history、schema 或 response model，优先复用。
- 避免为 MVP 引入大范围新设计。

## 10. MVP验收口径

### 10.1 功能成立

作者可以在系统中对指定 draft 完成一次人工编辑并成功保存。

### 10.2 审计成立

人工编辑不是静默覆盖，至少保留最小原因与来源信息。

### 10.3 主链成立

编辑后的 draft 可重新进入 Gate，并继续走 ChangeSet 与 Publish。

### 10.4 边界成立

新增能力未绕过 `ChangeSet` 直写 Canon，未扩散为蓝图级、场景级或扩卡工程。

### 10.5 回归可控

Workbench、Gate、Changeset、Published 等既有页面主能力无明显回退。

## 11. MVP与后续增强的关系

本阶段 MVP 只负责把“文本级人工修订闭环”做成立，不承担全量修订体系建设。后续增强方向可包括：
- 蓝图级修订。
- 场景级修订。
- 更复杂的编辑体验。
- 更强的审计与版本对比。
- 更丰富的对象层支撑。
- 更强的人工接管控制面。

上述方向均属于后续增强，不属于本阶段 MVP 的成立条件。

## 12. 最终确认结论

### 12.1 MVP核心

**围绕 `ChapterDraft` 补齐人工编辑、最小审计、重新进入 Gate、继续走 ChangeSet 与 Publish 的最小闭环。**

### 12.2 MVP最小交付物

- 一个后端人工编辑接口。
- 一个前端最小草稿编辑入口。
- 一条可走通的人工修订后回主链路径。
- 一组最小测试与验证。

### 12.3 MVP明确不包含

- 蓝图级修订。
- 场景级修订。
- 结构化设定卡扩充。
- 复杂编辑体验。
- 主链核心规则重写。

## 13. 一句话口径

**这一阶段的 MVP，不是把修订体系做全，而是先让草稿真正能改、改完能继续跑。**
