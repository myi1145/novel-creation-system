## 当前系统真实状态判断

### 当前系统在“结构化设定层”上做到什么程度

从**后端数据模型 + 路由实现 + 前端对象页**三处交叉印证：当前“结构化设定/对象层”并不是空白，而是已经落了**一组可版本化、可走 ChangeSet 的最小对象子集**：

- **已落地的对象类型（并且具备：创建 / 列表 / 历史 / 通过 ChangeSet 提议 update/restore/retire + 可选 bind_to_canon）**
  - 角色卡（CharacterCard）：对象路由提供 `POST /characters`、`GET /characters`、`/history`、以及 `.../changesets/update|restore|retire`。citeturn9view4turn8view0  
  - 规则卡（RuleCard）：同样具备列表/历史/通过 ChangeSet 更新、并包含 `bind_to_canon`。citeturn9view4turn8view0  
  - 伏笔卡（OpenLoopCard）：具备创建/列表/历史/ChangeSet 更新链路。citeturn9view4turn8view0  
  - 关系边（RelationshipEdge）：具备创建/列表/历史/ChangeSet 更新链路。citeturn9view4turn8view0  

- **已落地的 CanonSnapshot 数据结构（明确包含：rules / characters / relationships / open_loops / timeline_events 等域）**
  - `CanonSnapshotORM` 存储 `rule_cards`、`character_cards`、`relationship_edges`、`open_loops`、`timeline_events` 这类“正史可用快照域”。citeturn27view7turn28view6  
  - 同时，Card ORM（如 `CharacterCardORM` / `RuleCardORM`）均有 `logical_object_id`、`version_no`、`is_current_version`、`predecessor_version_id`、以及 `snapshot_id` / `is_canon_bound` 等字段，说明“版本机制 + Canon 绑定意图”已经进入实体层。citeturn27view7turn30view0

### 当前系统在“人工修订层”上做到什么程度

结论更明确：**主链路里存在“修订/人工接管”的语义与测试，但产品层与 API 层缺少真正可用的“人工编辑”闭环。**

- **后端流程语义已经把“人工接管/停止自动修订”纳入可验证行为**
  - 测试 `test_workflow_revision_policy.py` 明确断言：当“无改进/达到最大修订轮数”时，决策为 `stop_for_manual_review`，并在执行章节循环/序列时返回 `stage_status = attention_required`、`next_action = review_revised_draft`。这等于：系统已经承认“需要人工介入继续推进”。citeturn32view0  

- **但当前 API 暴露与前端交互并没有提供“人工编辑蓝图/场景/草稿后继续 Gate→ChangeSet→Publish”的能力**
  - `chapters.py` 路由提供的是：生成蓝图候选、选择蓝图、场景拆解、草稿生成、草稿“修订”（看命名与用法更像 Rewriter Agent 的 revise），以及发布与摘要/派生更新等；**没有草稿内容的人工更新/版本化接口，也没有蓝图/场景的人工修改接口。**citeturn13view0  
  - `WorkbenchPage.tsx` 的 UI 主体是“一键跑链路”：创建目标→生成蓝图候选→选择蓝图→场景拆解→草稿生成→草稿修订；并明确提示“进入 Gate / ChangeSet / Publish 页面继续闭环”，但页面本身不提供编辑器，也没有任何可编辑蓝图字段的控件（仅能填/选 blueprint_id）。citeturn19view0turn18view4  
  - `GatesPage.tsx` 仅支持输入 `draft_id` 执行 Gate 并展示 JSON 结果，不提供在 Gate 失败后“进入人工修订→回写→重跑 Gate”的产品动作。citeturn23view0  
  - `ChangesetsPage.tsx` 支持从草稿生成 ChangeSet 提议、审批/驳回/应用/回滚，但同样不提供“人工改草稿/改提议内容”的闭环。citeturn22view0turn22view4  

综合判断：当前系统更像是“**流程能跑 + 控制面骨架 + 对象/ChangeSet/回滚机制已进入工程收口**”，但在“作者真实可控创作”的关键点——**人工修订与接管**——仍是断点。citeturn4view0turn33view1turn19view3turn23view0turn32view0

## 三个候选方向对比

### 结构化设定卡层优先

**优势**

结构化对象层是该项目的“不可退化底座”：Schema 文档明确指出，如果角色、设定、章节目标、场景拆解、审核结论等都只是 prompt 里的自由文本，系统会退化为 prompt 拼装器/记忆片段堆积器，导致关系不可校验、权限边界不清、无法追踪与回滚等问题。citeturn10view0turn11view4  
而当前 CanonSnapshot 已经以结构化域承载人物/规则/关系/伏笔/时间线等，这意味着继续扩展 Location/Faction/Item/Terminology 等“世界对象”在架构上是顺势补齐。citeturn10view0turn27view7turn28view6  

**风险**

最大风险不是“做不出来”，而是“做出来也不立刻解决最大致命缺口”：即便补齐更多卡（地点/势力/物品/术语），作者仍然无法在当前 UI/API 上对蓝图/场景/草稿进行人工编辑并继续推进链路——系统仍更像“自动流水线”，而不是“可控创作”。citeturn33view1turn19view0turn13view0  
同时，对象卡体系一旦扩张，前端至少要新增多类型表单、引用关系选择器、版本历史与回滚视图等交互，这会把下一阶段从“补 1 个闭环断点”变成“扩一个对象子系统产品面”，范围很难不膨胀。citeturn10view0turn10view2turn21view0  

**与当前项目契合度**

契合度在“架构使命”维度很高（Schema-first 是核心前提），但在“当前阶段目标（最小生产化收口后，进入下一阶段）”维度不够锋利：因为对象底座已存在最小集（角色/规则/伏笔/关系 + ChangeSet 联动），下一阶段最缺的不是“再多几种卡”，而是“作者能不能把不满意的章产物改到满意”。citeturn9view4turn21view0turn33view1  

**为什么可能是下一阶段**

如果你的下一阶段目标被定义为“世界观/设定密度优先（例如修仙体系/势力版图/地点拓扑）”，那么优先扩卡（Location/Faction/Item/Terminology）会直接提升一致性闸门的可用输入，减少后续章节跑偏。只是这与现阶段前端闭环断点相比，优先级不应居首。citeturn10view0turn33view1

### 人工修订层优先

**优势**

这是当前系统从“能跑流程”跃迁到“能创作产品”的最短路径，且与 `/md` 的明确要求一致：

- 《章节循环工作流说明（V1）》把修订回路明确分为三层：文本级、场景级、蓝图级，并要求“修订必须记录来源与原因，修订后需重新通过相关闸门”。这不是可选项，而是章节主干闭环的一部分。citeturn33view1  
- 同一文档还明确了 V1 默认人工介入点包含：候选蓝图最终选择、关键章蓝图修改、多轮失败后的处理决策、最终发布确认。也就是：**人工修改/接管被定义为流程默认机制**。citeturn33view7turn33view1  
- 测试层已把“停止自动修订→转人工接管”的结果做成可断言行为（attention_required / review_revised_draft），说明后端工程心智已经把“人工接管”作为真实状态机分支。产品层若不补上，将长期出现“系统说要人工接管，但人无处可接”的断裂体验。citeturn32view0  

**风险**

风险主要集中在两点：

- 如果“人工修订”缺少结构化记录，未来做“对象引用/一致性校验/可回滚审计”时可能返工。因此修订层必须至少做到：把修订原因、修订者、修订前后版本关系写进可审计的记录（比如 ChapterStateTransition / draft_metadata）。citeturn33view1turn30view0  
- 若一次性把“蓝图/场景/草稿全编辑器化”，范围会瞬间变大（尤其是场景级修订涉及局部重写与再拼装），需要强行做 MVP 收敛。citeturn33view1turn30view0turn19view0  

**与当前项目契合度**

极高。原因是：当前仓库已经有工作态对象（ChapterBlueprint / SceneCard / ChapterDraft）与闸门、ChangeSet、发布等链路实体；缺的只是“Human User 能编辑工作态章节资产，并把它重新送回 Gate→ChangeSet→Publish”。这在数据结构上不存在从 0 到 1 的鸿沟。citeturn30view0turn13view0turn22view4turn23view0  

**为什么可能是下一阶段**

因为它是“最小生产化基础收口（后段）”之后，最能立刻定义“下一阶段产品价值”的一刀：把系统从“跑通链路”变为“作者可控地跑通链路”。而且这刀不会破坏你强调的 Canon 写入原则（Canon 不可被前端直接写入；一切 Canon 修改走 ChangeSet），因为我们编辑的是工作态（blueprint/scene/draft），不触碰 CanonSnapshot。citeturn11view4turn33view7turn30view0

### 二者合并的最小闭环优先

**优势**

它的逻辑正确：作者要“真正可控”，既需要“可编辑章产物”，也需要“有结构化设定真相可依赖”。Schema 文档与架构文档都强调：交互层要避免退化为单一写字框，同时要有对象编辑、候选对比、闸门报告、ChangeSet 审批、版本/回滚等控制点。citeturn10view0turn11view2turn12view1  

**风险**

但在当前仓库现状下，“合并”反而容易被误解成“开两条战线”：

- 对象体系要做深：至少要补齐 Location/Faction/Item/Terminology 等世界对象，并做引用关系与校验；  
- 修订体系要做宽：至少要做蓝图/场景/草稿编辑与重过闸门。  

两者都做，哪怕每个都做“最小”，也会导致**交互层、数据层、测试层同时大改**，更容易拖慢你当前阶段刚收口的工程稳定性与验收可靠性。citeturn38view0turn33view1turn10view0turn30view0  

**与当前项目契合度**

“目标契合”高，“阶段契合”中等。因为当前对象层底座已经不是零：角色/规则/伏笔/关系 + ChangeSet 的机制已存在；所以本阶段真正缺的是修订闭环，不是先把对象卡扩满。若仍以“合并”为名，可能把焦点从最致命断点移开。citeturn21view0turn9view4turn33view1  

**为什么可能是下一阶段**

只有在你把下一阶段目标定义为“**强设定题材 + 强一致性校验优先**”（例如修仙/悬疑线索等）并且确认“当前作者并不需要大量人工改稿、而更需要强约束生成”时，才值得合并推进。但从现有文档与测试对人工接管的强调来看，当前更合理的是先补修订断点。citeturn33view1turn32view0turn12view1

## 最终判断

- **主推荐方向：人工修订层优先**（核心理由：当前最致命断点是“系统要求人工接管，但前端/后端没有可用的人工编辑闭环”；补上后系统才从“工程原型”进入“创作产品”。）citeturn32view0turn33view1turn19view0  

- **次推荐方向：结构化设定卡层优先（作为下一阶段之后的阶段进入）**（核心理由：对象层底座已存在最小集，扩卡应在“作者可控创作闭环成立”之后，避免先把对象产品面做大、但创作仍不可控。）citeturn21view0turn10view0turn33view1  

- **暂不优先方向：二者合并的最小闭环优先（作为“阶段目标表述”可以成立，但不应作为“工程实施策略”启动）**

## 判断依据展开

以下从你要求的五个维度同时论证“为什么人工修订层是第一刀”。

### 架构一致性

架构文档与工作流文档对“人工介入/人工修订”不是模糊态度，而是明确把它写进 V1：  
- 架构文档在章节循环里写明：作者在候选蓝图选择阶段“可局部修改”。citeturn12view1  
- 工作流文档定义了三层修订（文本/场景/蓝图），并规定修订后要重新过闸门、记录来源原因。citeturn33view1  
- 测试层把“无改进→停止自动修订→转人工接管”写成了可重复断言的行为（attention_required / review_revised_draft）。citeturn32view0  

而当前代码/前端实际只实现了“选择蓝图 + 跑通生成链路 + 过闸门 + ChangeSet 审批/回滚”，缺少“人工修订”这个被文档与测试共同承诺的节点。citeturn13view0turn23view0turn19view0  

### 作者真实痛点

站在作者视角，当前最致命的问题不是“没有势力卡/地点卡”，而是：

- **蓝图不合适时**：作者只能“换候选/重跑”，无法在 UI 上编辑 blueprint 字段后继续。WorkBench 只能选 id，不能改内容。citeturn19view0turn18view5  
- **场景不合适时**：场景卡是生成出来的，但前端没有场景查看/编辑/局部重写的交互入口。citeturn13view0turn17view0  
- **草稿不合适时**：只能走 `/drafts/revise`（偏自动修订），没有“作者直接改草稿内容并作为下一次 Gate 输入”的接口与页面。citeturn13view0turn23view0  
- **设定不合适时**：虽然对象页支持更新/恢复/退役的 ChangeSet 提议，但对象类型仅覆盖四类，且更新 payload 以 JSON 文本为主，对普通作者而言“可用但不友好”。citeturn21view0turn20view8  

也就是说：当前系统更像“能跑流程”，但作者的“可控性”主要靠重跑与选择，而不是靠编辑与接管。这与“把高杠杆决策点交给作者”的目标不匹配。citeturn12view2turn33view1turn19view0  

### 产品可感知价值

从前端已经落地的概览/运行中心/工作台（验收记录中明确这些页面已经交付）往下走，下一阶段用户最容易“肉眼感知”的提升不是多一个卡类型，而是：  
**我能把不满意的章节产物改到满意，然后继续 Gate→ChangeSet→Publish。**citeturn5view0turn33view1turn23view0  

原因很直接：对象卡扩充属于“长期收益”，但人工修订是“立即可用性”。没有编辑器与接管，系统就很难从 demo 进入真实写作。citeturn33view1turn19view0  

### 后续扩展成本

两条路径的返工风险对比：

- **先做结构化设定卡层（扩卡）再做人工修订**
  - 优点：未来修订时可以更强地引用对象、做一致性校验。citeturn10view0turn11view4  
  - 但缺点是：在作者还不能编辑章节资产时，你会把大量工程量投入到“约束与真相结构”，却仍无法形成作者可用闭环，导致下一阶段仍不得不再做一套“编辑/接管 UI + 工作态写入协议”。前后两次大变更，且第一次无法带来决定性可用性提升。citeturn33view1turn19view0  

- **先做人工修订层再扩结构化设定卡层**
  - 风险：如果修订记录不结构化，将来会难以审计与回滚。  
  - 但这一风险可以通过“最小结构化记录”控制：你已经有 `ChapterStateTransitionORM` 和 `draft_metadata`，完全可以在不新增复杂对象的情况下记录 `edited_by / reason / before_ref / after_ref` 等最小审计信息，从而把返工风险压到可接受。citeturn30view0turn33view1  

因此，从“避免返工/保持健康边界”的角度，**先修订、后扩卡**更容易保持数据边界：工作态章节资产的编辑与审计先稳定下来，再扩更多世界对象。citeturn30view0turn10view0turn33view1  

### 工程最小可行性

工程可行性上，“人工修订层”是当前对仓库最友好的增量：

- 数据层已经存在 ChapterBlueprint / SceneCard / ChapterDraft / GateReview / ChangeSet / PublishRecord，不需要重建模型；只需要在现有工作态对象上补“人工更新/记录/状态流转”即可。citeturn30view0turn29view2  
- 前端已经有工作台、Gate 页、ChangeSet 页：只要新增一个“Draft 编辑页”（或在 Workbench 内嵌编辑器），并提供保存后引导重跑 Gate，就能形成可用闭环。citeturn17view0turn23view0turn22view4  
- CI/验收层已有 stage acceptance 与真实 provider 回归工作流；人工修订层的改动可以被限定为“新增 API + 新增前端路由 + 新增测试”，不需要重写主链。citeturn38view0turn31view0

## 下一阶段最小落地范围

这一部分我按“人工修订层优先”的主方向，收敛为**一个可交付、可排期、可直接开工**的最小范围（MVP），并明确“不做什么”。

### 下一阶段只做什么

只做一件事：**补齐“文本级人工修订”的闭环**，让作者能在系统里真正接管并继续主链。

具体最小交付物（强约束）：

1) **后端：新增“人工编辑草稿内容”API（工作态写入，不触碰 Canon）**  
- 在现有 `/api/v1/chapters` 路由体系下，为 `ChapterDraft` 增加一个明确的人工更新入口（例如 `PATCH /api/v1/chapters/drafts/{draft_id}` 或 `POST /api/v1/chapters/drafts/{draft_id}/manual-edit`）。  
- 行为要求：  
  - 更新 `chapter_drafts.content`；  
  - 在 `draft_metadata` 写入最小审计字段：`edited_by`、`edit_reason`、`edited_at`、`source_type=human_edited`、`source_ref`；  
  - 追加一条 `chapter_state_transitions`（trigger_type=human_edit，from_status/to_status 合理落位，例如 from=`draft_ready` to=`draft_ready` 或 to=`revision_required`，以你现有状态机为准），确保“可追踪来源与原因”这一文档要求被落实到数据层。citeturn30view0turn33view1  

2) **前端：新增“草稿编辑器页”并挂接到现有导航**  
- 新增一个页面（建议独立路由，避免 Workbench 膨胀）：例如 `/projects/:projectId/drafts/:draftId/edit`。  
- 页面能力做到最小可用：  
  - 拉取并展示 `draft.content`（允许大文本编辑）；  
  - 输入 `edit_reason`；  
  - 保存后提示“请回 Gate 页重跑闸门”；并能一键跳转到 Gate 页（携带 draft_id）。citeturn23view0turn17view0turn14view0  

3) **链路闭合验证：人工修订后可继续 Gate→ChangeSet→Publish**  
- Gate 页仍按现有方式输入 draft_id 执行审查。citeturn23view0  
- ChangeSet 页仍按“从草稿生成提议”按钮生成 ChangeSet，并审批/应用/回滚。citeturn22view4turn22view0  
- 发布流程维持现状（publish_service / publish_record / snapshot / immutable log）。工作流文档对发布动作的要求（发布时 apply ChangeSet 到 CanonSnapshot 等）继续成立。citeturn33view7  

4) **最小守门：把“系统要求人工接管”变成“用户有地方接管”**  
- 当 workflow/tests 语义返回 `attention_required` / `review_revised_draft` 时，前端至少能让用户把 draft_id 带进编辑器页完成接管。citeturn32view0turn23view0  

### 下一阶段明确不做什么

为了保证“最小可行 + 不破坏收口稳定性”，以下明确不做（全部后置）：

- **不做蓝图级修订 UI**（不做 ChapterBlueprint 的编辑器化，不做蓝图版本分叉/重算场景/重算草稿的复杂联动）。citeturn30view0turn33view1  
- **不做场景级修订 UI**（不做 SceneCard 的逐场景编辑与局部重写拼装）。citeturn30view0turn33view1  
- **不扩结构化设定卡类型**（不做 LocationCard / FactionCard / ItemCard / TerminologyCard 等新对象类型；对象层本阶段只维持现有四类对象能力与 ChangeSet 联动）。citeturn10view0turn21view0  
- **不做对象编辑体验重构**（ObjectsPage 仍然可用，但不把 JSON 编辑升级为复杂表单/引用选择器）。citeturn20view8turn21view0  
- **不改 Canon 写入规则**（Canon 不允许前端直写；所有 Canon 改动仍必须通过 ChangeSet 审批/Apply/回滚）。citeturn11view4turn14view0  

### 为什么这样收敛最合理

这套收敛对应工作流文档的修订原则：“能小改，不大改；能局部重写，不整章重跑；修订必须记录来源与原因；修订后需重新通过相关闸门”。把下一阶段限定为“文本级人工修订闭环”，正好是最小且最能提升可用性的那层修订。citeturn33view1
