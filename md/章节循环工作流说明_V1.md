# 章节循环工作流说明（V1）

## 1. 文档信息

- 文档名称：章节循环工作流说明
- 版本：V1
- 文档定位：正式定版稿
- 适用范围：面向多题材长篇小说多 Agent 创作控制系统的章节级主流程设计
- 上游依赖：
  - 《小说多 Agent 系统最终架构设计文档（V1）》
  - 《Story State / Canon State 状态模型设计（V1）》
  - 《结构化创作对象 Schema 设计（V1）》
  - 《Agent 契约说明（V1）》
- 下游用途：
  - 章节生成编排器开发
  - 工作流节点拆解与任务调度实现
  - 质量闸门接入
  - ChangeSet 提取、审批与发布接入

---

## 2. 文档目标

本文件用于正式定义“章节级创作闭环”的统一工作流，解决以下问题：

1. 每一章从哪里开始、到哪里结束
2. 哪些输入必须提供，哪些输入禁止直接透传
3. 候选方案如何生成、过滤、选择
4. 场景拆解、正文生成、审阅修订如何衔接
5. 状态变化如何从章节内容中提取并进入 Canon
6. 哪些节点允许自动推进，哪些节点必须人工介入
7. 失败、回退、重试、终止、重开如何处理

本文件不讨论“具体 prompt 文案”，而是确定：

- 章节工作流的阶段划分
- 每阶段的输入 / 输出 / 约束
- 节点状态流转
- 审批规则
- 自动化边界
- MVP 最小闭环

---

## 3. 设计原则

### 3.1 章节工作流是系统主线，不是 Agent 自由协作

章节循环必须由 Workflow / Orchestrator 主导，Agent 只在规定节点执行规定任务。任何 Agent 都不能跳过流程直接写正史、直接发布章节、直接修改对象真相。

### 3.2 规划、生成、审阅、入史必须分阶段

章节级流程至少拆为四大阶段：

1. 章目标与约束确定
2. 候选方案与蓝图选择
3. 正文生产与修订
4. 状态提取、审批与发布

不得把以上环节压缩为“一个大 prompt 一次完成”。

### 3.3 先蓝图，后正文

正文生成必须以后验确定的 `ChapterBlueprint` 和 `SceneCard` 为输入，不允许 Writer 在无蓝图、无场景结构、无节奏约束的情况下直接生成整章正文。

### 3.4 先审阅，后入史

任何章节内容即使文本质量很高，只要未完成质量闸门与 ChangeSet 审批，就不属于 Canon 正史。

### 3.5 节点可重做，但状态必须可追踪

章节循环中的生成、修订、提取、审核节点允许局部重跑，但必须保留：

- 节点版本
- 节点输入快照
- 执行时间
- 执行者
- 重做原因
- 最终采用版本

---

## 4. 章节循环在系统中的位置

章节循环是卷级创作计划落地到实际发布内容的核心闭环，位于以下层级中：

- 项目初始化流程：确定题材、基线设定、初始 Canon
- 卷规划流程：确定卷目标、卷冲突、卷阶段节奏
- **章节循环流程：逐章推进叙事与状态演化**
- 发布后派生任务：摘要、索引、图谱、回顾材料、下一章预热约束

章节循环的触发条件通常为：

1. 当前项目已完成初始化
2. 所属卷存在有效的 `VolumeBlueprint`
3. 上一章已完成发布或系统允许首章启动
4. 当前项目不存在阻断性审核失败

---

## 5. 章节循环总体阶段

V1 正式定义章节循环为 **八步主流程 + 两类辅助流程**。

### 5.1 八步主流程

1. 章目标输入（Goal Initialization）
2. 候选章蓝图生成（Blueprint Proposal）
3. 候选章蓝图过滤与选择（Blueprint Selection）
4. 场景拆解（Scene Decomposition）
5. 正文生成（Draft Writing）
6. 多层质量闸门（Quality Gates）
7. ChangeSet 提取与审议（Change Extraction & Review）
8. 发布与入史（Publish & Apply）

### 5.2 两类辅助流程

1. 修订回路（Revision Loop）
2. 失败回退 / 人工接管流程（Fallback & Human Override）

---

## 6. 八步主流程详细定义

## 6.1 第一步：章目标输入（Goal Initialization）

### 6.1.1 目标

在开始本章之前，先由系统确定本章应该“承担什么任务”，而不是先生成正文再事后解释。

### 6.1.2 必要输入

- 当前 `CanonSnapshot`
- 当前 `VolumeBlueprint`
- 当前章节序号 / 章位次
- 上一章发布记录
- 上一章摘要
- 当前未闭合 `OpenLoopCard`
- 题材配置与节奏模板
- 当前卷的阶段信息（起势 / 推进 / 转折 / 爆发 / 收束等）

### 6.1.3 处理逻辑

由 `Pacing Engine + Rule Service + Workflow Service` 共同输出本章约束包：

- 本章主推进目标
- 本章次推进目标
- 必须承接的历史事实
- 允许触发的冲突级别
- 建议信息揭示量
- 视角约束
- 伏笔动作建议（提醒 / 加深 / 回收 / 不触碰）
- 禁止事项

### 6.1.4 输出对象

输出 `ChapterGoalSpec`，至少包含：

- `chapter_index`
- `goal_summary`
- `must_progress_arcs`
- `must_reference_facts`
- `allowed_new_elements`
- `pacing_constraints`
- `open_loop_constraints`
- `hard_forbidden_rules`
- `genre_constraints`

### 6.1.5 规则

- 本阶段不得生成正文
- 本阶段不得直接修改 Canon
- 若输入 Canon 不完整或卷规划缺失，则流程阻断

---

## 6.2 第二步：候选章蓝图生成（Blueprint Proposal）

### 6.2.1 目标

让系统围绕同一章目标生成多个结构上可比较的推进方案，而不是默认单路径。

### 6.2.2 执行主体

- `Chapter Planner Agent`
- 必要时可结合 `Open Loop Planner Agent`

### 6.2.3 输入

- `ChapterGoalSpec`
- 当前相关对象引用
- 当前卷目标
- 最近章节摘要
- 题材风格约束

### 6.2.4 输出

输出 3~7 个 `ChapterBlueprintCandidate`。

每个候选至少包含：

- `candidate_id`
- `chapter_purpose`
- `main_conflict`
- `sub_conflicts`
- `arc_progression`
- `open_loop_actions`
- `scene_count_estimate`
- `hook_strategy`
- `risk_notes`
- `canon_dependency_refs`
- `expected_state_changes`

### 6.2.5 约束

- 候选方案之间必须有可区分性，禁止换皮重复
- 候选方案必须显式引用其依赖的 Canon 事实
- 不允许把“正文段落”冒充“章蓝图”

---

## 6.3 第三步：候选章蓝图过滤与选择（Blueprint Selection）

### 6.3.1 目标

先过滤掉显著不合格方案，再从可行方案中确定本章正式蓝图。

### 6.3.2 过滤规则

先通过硬规则服务过滤：

- 是否违反世界规则
- 是否破坏人物当前状态
- 是否与卷目标明显脱节
- 是否节奏过冲或节奏空转
- 是否引入过量新元素
- 是否存在高风险设定污染

### 6.3.3 选择方式

V1 默认采取：

- 规则过滤后保留 2~4 个候选
- 由作者 / 编辑人工选择
- 支持在选中方案上进行局部修改

### 6.3.4 输出

输出正式 `ChapterBlueprint`。

### 6.3.5 人工介入要求

以下情况必须人工选择，不允许自动选定：

- 关键转折章
- 大型伏笔回收章
- 主要人物关系阶段变化章
- 新地图 / 新势力 / 新规则首次登场章
- 卷末章 / 卷首章

---

## 6.4 第四步：场景拆解（Scene Decomposition）

### 6.4.1 目标

将章蓝图拆解为场景序列，使正文生成建立在可校验的中间结构上。

### 6.4.2 执行主体

- `Scene Decomposer Agent`

### 6.4.3 输入

- 已定稿 `ChapterBlueprint`
- 相关 `CharacterCard`
- `LocationCard`
- `OpenLoopCard`
- 题材节奏约束

### 6.4.4 输出

输出一组有序 `SceneCard`。

每个 `SceneCard` 至少包含：

- `scene_id`
- `scene_order`
- `scene_goal`
- `enter_state`
- `exit_state`
- `participating_entities`
- `location_ref`
- `conflict_type`
- `emotional_curve`
- `information_delta`
- `open_loop_action`
- `expected_transition`

### 6.4.5 约束

- 场景必须覆盖章蓝图的主要推进点
- 场景不得无目的堆叠
- 场景数量应受题材模板和章字数预算约束

---

## 6.5 第五步：正文生成（Draft Writing）

### 6.5.1 目标

基于章蓝图和场景卡生成章节正文草稿。

### 6.5.2 执行主体

- `Writer Agent`
- 必要时可接 `Dialogue Enhancer Agent` 或 `Rewriter Agent` 进行局部增强

### 6.5.3 输入

Writer 的标准输入必须为：

- `ChapterBlueprint`
- 有序 `SceneCard` 集合
- 相关 `CharacterCard`
- 相关 `RuleCard`
- 最近章节摘要
- 风格配置
- 字数预算 / 篇幅约束

### 6.5.4 禁止输入

- 不得直接注入未经筛选的全书长文本
- 不得注入未经审批的 speculative 真相作为既定事实
- 不得将派生摘要替代 Canon 事实本体

### 6.5.5 输出

输出 `ChapterDraft`：

- `draft_id`
- `chapter_index`
- `based_on_blueprint_id`
- `text`
- `scene_alignment_report`
- `estimated_word_count`
- `draft_version`

### 6.5.6 生成策略

V1 推荐：

- 逐场景生成
- 场景间保留衔接摘要
- 末尾统一做一次整章流畅性拼接修订

这样更利于后续定位问题与局部重写。

---

## 6.6 第六步：多层质量闸门（Quality Gates）

### 6.6.1 目标

在正文生成后，对其进行分层质量检查，未通过前不得进入发布。

### 6.6.2 默认闸门顺序

V1 默认顺序：

1. `Canon Gate`
2. `Narrative Gate`
3. `Character Voice Gate`
4. `Style Gate`

必要时可扩展：

- `Pacing Gate`
- `Repetition Gate`
- `Safety Gate`

### 6.6.3 输入

- `ChapterDraft`
- `ChapterBlueprint`
- `SceneCard`
- 相关对象引用
- 当前 `CanonSnapshot`

### 6.6.4 输出

每个闸门输出一个 `ReviewResultCard`，包含：

- `gate_name`
- `status`（pass / fail / revise）
- `severity`
- `issues`
- `suggested_repairs`
- `object_refs`

### 6.6.5 结果处理

- `pass`：进入下一闸门
- `revise`：进入修订回路
- `fail`：阻断，需人工介入或回退至上游节点

### 6.6.6 关键约束

- 闸门不能直接覆盖正文，只能输出修订意见或修订任务
- 同一草稿可多次过闸，但必须保留历史版本

---

## 6.7 第七步：ChangeSet 提取与审议（Change Extraction & Review）

### 6.7.1 目标

从已通过质量闸门的章节正文中提取“真实发生的状态变化”，形成正式入史提案。

### 6.7.2 执行主体

- `Metadata Extractor Agent`
- `Change Proposal Agent`
- 规则服务辅助校验

### 6.7.3 输入

- 最终可发布 `ChapterDraft`
- `ChapterBlueprint`
- 当前 `CanonSnapshot`
- 上下游对象引用

### 6.7.4 输出

输出 `ChangeSetProposal`，至少包括：

- 新增事实
- 已知事实状态变化
- 人物关系变化
- 时间线推进
- 伏笔状态变化
- 新对象候选
- 不确定项
- 证据片段定位

### 6.7.5 审议规则

- 低风险、强可验证变化可走半自动审批
- 高风险、重大设定变化必须人工审批
- 不确定项禁止默认入史

---

## 6.8 第八步：发布与入史（Publish & Apply）

### 6.8.1 目标

将通过审核的章节正文与 ChangeSet 正式写入系统发布面与正史状态。

### 6.8.2 前置条件

必须同时满足：

- 正文版本已通过必须闸门
- ChangeSet 已审批
- 当前流程没有阻断性异常
- 项目处于允许发布状态

### 6.8.3 处理动作

发布时执行：

1. 标记 `ChapterDraft` 为发布稿
2. 写入 `PublishRecord`
3. 将已批准 `ChangeSet` 应用到 `CanonSnapshot`
4. 追加 `ImmutableLog`
5. 刷新章节摘要
6. 刷新相关派生索引
7. 生成下一章输入种子

### 6.8.4 输出

- `PublishedChapter`
- 更新后的 `CanonSnapshot`
- `ImmutableLogEntry`
- `NextChapterSeed`

---

## 7. 修订回路（Revision Loop）

### 7.1 定义

修订回路不是整章推倒重来，而是针对闸门提出的问题，在尽量小的范围内完成修复。

### 7.2 修订触发条件

- 任一闸门返回 `revise`
- 人工审阅提出局部修改
- ChangeSet 提取阶段发现正文与蓝图不一致

### 7.3 修订层级

V1 定义三层修订：

1. **文本级修订**：仅改正文表述，不改场景结构
2. **场景级修订**：调整单个或少量 `SceneCard`，再重写相关片段
3. **蓝图级修订**：回退到 `ChapterBlueprint` 重新选择 / 修改

### 7.4 修订原则

- 能小改，不大改
- 能局部重写，不整章重跑
- 修订必须记录来源与原因
- 修订后需重新通过相关闸门

---

## 8. 失败回退与人工接管流程

## 8.1 失败类型

章节循环中可能出现的失败包括：

1. 输入缺失失败
2. 候选方案质量过低
3. 闸门持续失败
4. ChangeSet 提取不稳定
5. 发布前审批被拒
6. 技术执行失败（超时 / 中断 / 解析失败）

## 8.2 回退策略

### 8.2.1 轻度失败

例如：风格偏差、轻微人物语气偏移。

处理：
- 进入文本级修订
- 不回退到蓝图层

### 8.2.2 中度失败

例如：场景逻辑断裂、推进不足、伏笔动作错误。

处理：
- 回退到场景拆解层
- 保留章目标和蓝图

### 8.2.3 重度失败

例如：章蓝图本身错误、重大设定冲突、节奏完全失控。

处理：
- 回退到蓝图选择层
- 必要时重新生成候选章蓝图

### 8.2.4 阻断失败

例如：卷规划缺失、Canon 状态不一致、重大审批被拒。

处理：
- 暂停章节流程
- 转人工接管
- 修复上游问题后重启

## 8.3 人工接管场景

以下情况必须支持人工接管：

- 连续两轮以上修订仍未过闸
- 关键剧情章产生多轮冲突
- ChangeSet 争议较大
- 作者想直接修改蓝图或正文
- 项目进入重要里程碑章节

---

## 9. 节点状态机定义

V1 建议为章节流程定义统一状态机。

### 9.1 章节主状态

- `pending`：待启动
- `goal_ready`：章目标已生成
- `blueprint_proposed`：候选蓝图已生成
- `blueprint_selected`：正式蓝图已选定
- `scene_ready`：场景拆解完成
- `draft_ready`：正文草稿完成
- `under_review`：正在过闸
- `revision_required`：需要修订
- `changeset_pending`：待提取 / 审议 ChangeSet
- `approval_pending`：待人工审批
- `publish_ready`：可发布
- `published`：已发布
- `blocked`：阻断
- `abandoned`：废弃版本

### 9.2 转移规则

- 只允许按工作流转移，不允许跳跃式写入 `published`
- 所有转移都应记录触发者、时间、来源版本
- `blocked` 状态只能由人工或恢复任务解除

## 11. 自动化边界与人工介入点

## 11.1 允许自动化的节点

V1 中默认允许自动推进：

- 章目标生成
- 候选章蓝图生成
- 场景拆解
- 正文初稿生成
- 多层闸门执行
- ChangeSet 初步提取
- 发布后的派生索引刷新

## 11.2 默认人工介入点

V1 中默认人工介入：

- 候选章蓝图最终选择
- 关键章蓝图修改
- 高风险 ChangeSet 审批
- 多轮失败后的处理决策
- 最终发布确认

## 11.3 可配置策略

后续版本可允许按项目策略配置：

- 常规章自动选蓝图
- 常规章自动发布
- 低风险 ChangeSet 自动审批

但 V1 默认不启用全自动发布。

---

## 12. 可观测性与审计要求

章节循环必须具备可观测性，至少要能追踪：

- 当前章运行到哪一步
- 每一步用了哪个版本的输入
- 哪个 Agent 生成了哪个结果
- 哪个闸门卡住了
- 哪个 ChangeSet 被批准 / 被拒绝
- 当前采用的是第几版草稿
- 为什么发生回退 / 阻断

建议每章都有一份 `ChapterRunRecord`，作为调试与复盘主记录。

---

## 13. MVP 最小闭环

V1 的 MVP 不要求把所有辅助能力一次做满，但必须完成最小章节闭环。

### 13.1 MVP 必须具备

1. `ChapterGoalSpec` 生成
2. 至少 3 个候选 `ChapterBlueprint`
3. 人工选择正式蓝图
4. `SceneCard` 拆解
5. `Writer Agent` 正文初稿
6. 至少两个闸门：`Canon Gate` + `Style Gate`
7. `ChangeSetProposal` 提取
8. 人工审批
9. 发布并写入 `CanonSnapshot`

### 13.2 MVP 暂不强制

- 自动节奏纠偏
- 高级统计分析
- 大规模并行生成
- 复杂多 Reviewer 协商机制
- 高级读者反馈闭环

---

## 14. 关键风险与控制策略

### 14.1 风险：章节流程被简化回“一次生成”

控制：
- 强制蓝图节点
- 强制场景节点
- 强制 ChangeSet 节点

### 14.2 风险：修订回路失控，成本爆炸

控制：
- 限制自动修订轮数
- 优先局部修订
- 超限后转人工接管

### 14.3 风险：状态提取不稳定

控制：
- ChangeSet 输出必须附证据片段
- 不确定项必须单独标记
- 高风险项必须人工审批

### 14.4 风险：关键章自动化过度

控制：
- 关键章节强制人工选择 / 审批
- 关键章禁用默认自动发布

## 15. V1 结论

章节循环工作流是小说多 Agent 创作控制系统的章节级主干闭环。

其正式路线为：

**先确定章目标，再生成候选章蓝图，再选定正式蓝图，再拆解场景，再生成正文，再通过多层质量闸门，再提取 ChangeSet，再审批发布入史。**

该流程的核心意义不在于“多几个步骤”，而在于：

- 防止正文脱离控制目标
- 防止写完再补状态
- 防止审阅与入史混在一起
- 防止 Agent 越权
- 防止长篇在几十章后失控

因此，章节循环不是写作辅助流程，而是系统实现“长期稳定创作”的核心生产线。
