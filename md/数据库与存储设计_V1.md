## 2. 文档目的

前面的文档已经回答了以下问题：

- 系统中枢是什么

6. 工作态、候选态、审阅态、派生态如何隔离。
7. MVP 阶段到底先建哪些表，哪些可以延后。

## 3. 设计原则

### 3.1 真相主库唯一

系统中所有“后续创作必须依赖的事实”都必须有唯一真相来源。

V1 中唯一真相来源为：

- **主关系型数据库中的 Canon Snapshot + 已审批 Creative Objects + 已应用 ChangeSet + 已发布记录**
- **不可变事件日志中的 Canon 演化历史**

向量库、图数据库、缓存、副本、全文索引、检索摘要都不是正史真相来源。

### 3.2 结构化主存优先于派生索引

角色、规则、关系、伏笔、章蓝图、场景卡、审核结果、审批记录等，都应优先以结构化主存落在关系型数据库中。

派生索引只服务于：

- 召回
- 检索
- 辅助分析
- 关系遍历
- 可视化
- 模型上下文拼装

不能反向取代主库结构化对象。

### 3.3 Canon / Working / Review / Workflow / Derived 必须分层

不同状态层的数据，不能混放在同一语义空间中。

至少要在存储层明确区分：

- Canon Truth Data
- Working State Data
- Review State Data
- Workflow Runtime Data
- Derived Index Data

即便某些表物理上在同一个数据库里，也必须在逻辑上分域、分表、分权限、分生命周期。

### 3.4 ChangeSet 是唯一正史写入入口

任何进入 Canon 的修改，都不能绕过 ChangeSet。

因此数据库层必须支持：

- ChangeSet 存储
- ChangeSet 审批状态
- ChangeSet 应用记录
- ChangeSet 回滚引用
- ChangeSet 与来源对象、发布记录、WorkflowRun 的关联

### 3.5 Snapshot 与 Log 并存

单独存快照不够，因为无法解释“怎么变成现在这样”；单独存日志也不够，因为读取成本高。

因此 V1 必须同时具备：

- **Canon Snapshot**：高频读取的当前真相视图
- **Immutable Log**：可回放、可审计的状态演化记录

### 3.6 表达稳定性优先于过度范式化

本系统是强业务约束系统，而不是单纯追求教科书式范式设计。

因此 V1 采用策略：

- 核心主实体保持明确表结构
- 变化频率高、题材差异强的补充字段放入 `extension_fields`
- 对象正文与结构元数据适当分离
- 优先保证状态清晰、查询可控、演化可扩展

### 3.7 先逻辑分层，后物理拆分

V1 可以先采用“一个主关系库 + 一个日志表域 + 一个对象文件存储 + 一个派生索引层”的实现方式。

V2/V3 再根据规模需要考虑：

- 独立事件库
- 独立向量库
- 独立图数据库
- 冷热分层
- 审计仓库
- 数据湖 / 离线分析层

## 4. 总体存储架构

V1 推荐采用五类存储职责分层。

### 4.1 主关系型数据库（Primary Relational Store）

负责：

- 项目与题材配置
- Canon Snapshot
- Creative Objects
- ChangeSet
- ApprovalTask
- PublishRecord
- WorkflowRun / TaskRun
- ReviewResult
- 用户与协作身份
- 章节级运行记录
- 关键业务枚举与配置

定位：

**唯一业务主库 + 唯一正史读取主入口**

### 4.2 不可变事件日志存储（Immutable Event Store）

负责：

- ChangeSet 应用事件
- Canon 关键状态迁移事件
- 发布事件
- 回滚事件
- 审批事件
- Workflow 关键里程碑事件
- 对象版本变更事件

定位：

**状态演化审计源**

说明：

V1 可以先以主库中的追加型事件表实现；不要求一开始就引入独立 Event Sourcing 基础设施。

### 4.3 对象文件存储（Object Blob Store）

负责：

- 草稿正文大文本
- 审阅长文本报告原文
- Prompt 渲染快照
- 模型输入输出原文
- 长篇摘要归档
- 导出文件
- 调试工件

定位：

**非主结构化对象的大文本附件层**

说明：

V1 可以先采用：

- 本地文件系统
- MinIO / S3 兼容对象存储
- 数据库存路径，文件本体不直接塞主表

### 4.4 派生索引层（Derived Index Layer）

负责：

- 向量检索索引
- 全文检索索引
- 关系图谱索引
- 章节摘要索引
- 人物语料切片索引
- 风格样本索引

定位：

**辅助检索与分析层**

说明：

派生索引全部可重建，不可视为真相主存。

### 4.5 缓存与运行态存储（Cache / Ephemeral Store）

负责：

- 会话级上下文缓存
- 工作流短期运行态缓存
- 任务队列消息
- 并发锁
- 热门对象缓存
- Prompt 片段缓存

定位：

**短生命周期运行辅助层**

说明：

V1 可选 Redis，也可以在低并发场景下先不引入复杂缓存体系。

## 5. 逻辑数据域划分

V1 在逻辑上将存储对象划分为九个数据域。

### 5.1 Project Domain（项目域）

存储：

- 项目基础信息
- 项目配置
- 当前启用题材
- 当前卷 / 当前章定位
- 项目状态
- 默认策略开关

### 5.2 Canon Domain（正史域）

存储：

- 当前 Canon Snapshot
- Canon 子域快照
- 已生效世界规则
- 已生效角色当前状态
- 已生效关系边
- 已生效伏笔状态
- 已发布章节元数据

### 5.3 Creative Object Domain（创作对象域）

存储：

- CharacterCard
- RuleCard
- RelationshipEdge
- OpenLoopCard
- ChapterBlueprint
- SceneCard
- ReviewResultCard
- ChangeProposalCard
- GenreProfile 等对象实例

### 5.4 Workflow Domain（工作流域）

存储：

- WorkflowDefinition
- WorkflowRun
- TaskRun
- NodeExecution
- RetryRecord
- RuntimeContextRef

### 5.5 Review Domain（审阅域）

存储：

- 闸门结果
- 问题条目
- 严重级别
- 修订意见
- 修订轮次
- 对应草稿版本

### 5.6 Change Domain（变更域）

存储：

- ChangeSet
- ChangePatchOperation
- ChangeApproval
- ApplyRecord
- RollbackRecord

### 5.7 Publish Domain（发布域）

存储：

- PublishRecord
- 发布版本号
- 发布对象引用
- 发布人
- 发布时间
- 发布时快照引用

### 5.8 Derived Domain（派生域）

存储：

- 向量块
- 图索引节点边
- 章节摘要
- 角色语料片段
- 风格样本片段
- 可检索标签索引

### 5.9 Audit / Ops Domain（审计运维域）

存储：

- 操作审计
- 权限审计
- 调用审计
- 关键失败记录
- 模型调用摘要
- 运行成本摘要

## 6. 关系型主库的正式职责边界

### 6.1 必须进入主库的数据

以下数据必须进入主关系库：

1. 项目主信息
2. GenreProfile 的启用配置
3. 所有核心结构化对象的元数据与结构字段
4. Canon Snapshot 当前视图
5. ChangeSet 及审批链
6. PublishRecord
7. WorkflowRun 与 TaskRun
8. ReviewResult 结构化摘要
9. Object Version 与引用关系
10. 用户、角色、审批人等协作信息

### 6.2 不建议直接塞入主库正文列的数据

以下数据不建议作为高频主表正文大字段长期堆积：

- 章节完整草稿全文
- 多轮修订原文全文
- 模型输入 Prompt 全文
- 模型输出原始 JSON / 长文本
- 超长审阅报告原文
- 导出 DOCX / Markdown 全文备份
- 大型图谱可视化快照

这些内容应：

- 主库存元数据与路径引用
- 正文放对象存储
- 必要时在主库中保留摘要与哈希

### 6.3 不属于主库真相职责的数据

以下数据不能当作主库真相来源：

- 向量召回结果
- 图数据库中的关系推断
- LLM 自动生成摘要
- 缓存中的临时上下文
- Prompt 渲染中间结果
- 前端本地暂存编辑态

## 8. 核心表级设计说明

以下设计以逻辑字段为主；具体 SQL 类型可在后续《数据库建表设计（V1）》中进一步细化。

## 8.1 `project`

用途：

记录小说项目主信息，是全部数据的顶级归属对象。

核心字段建议：

- `project_id`
- `project_code`
- `title`
- `description`
- `status`
- `active_genre_profile_id`
- `current_volume_index`
- `current_chapter_index`
- `owner_user_id`
- `created_at`
- `updated_at`

说明：

`project` 是跨所有对象与运行数据的根引用，核心业务表都应带 `project_id`。

## 8.2 `creative_object`

用途：

全部结构化创作对象的公共主表，用于承载对象公共协议。

- `object_type`
- `display_name`
- `object_version`
- `status`
- `canon_level`

- `creative_object` 负责公共元数据。

## 12. 读写权限边界

## 12.1 Agent

Agent 可：

- 读取 Canon Snapshot
- 读取相关对象
- 写入候选对象
- 写入草稿资产
- 写入审阅结果
- 生成 Change Proposal

Agent 不可：

- 直接写 Canon 表
- 直接改 `is_current` Snapshot
- 直接改 PublishRecord 状态为已发布
- 直接执行 ChangeSet Apply

## 12.2 Workflow Service

Workflow Service 可：

- 创建工作流实例
- 挂载对象引用
- 触发 Agent 任务
- 聚合审阅结果
- 创建 ChangeSet
- 发起审批任务
- 在批准后调用 Apply Service

Workflow Service 不应绕过审批规则直接写正史。

## 12.3 Apply Service

Apply Service 是唯一允许执行正史写入的服务。

它可：

- 校验 ChangeSet 状态
- 原子更新 Canon 子域表
- 生成新 Snapshot 版本
- 写 Immutable Log
- 写 PublishRecord
- 触发索引重建任务

## 12.4 Human Approver / Editor

人工可：

- 审批或拒绝 ChangeSet
- 编辑候选对象
- 触发回滚
- 发起人工修订

人工对正史的直接编辑也必须形成 ChangeSet，不允许静默改库。

## 13. 事务与一致性策略

### 13.1 强一致写入范围

以下操作应在同一事务或同一原子提交逻辑中保证一致：

1. ChangeSet 状态从 `approved` 到 `applied`
2. Canon 子域表更新
3. Canon Snapshot 版本头切换
4. PublishRecord 写入
5. Immutable Log 写入

### 13.2 最终一致范围

以下内容可以采用最终一致：

- 向量索引刷新
- 图谱索引刷新
- 全文索引刷新
- 缓存刷新
- 统计报表更新

### 13.3 幂等要求

以下操作必须具备幂等能力：

- Apply ChangeSet
- Rebuild Derived Index
- Retry Review Run
- Publish Callback
- Rollback Replay

## 14. 回滚设计

### 14.1 回滚原则

回滚不是“把表改回去”，而是一次正式业务操作。

因此回滚必须：

- 有 `rollback_record`
- 引用原 `change_set_id`
- 说明回滚原因
- 生成新的业务事件
- 可重放、可审计

### 14.2 V1 回滚策略

V1 推荐采用：

**基于 ChangeSet 的逻辑回滚 + 新 Snapshot 重建**

不建议直接做底层物理删改。

### 14.3 回滚影响范围

回滚至少要更新：

- Canon 子域表
- Canon Snapshot 头
- PublishRecord 状态或新增修订发布记录
- Immutable Log
- 派生索引重建任务

## 17. 风险与控制

### 17.1 风险：把向量库当真相库

问题：

- 检索结果不稳定
- 事实无法审计
- 版本不可解释

控制：

- 所有正史读取以主库 Canon 为准
- 向量库只返回候选引用，不返回事实定论

### 17.2 风险：正文全塞数据库

问题：

- 主表臃肿
- 查询与备份成本高
- 后续迁移困难

控制：

- 文本大字段进入对象存储
- 主库存路径、哈希、元数据

### 17.3 风险：快照只有大 JSON

问题：

- 查询困难
- 差异比对困难
- 局部更新粗暴

控制：

- Snapshot 头表 + 子域状态表组合
- 允许必要 JSON 扩展，但不让核心结构全退化成 JSON 黑盒

### 17.4 风险：工作态与正史混表

问题：

- 候选态污染正史
- 审阅态被误读为已生效
- 逻辑混乱

控制：

- 状态分域
- 字段明确 `status`、`canon_level`
- 发布前对象一律不得标为 Canon 生效对象

### 17.5 风险：回滚变成手工修表

问题：

- 不可审计
- 容易漏改
- 后续索引失真

控制：

- 所有回滚都走正式业务记录
- 回滚后触发 Snapshot 与索引重建

## 19. V1 结论

V1 的数据库与存储设计正式确定如下：

1. **主关系库是唯一业务主库与正史主读入口。**
2. **Canon 正史必须通过 ChangeSet 进入，不允许绕过。**
3. **Snapshot 与 Immutable Log 必须并存。**
4. **结构化对象优先进入主库，对象正文与大文本进入对象存储。**
5. **向量、图谱、全文检索都属于派生索引层，不是真相层。**
6. **工作态、审阅态、运行态与正史态必须逻辑分层。**
7. **MVP 先做主库闭环，先证明“连续章节创作控制”成立，再扩展复杂存储体系。**

该设计的目标不是一开始把存储系统做得很重，而是先保证：

- 真相唯一
- 正史可审计
- 写入可审批
- 发布可追溯
- 回滚可执行
- 派生可重建

这才是小说多 Agent 创作控制系统在工程层面真正成立的基础。
