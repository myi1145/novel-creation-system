# novel-creation-system 系统级重新审查报告

## 总体结论

当前 `novel-creation-system`（基于 current main）已经把“章节级闭环”做成了工程化主链：目标→蓝图→场景→草稿→Gate→ChangeSet→Publish，并进一步强化了发布后摘要、派生更新、版本差异/重发建议、连续章节（chapter-sequence）等“后处理与可审计能力”。项目文档与实现都明确把阶段焦点放在“章节主链 + 闸门 + 入史 + 发布”，并且当前阶段（handoff/decision 口径）正在推进“四类基础卡槽（角色/术语/势力/地点）的 JSON/CSV 导入导出（docs-only）”。citeturn43view0turn45view0

但系统级最大结构性问题也很清晰：**上游缺少“全书级规划与设定资产生成”的主线**，导致章节蓝图与章节目标的上游依据主要来自“上一章摘要/伏笔承接/临时卷目标”，而非“全书/分卷/阶段结构 + 世界观规则 + 设定资产（卡）”。这与 V1 文档体系中强调的“规划与写作分离”“卷规划流程”“卷蓝图/世界对象体系”等设计原则形成明显落差。citeturn37view2turn37view9turn38view8

因此：**你提出的关键疑问成立**——系统在“章节级流程与后处理”方面显著超前，而“全书级规划（世界观/大纲/分卷）→ 设定资产种子化 → 反哺章节生成”的上游主线缺口，正在成为下一阶段应优先补齐的“主线债务”，否则章节闭环越强，长期漂移的成本越高。citeturn43view0turn32view4turn36view7

## 当前系统能力盘点与真实可用程度

从 README 的“当前能力边界”与后端路由/服务实现可确认，目前系统“可联调/可回归”的能力主体集中在章节工作流与发布链路：单章主链、连续章节、Gate + ChangeSet + Publish 串联、发布后摘要/派生更新、修订策略与质量增益对比（baseline/Delta）等。citeturn43view0turn13view0turn12view3

从后端接口与数据库模型看，关键链路基本齐备：

- **项目创建**：后端 `projects.py` 提供创建与列表接口；`ProjectORM` 核心字段为 `project_name / premise / genre_id / current_chapter_no`，当前项目层并无“全书大纲/卷规划/世界观资产”的一等结构化对象入口。citeturn11view0turn25view0  
- **章节主链**：`chapters.py` 覆盖 chapter goal、蓝图获取/人工编辑/确认、场景拆解、草稿生成/修订/人工编辑、发布、发布历史/发布记录等；并提供“下游依赖状态/重跑”一类工程化后处理接口，显示主链的可维护性与可回滚意识较强。citeturn12view1turn12view0turn12view3  
- **工作流执行**：`workflows.py` 明确存在 `chapter-cycle` 与 `chapter-sequence` 执行与报告接口，说明系统已把“章节流水线”作为第一主线。citeturn13view0  
- **Canon/ChangeSet/发布一致性**：数据库存在 `CanonSnapshotORM`、`ChangeSetORM`、`PublishedChapterORM`、`PublishRecordORM` 等，且发布记录与快照/changeset 强关联，符合 V1“可审计入史”的核心约束。citeturn26view0turn27view2  

前端工作台也验证了“产品路径=章节路径”的事实：路由以 `workbench / gates / changesets / published / release-readiness / publish-history` 为核心，并明确“Canon 初始化仅调用 `/canon/snapshots/init`”“Canon 长期写入必须通过 ChangeSet 审批/应用链路”，其作者术语说明与最小闭环操作也完全围绕章节蓝图/场景/草稿/闸门/入史/发布展开。citeturn41view1

结论：**章节级闭环与后处理能力是真实可用的主资产**；但也因此更凸显“全书级规划缺口”的结构性影响（见下节）。

## 当前系统核心偏差

### 章节工作流被强化成“事实上的唯一主线”

README 明确将系统定位为围绕“章节主链、质量闸门、ChangeSet 入史、发布后派生更新”建立流程；并把 V1 约束中的固定链路写成项目准绳（目标→蓝图→场景→草稿→Gate→ChangeSet→Publish）。citeturn43view0

后端也确实围绕这一主线组织：`workflows` 以 `chapter-cycle` 与 `chapter-sequence` 为核心入口；`chapters` 服务对蓝图/场景人工编辑后，显式标记下游（scenes/draft/gate/changeset/publish）可能失效并要求确认重跑，体现了“章节后处理强化”。citeturn13view0turn32view3turn12view0

但 V1 架构文档明确提出“规划与写作分离”，并在 Workflow 层列出“项目初始化流程、题材模板初始化流程、卷规划流程、章节八步循环流程”等多条主线，其中“卷规划/项目初始化”是章节循环的上游。当前实现中，上游流程没有同等地位（甚至没有明显入口）。citeturn37view2turn37view9

### “Canon/对象体系”存在割裂：版本化正史对象 vs. 四类基础卡槽

从数据库与服务实现能观察到两个并行体系：

- **版本化对象 + CanonSnapshot/ChangeSet 体系**：`ChangeSetORM` 以 `patch_operations`、`base_snapshot_id/result_snapshot_id` 等方式承载“唯一入史入口”；`object_service` 明确禁止“对象创建直接写入 Canon”，要求通过 ChangeSet；并提供 `state_scope=canon/working` 的语义分流与 `is_canon_bound` 机制。citeturn27view2turn29view4turn29view3  
- **四类基础“结构化卡槽”体系**：存在 `structured_character_cards / terminology_cards / structured_faction_cards / structured_location_cards` 四表，字段包含 `last_update_source` 与 `is_canon`（默认 false），并配套 JSON/CSV 导入导出与模板/校验报告能力。citeturn22view0turn22view1turn20view5  

这一割裂带来两类典型风险：

1) **“哪个卡才是创作资产真相”的歧义**：同为“角色卡”，一边是版本化对象可绑定 Canon（严格入史）；另一边是结构化卡槽可被批量导入导出（强调迁移与协作）。如果没有清晰的“从卡槽→ChangeSet→Canon”的桥梁，卡槽就更像“表格化备忘录”，而非“驱动生成与校验的资产”。citeturn29view4turn22view0turn45view0  

2) **章节生成上游依据偏弱**：章节侧目前主要依赖 continuity pack（上一章摘要、open loops、next_chapter_seed、canon_summary、题材规则摘要等）组织输入；其本质是“章与章之间的承接控制”，而非“全书结构规划”。当全书规划缺失时，continuity pack 会被迫承担“大纲替代品”的作用，长期会导致结构漂移（例如卷目标临时化、阶段推进不可控）。citeturn32view4turn34view2  

### 文档体系的阶段主线正在固化为“卡槽导入导出 + 章节后处理”

当前阶段决策文档 `01_基础卡槽导入导出方向确认` 明确：下一阶段主方向是四类基础卡槽的 JSON/CSV 导入导出，且定位为 docs-only，暂不做“从正文自动抽取”“LLM 自动建卡”“自动生成 ChangeSet”等。citeturn45view0

README 也同步强调：阶段判断以 handoff 为准，且“当前进入基础卡槽导入导出方向确认与边界立项阶段”。citeturn43view0

这并不“错误”——它确实能增强协作/迁移能力，但从系统架构主线看，它进一步把资源投入到“资产维护工具链”，而不是“资产生成主线”。如果不先补齐全书级规划与资产生成，导入导出的收益会被限制在“批量维护效率”，难以转化为“长期创作稳定性”。citeturn45view0turn37view2

## 世界观、大纲、卡槽的正确层级

结合 V1 状态模型与当前实现，可以给出更“工程化”的层级判定（重点不是概念，而是：谁写、谁读、写入边界在哪里）。

### 需要提升为全书级对象的内容

V1 的 Canon State 范围明确包含“已确认的世界观规则”“已确认的卷目标与主线阶段状态”；并强调进入 Canon 必须形成 ChangeSet、通过审批、写入 Canon Snapshot 与 Immutable Log。citeturn35view5turn35view6

与此同时，V1 架构与工作流文档都把 `VolumeBlueprint` 作为章节循环的关键输入对象之一，意味着：**卷/阶段规划不是可选附属品，而是章节主链上游的硬输入**。citeturn38view8turn37view9

因此，下列对象应当在系统中被建模为“全书级”或至少“Project 级/Story Planning 级”的一等对象（并具备版本与确认机制）：

- 全书世界观（尤其是可约束生成的“世界规则/硬约束”）  
- 全书主线大纲（至少到分卷/阶段/关键转折点层）  
- 分卷/阶段规划（VolumeBlueprint/StagePlan，含阶段目标、推进节奏、关键角色弧光与伏笔策略）  
- 核心设定资产种子（核心角色/势力/地点/术语的种子集及引用关系）  

这些应当在“Project/Story State”层先形成可编辑版本，再通过 ChangeSet 成为 Canon 的可依赖事实（世界规则、阶段状态、确认的核心资产）。citeturn35view6turn27view2turn37view2

### 仍然应保持为章节级对象的内容

章节蓝图、场景卡、章草稿、闸门报告、发布前一致性检查、发布记录等，属于“章节循环工作流”的过程对象与产物对象，天然是章节级（或章节批次级）内容，且 V1 明确指出候选蓝图/候选场景/审阅意见不应直接视为 Canon。citeturn35view5turn12view0turn41view1

但关键在于：**章节级对象应当被全书级对象约束/引用，而不是相反**。当前系统在这一点上缺口最大。citeturn38view8turn32view4

## 卡槽体系重新定位

### 现状判定

四类基础卡槽（角色/术语/势力/地点）目前在 schema、存储与导入导出服务上都已形成“可批量维护”的闭环：类型齐全、字段结构明确、支持 JSON/CSV、模板下载、导入校验报告，并且字段包含 `is_canon` 与 `last_update_source`。citeturn16view0turn20view5turn19view5

但从“系统架构位置”看，卡槽当前更像**工作态资产库（Working Library）**，而非“驱动章节生成与正史演化”的核心资产，因为：

- 阶段决策文档明确规定：导入仅进入卡槽工作态，不等同 Canon 正史写入，也不自动生成 ChangeSet。citeturn45view0  
- 与此同时，系统已经存在严格的 ChangeSet→CanonSnapshot 入史机制与对象写入边界（object_service 甚至在逻辑上禁止直接写 Canon）。citeturn29view4turn27view2  

这意味着：**卡槽与 Canon/ChangeSet 主链之间尚缺一条“制度化桥梁”**，导致卡槽很难从“存储”升级为“创作资产”。

### 正确的升级方向

你提出的三态划分（候选卡/作者确认卡/已进入 Canon 的正式卡）与 V1 文档“候选/工作态/正史/审阅/派生分层”的原则是同构的。citeturn36view7turn35view6

从工程实现角度，建议将卡槽重新定位为：

- **Candidate（候选）**：由“全书级规划生成”与“章节产物抽取”产生，默认不进入 Canon，不产生强约束，只用于提示“可能需要建卡/补卡”。  
- **Author-Confirmed（作者确认）**：作者在卡槽内确认字段结构与命名一致性，成为后续章节生成的稳定输入（但仍未必等同 Canon）。  
- **Canon-Bound（正式设定/正史绑定）**：当某卡被判定为“后续必须遵守的事实”时，通过 ChangeSet 进入 Canon Snapshot/或 Canon 对象体系，使其具备冲突校验、可追溯来源与审批门禁。citeturn35view5turn27view2turn29view4  

同时，需要明确卡槽的“影响路径”：

- 卡槽（Author-Confirmed）应当进入 Agent Gateway/章节规划输入包，成为蓝图/场景/草稿生成的**上游依据之一**（与 canon_summary、open_loops、题材规则一起）。否则卡槽只会停留在 UI 侧的资料库。citeturn38view4turn32view4  

## 推荐新增主线

### 是否应该新增“全书级规划与卡槽种子生成”

应该新增，而且应当作为**下一阶段真正的主线**（而不是继续把资源投入到章节后处理/卡槽批量维护）。

理由并不抽象，而是基于 current main 的“结构事实”与 V1 设计目标：

- 现有系统的可控能力主要体现在：章节循环可执行、入史可审计、发布可回溯、连章承接可观测。citeturn43view0turn13view0  
- 但 V1 明确要求：Workflow 层应包含“卷规划流程”，章节循环关键输入对象中应包含 `VolumeBlueprint`；Canon State 应包含“世界规则、卷目标与主线阶段状态”。这些在 current main 尚未形成对应的一等对象主链。citeturn38view8turn35view5turn37view9  
- 当前章节目标侧的“current_volume_goal + continuity pack”是在用“局部承接机制”替代“全书结构规划”，其可扩展性与长期稳定性存在天然上限。citeturn32view4turn34view2  

### 这条新主线应该放在现有流程的哪个位置

建议把它放在：

**Project 创建/题材装载之后 → 第一次章节 goal 之前**，作为“项目上游初始化/规划阶段”的必经步骤（至少对新项目是必经；对存量项目可渐进补齐）。

在流程上，可以被视为对既有“章节循环工作流”的前置工作流：

- `Project Init / Story Planning Workflow`  
  - 输出：全书世界观（结构化规则/约束）、全书大纲（分卷/阶段结构）、核心资产种子（角色/势力/地点/术语候选）、初始 `VolumeBlueprint`（或 StagePlan）  
  - 进入 Story State（工作态），并提供“作者确认/审批入口”
- 之后再进入既有：章节循环（goal→blueprint→...）citeturn37view9turn43view0turn12view1  

### 与 Canon / Story State / ChangeSet 的关系

建议遵循 V1 的边界规则，用“状态层”而不是“功能堆叠”来约束新增能力：

- **Story Planning 产物默认属于 Story State（工作态/候选态）**，允许反复生成、人工修订、对比版本；  
- **当规划/设定被确认要成为“后续必须遵守的事实”时**：  
  - 通过 ChangeSet 进入 Canon（世界规则、阶段状态、确认的核心对象）  
  - 并写入 Canon Snapshot 与日志，以便后续章节的 Gate 校验与冲突检测能引用“权威真相”。citeturn35view6turn35view5turn27view2  

## 三种改造方案对比与最终推荐

### 方案 A：保守收口方案

改什么：将“全书级规划缺口”明确写入文档主线，但实现上只做**最小入口**，不重塑主链。具体表现为：

- 在文档中明确：当前系统主线以章节循环为核心，上游全书规划尚缺；  
- 在项目层增加“规划入口说明/手工规划指引”，让用户以文本方式维护“全书大纲/卷计划/世界规则摘要”，并把它作为 chapter goal 的额外手工输入；  
- 继续完成当前 docs-only 的卡槽导入导出设计闭环（不进入实现或仅实现最小导入导出）。citeturn45view0turn43view0  

不改什么：不新增全书级对象模型，不引入 VolumeBlueprint 的工程对象，不改变 chapter-cycle 的输入协议。

风险：低（几乎不动后端主链），但**结构性问题会继续累积**：章节蓝图仍缺乏可计算、可引用的全书依据；卡槽仍可能停留在“资料库”。citeturn32view4turn36view7

收益：见效快、干扰小；适合“短期维持可用性/联调节奏”。

工期复杂度：低。

适合当前阶段：如果团队当前必须优先完成“迁移/备份能力”以支撑协作落地，则可作为过渡；但无法解决你提出的核心疑问。citeturn45view0

### 方案 B：推荐改造方案

改什么：新增“全书级规划与卡槽种子生成”主线，并以**最小侵入**方式接入现有章节链路，使系统从“章驱动”升级为“书→卷/阶段→章驱动”。

关键改造点：

- 在 Project/Story State 层新增一组“规划对象”（可版本、可确认）：  
  - 世界观规则集（可约束后续生成）  
  - 全书大纲（分卷/阶段结构）  
  - VolumeBlueprint/StagePlan（可映射到章目标生成）citeturn35view5turn38view8turn37view9  
- 新增“从全书规划生成卡槽候选”的机制：  
  - 生成 Candidate 角色/势力/地点/术语卡  
  - 作者确认后进入 Author-Confirmed 状态（卡槽成为真正资产，而非仅存储）  
  - 需要入史的部分再通过 ChangeSet 进入 Canon（明确门禁与审计）。citeturn36view7turn27view2turn45view0  
- 修改章节循环的**输入契约**（而非重写流程）：  
  - chapter goal/blueprint 生成在现有 continuity pack 的基础上，引入“当前卷/阶段的计划片段 + 规划约束摘要 + 已确认卡槽资产摘要”。这样蓝图不再孤立生成。citeturn32view4turn34view2turn38view8  

不改什么：不推翻既有章节主链、Gate/ChangeSet/Publish 的工程闭环，不强制立刻把四类卡槽迁移到版本化对象体系（先做桥梁/映射）。
