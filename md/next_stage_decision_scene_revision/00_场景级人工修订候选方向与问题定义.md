# 00_场景级人工修订候选方向与问题定义

## 1. 文档目的

本文件用于在“蓝图级人工修订最小闭环已签收”后，明确下一阶段候选方向、判断标准与问题定义，作为 `01~04` 文档的上游输入。

本文件是问题定义件，不是执行工单。

## 2. 当前阶段背景

已完成并签收事实：
- `ChapterBlueprint` 人工编辑能力已落地；
- 蓝图级最小审计与历史查询已可用；
- 蓝图修订后可稳定回接“场景拆解 -> 草稿 -> Gate -> ChangeSet -> Publish”主链。

因此，当前阶段核心不再是继续扩蓝图级实现，而是确定“下一层人工修订控制点应落在哪里”。

## 3. 候选方向清单

### 3.1 候选 A：场景级人工修订立项（推荐）

聚焦 `SceneCard` 及其与蓝图、草稿之间的衔接语义，先在立项层完成方向确认、边界定义、MVP 范围和轮次规划。

### 3.2 候选 B：继续增强复杂蓝图编辑器（不推荐为当前主线）

在蓝图层继续扩展复杂交互、联动策略与高级编辑能力。

### 3.3 候选 C：先扩结构化设定卡层（不推荐为当前主线）

优先扩展世界设定对象面宽与深度，再回到章节推进链路。

## 4. 判断标准

下一阶段方向选择使用以下标准：
1. 与主链“目标 -> 蓝图 -> 场景 -> 草稿 -> Gate -> ChangeSet -> Publish”的一致性；
2. 对返工成本的抑制能力；
3. 对 Canon / Story State 边界稳定性的影响；
4. 对范围可控性的支持程度（是否可做 docs-only 立项收敛）；
5. 与已签收蓝图级阶段的递进关系是否自然。

## 5. 关键问题定义

本阶段需要先回答以下问题：
- 场景层是否是蓝图与草稿之间的下一高杠杆人工控制点；
- 场景级人工修订在本阶段应做到什么、不做什么；
- 如何避免把“方向确认”误写成“立即开发”；
- 如何保证不破坏 Gate / ChangeSet / Publish 与 Canon 写入边界。

## 6. 初步判断

基于上述标准，候选 A（场景级人工修订立项）具备更高一致性与更低阶段漂移风险，适合作为下一阶段主方向进入 `01` 文档确认。

## 7. 输出约束

本文件输出仅用于方向比较与问题定义，不包含：
- 实现接口细节；
- 前后端开发任务拆解；
- 具体执行工单。
