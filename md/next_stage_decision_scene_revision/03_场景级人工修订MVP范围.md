# 03_场景级人工修订MVP范围

## 1. 文档目的

本文件用于在立项层定义“场景级人工修订”最小可成立范围，为后续实现决策提供稳定约束。

本文件仅定义 MVP 范围，不定义实现任务。

## 2. MVP 结论先行

场景级人工修订的 MVP 在立项层应收敛为：

**先定义最小闭环语义与验证口径，不提前定义具体代码与页面实现。**

## 3. MVP 必须回答的问题

立项层 MVP 至少要覆盖：
- 场景级人工修订的最小业务对象范围是什么；
- 场景修订后如何保持与主链语义一致；
- 哪些行为可进入后续实现讨论，哪些必须后置；
- 最小验收口径是什么（文档层验收）。

## 4. MVP 最小范围（立项语义）

### 4.1 业务对象范围

- 仅围绕 `SceneCard` 及其与 `ChapterBlueprint`、`ChapterDraft` 的输入输出关系；
- 不扩展到结构化设定卡层；
- 不追加复杂蓝图编辑器增强需求。

### 4.2 主链一致性范围

- 继续遵循“目标 -> 蓝图 -> 场景 -> 草稿 -> Gate -> ChangeSet -> Publish”；
- 不改变 Canon 写入边界；
- 不定义绕过 Gate / ChangeSet 的捷径。

### 4.3 人工修订语义范围

- 明确“场景修订属于中游控制动作”，不是发布动作；
- 明确“场景修订不直接等于 Canon 写入”；
- 明确“场景修订结果仅作为后续草稿与质量闸门阶段输入”。

## 5. MVP 不包含内容

- 不包含场景编辑器实现；
- 不包含场景级接口实现；
- 不包含前端交互细节定稿；
- 不包含数据库结构调整；
- 不包含自动联动重算与复杂工作流改造；
- 不包含执行级测试用例与发布计划。

## 6. MVP 验收口径（本轮）

本轮 docs-only 验收采用以下标准：
- MVP 范围已文档化且边界清楚；
- 与方向确认、阶段边界文档一致；
- 不越界为实现任务单。

## 7. 结论

场景级人工修订的 MVP 当前仅处于“立项定义完成”状态，
**未进入“可直接开发”状态。**
