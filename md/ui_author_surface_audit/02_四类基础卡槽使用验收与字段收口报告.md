# 四类基础卡槽使用验收与字段收口报告

## 1. 验收范围

本轮仅验收并收口以下四类基础卡槽：角色卡、术语卡、势力卡、地点卡。  
覆盖页面：
- `frontend/src/pages/CharacterCardsPage.tsx`
- `frontend/src/pages/TerminologyCardsPage.tsx`
- `frontend/src/pages/FactionCardsPage.tsx`
- `frontend/src/pages/LocationCardsPage.tsx`
- `frontend/src/components/ProjectNav.tsx`
- `frontend/README.md`

本轮未扩展新卡型，未改 Canon / ChangeSet 主链。

## 2. 四类卡槽当前状态

- 四类卡槽已具备人工新建、编辑、查看闭环。
- 入口在项目导航中可直接到达，且标注为“基础”能力。
- 页面顶部已明确说明用途与阶段边界（人工维护、不会自动入 Canon）。

## 3. 每类卡槽字段问题

### 角色卡
- 原问题：列表只显示名称；详情为 raw JSON，作者不易快速理解。
- 收口：列表补充“身份定位 / 当前状态 / 是否进入正式设定”；详情按分组展示。

### 术语卡
- 原问题：术语定义与使用规则无填写引导；详情 raw JSON。
- 收口：补充字段提示（定义摘要、使用规则、首次出现章节用途）；列表突出类型与定义摘要。

### 势力卡
- 原问题：空状态偏泛；列表未突出立场与状态。
- 收口：空状态改为“主角所在宗门/王朝/敌对势力”；列表突出类型、立场、当前状态。

### 地点卡
- 原问题：空状态偏泛；列表未突出区域与剧情作用。
- 收口：空状态改为“出身地/宗门/关键秘境”；列表突出类型、所属区域、剧情作用。

## 4. 页面体验问题

- 详情区原先 raw JSON 会暴露工程字段，不符合作者主路径认知。
- 表单提示原先偏字段名映射，缺少“为什么填、怎么填”的说明。
- 按钮文案原先“创建/保存修改”不统一。

## 5. 建议保留字段

- 四类卡共同保留：名称（或术语名）、类型、简介/定义、首次出现章节、当前状态。
- 角色卡保留：身份定位、性格关键词、关系备注。
- 势力卡保留：立场、目标、核心成员、活动范围。
- 地点卡保留：所属区域、关键特征、相关势力、剧情作用。

## 6. 建议弱化字段

- `is_canon`：仅以中文“是否进入正式设定”呈现，附“仅标记、不自动入史”说明。
- `last_update_source`：仅以中文“来源：人工维护”呈现。

## 7. 建议后置字段

- `created_at` / `updated_at`：放在“来源与审计”分组末尾展示。
- 内部 `id` / `project_id`：作者主路径不突出展示。

## 8. 本轮实际修正项

1. 四类页面顶部用途说明统一并补充阶段边界。
2. 四类列表改为“名称 + 类型/定位 + 状态/摘要”的作者视图。
3. 四类详情改为“基础信息 / 设定内容 / 关系关联 / 当前状态 / 来源与审计”分组。
4. 四类表单提示补全，并新增“最近更新来源：本阶段主要为人工维护”。
5. 空状态文案改为作者可立即执行的首卡建议。
6. 按钮文案统一为“新建X卡 / 保存X卡”。
7. 项目导航四类入口加“（基础）”标注，降低误解为完整知识图谱的风险。
8. README 同步四类卡槽用途、阶段边界与暂不扩展内容。

## 9. 仍不建议做的事项

- 不扩展物品卡、伏笔卡、关系图谱。
- 不做自动抽取、LLM 自动生成卡片。
- 不接入章节上下文自动回填。
- 不改 Canon / ChangeSet / stale 主链机制。

## 10. 是否可以进入下一批卡槽

结论：**暂不建议立刻进入下一批卡槽开发**。  
建议先让作者实际使用四类基础卡槽一段时间，收集“字段可填写率、编辑频率、返工率、检索效率”等反馈后，再决定是否进入物品卡/伏笔卡等下一批能力。
