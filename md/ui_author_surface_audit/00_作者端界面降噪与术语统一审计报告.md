# 00_作者端界面降噪与术语统一审计报告

## 1. 审计结论摘要

本轮对作者端主路径页面完成了 **ui-audit only** 审计（不改代码）。结论如下：

- **工程信息暴露偏重**：多个主路径页面直接展示 `project_id / draft_id / blueprint_id / scene_id / publish_record_id`，并暴露 raw JSON、后端状态枚举与内部字段名，明显超出作者任务完成所需信息。
- **英文术语残留仍然系统性存在**：不仅有页面标题层的术语（如 Version Diff、Release Readiness 对应语义），还存在字段级暴露（如 `trigger_type`、`source_type`、`edit_reason`、`stale_id`、`patch_operations`、`published_chapter_id`）。
- **作者流程理解成本较高**：主路径虽然已经按“目标→蓝图→场景→草稿→质量检查→变更提案→发布”组织，但按钮语义中仍夹杂“重跑/状态机/字段解释”，空状态和错误提示在部分页面仍偏技术视角。
- **流程断点主要集中在“信息层级未分层”**：作者面与诊断面耦合，导致同一个页面同时承担“创作操作 + 调试诊断 +接口结果透出”，噪音过高。
- **优先整改方向**：
  1) 主路径先“去工程裸露”；
  2) 再统一术语中文化；
  3) 最后把必要诊断信息折叠到“诊断区/开发者视图”。

## 8. 下一轮最小整改清单

### P0：必须先改

1. 主路径页面移除/折叠 raw JSON 面板（Workbench、Gates、Changesets、Published）。
2. 去除作者首屏的内部编号强依赖输入（`draft_id`、`published_chapter_id` 由系统带入或列表选择）。
3. 清理中英混杂字段名直出（`edit_reason`、`trigger_type`、`source_type`、`patch_operations` 等）。
4. 将“作者操作文案”与“技术状态文案”分层：主文案只保留作者可执行动作。
5. 对“审批/应用/回滚”等关键 CTA 做作者语义改写，避免误操作。

### P1：建议本轮一起改

1. 统一空状态：每条空状态必须给出“下一步去哪、做什么”。
2. 统一错误提示：先产品化解释，再给可操作建议，不直接暴露接口语义。
3. Release Readiness / Publish History / Version Diff 的状态标签全中文化（含子状态）。
4. ProjectNav 增加“作者主路径优先”顺序，弱化诊断入口优先级。
5. 编辑历史区改为“作者可读版 + 诊断折叠详情”。

### P2：后续优化

1. 建立“作者视图 / 诊断视图”切换机制（或统一折叠诊断区）。
2. 将版本差异页进一步产品化为“是否建议重发”的结论页。
3. 对发布历史页做信息架构收敛：默认展示“最近发布 + 是否建议重发”。
4. 梳理 README 的作者术语词典与研发字段词典双轨说明。
