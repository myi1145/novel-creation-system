## 2. 文档目标

本文件用于将前述架构设计文档中的核心结论，翻译为可执行、可排期、可验收的 MVP 开发任务清单。

MVP 阶段的目标不是一次性做成平台完整版，而是尽快验证以下关键命题：

1. 系统是否能够以 **Story State / Canon State** 为中枢稳定运行；
2. 系统是否能够依靠 **结构化对象 + 工作流 + 多层质量闸门** 而不是单纯长 prompt 完成章节闭环；
3. 系统是否能够完成至少一个题材模板下的 **连续多章稳定创作**；
4. 系统是否能够做到 **正史写入可审批、可回滚、可追踪**；
5. 系统是否能够在作者参与下形成“候选蓝图 -> 场景拆解 -> 正文生成 -> 评审修订 -> 发布入史”的最小闭环。

因此，MVP 阶段的第一原则是：

**先验证创作控制闭环成立，再扩展平台能力。**

---

## 3. MVP 范围定义

### 3.1 MVP 必须交付的能力

MVP 阶段必须落地以下七项核心能力：

1. **项目初始化与题材装载**
   - 能创建创作项目
   - 能绑定一个题材配置
   - 能初始化基础 Canon 状态与基础对象集

2. **Story State / Canon State 最小中枢**
   - 能维护 Canon Snapshot
   - 能维护 Immutable Log
   - 能维护 Working / Speculative State
   - 能通过 ChangeSet 写入正史

3. **最小结构化对象层**
   - 角色卡
   - 世界规则卡
   - 关系边
   - 伏笔卡
   - 章蓝图
   - 场景卡
   - 审核结果卡

4. **章节循环主流程**
   - 章目标输入
   - 候选章蓝图生成
   - 蓝图选择
   - 场景拆解
   - 正文生成
   - 多层质量闸门
   - ChangeSet 提议
   - 发布入史

5. **最小 Agent 闭环**
   - Chapter Planner Agent
   - Scene Decomposer Agent
   - Writer Agent
   - Canon Review Agent
   - Style Review Agent
   - Summarizer Agent
   - Change Proposal Agent

6. **最小交互控制台**
   - 项目创建
   - 题材模板选择
   - 当前章蓝图对比与选择
   - 场景卡查看
   - 评审结果查看
   - ChangeSet 审批
   - 发布操作

7. **基础可观测与可回滚能力**
   - 操作日志
   - 任务日志
   - 闸门结果记录
   - ChangeSet 审批记录
   - Canon 快照版本记录
   - 基于版本的回滚

### 3.2 MVP 明确不做的内容

以下内容不进入 V1 MVP 必做范围：

- 多用户复杂权限系统
- 丰富编辑器体验
- 可视化图谱大屏
- 插件市场
- 成本治理平台
- 多模型调度平台
- 高级统计分析后台
- 自动长期排程生成
- 多端产品化交互
- 全题材一次性铺开

MVP 只要求先在 **1~2 个优先题材模板** 上验证闭环有效。

## 4. MVP 成功标准

MVP 完成后，应至少达到以下验收目标：

### 4.1 功能性成功标准

1. 能创建新小说项目；
2. 能选择题材模板并初始化项目；
3. 能生成多个候选章蓝图；
4. 能进行蓝图选择并触发场景拆解；
5. 能基于场景卡生成正文草稿；
6. 能完成至少两类质量闸门检查；
7. 能生成 ChangeSet 并进行审批；
8. 能将审批后的状态写入 Canon Snapshot；
9. 能形成章节发布记录；
10. 能对发布后的 Canon 进行版本回滚。

### 4.2 创作控制成功标准

1. 连续 5~10 章测试中，无明显正史事实随机漂移；
2. 人物核心身份、关系与关键设定不发生无依据冲突；
3. 伏笔状态可追踪，至少支持 open / remind / payoff / closed 四态；
4. 章节推进目标可以通过蓝图显式表达；
5. 草稿评审失败后可以进入修订回路，而不是直接发布。

### 4.3 工程成功标准

1. 所有核心写入都经由 ChangeSet；
2. 所有关键流程节点都有日志；
3. 所有关键对象都有 schema 校验；
4. 工作流失败后支持重试或回退；
5. 所有已发布章节都能追溯对应的蓝图、草稿、闸门结果与 ChangeSet。

## 5. 开发总原则

### 5.1 先中枢，后外壳

优先开发：
- 状态中枢
- 对象层
- 工作流
- 质量闸门
- ChangeSet

延后开发：
- 重 UI
- 重产品包装
- 重平台管理后台

### 5.2 先闭环，后扩展

优先把单章循环跑通，再扩展：
- 多卷规划
n- 多题材组合
- 高级规则包
- 多用户协作

### 5.3 先确定协议，再接 Agent

所有 Agent 接入前，先把以下内容定死：
- 输入对象
- 输出对象
- 允许读取范围
- 禁止写入范围
- 错误返回结构
- 审计记录格式

### 5.4 先做最小闸门，再做复杂评分

MVP 阶段只先做好：
- Schema Gate
- Canon Gate
- Style Gate（基础版）

Narrative Gate、Voice Gate 可以先做轻量版或半自动版。

## 6. 开发阶段划分

MVP 建议拆为六个阶段推进。

### 阶段 A：项目底座与工程骨架
目标：搭建基础工程、统一协议、初始化运行框架。

### 阶段 B：状态中枢与对象层
目标：让系统具备“状态化”与“对象化”基础。

### 阶段 C：章节主流程闭环
目标：让系统能从章目标走到章节发布。

### 阶段 D：Agent 接入与评审闭环
目标：让关键 Agent 按契约接入，并形成修订回路。

### 阶段 E：题材模板与控制台
目标：让用户可实际创建项目并操作闭环。

### 阶段 F：联调、验收与样书测试
目标：完成连续章节测试、问题修复与 MVP 交付。

## 7. 详细任务拆解

---

## 7.1 阶段 A：项目底座与工程骨架

### A-1 工程目录与模块骨架

**目标**：建立可持续扩展的项目结构。

**任务内容**：
- 建立后端目录结构
- 建立前端或控制台目录结构
- 建立配置模块
- 建立日志模块
- 建立异常处理与统一响应模块
- 建立任务执行器基础骨架

**交付物**：
- 项目工程目录
- 基础配置文件
- 启动脚本
- 日志与异常模块

**验收标准**：
- 项目能本地启动
- 模块边界清晰
- 日志和配置可正常读取

### A-2 统一对象协议与通用 Schema 基础库

**目标**：建立所有 Creative Object 的公共字段协议。

**任务内容**：
- 定义 BaseObjectSchema
- 定义对象公共元数据字段
- 定义版本字段
- 定义状态字段
- 定义对象引用结构
- 定义审计字段

**交付物**：
- 公共 schema 基础类
- 对象引用协议
- 版本协议

**验收标准**：
- 所有对象均可继承公共 schema
- 能完成基础校验

### A-3 工作流引擎最小骨架

**目标**：建立后续章节主流程的编排基础。

**任务内容**：
- 定义流程节点接口
- 定义流程上下文对象
- 定义节点状态机
- 定义失败重试与中断恢复机制
- 定义运行日志记录

**交付物**：
- Workflow Engine 雏形
- Node Handler 协议
- 流程运行日志结构

**验收标准**：
- 能执行一个包含多个节点的样例流程
- 节点失败可记录并中断

---

## 7.2 阶段 B：状态中枢与对象层

### B-1 Story State / Canon State 数据模型落地

**目标**：实现状态中枢基础表结构与核心服务。

**任务内容**：
- 建立 Canon Snapshot 存储结构
- 建立 Immutable Log 存储结构
- 建立 Working State 存储结构
- 建立 State Query Service
- 建立 Snapshot 构建服务

**交付物**：
- 状态存储模型
- 状态查询服务
- Snapshot 重建服务

**验收标准**：
- 能读取当前 Canon Snapshot
- 能写入并读取 Working State
- 能从日志重建快照

### B-2 ChangeSet 机制实现

**目标**：实现正史唯一写入入口。

**任务内容**：
- 定义 ChangeSet schema
- 实现 ChangeSet 创建服务
- 实现 ChangeSet 审批服务
- 实现 ChangeSet Apply 服务
- 实现 ChangeSet 回滚服务

**交付物**：
- ChangeSet 存储模型
- 审批服务
- Apply / Rollback 服务

**验收标准**：
- 任意正史写入必须经过 ChangeSet
- 已应用 ChangeSet 可追踪
- 可执行回滚

### B-3 核心 Creative Object 落地

**目标**：实现 MVP 必需对象。

**任务内容**：
- CharacterCard
- RuleCard
- RelationshipEdge
- OpenLoopCard
- ChapterBlueprint
- SceneCard
- ReviewResultCard

**交付物**：
- 对象 schema
- 对象 CRUD / Query 服务
- 对象引用校验

**验收标准**：
- 对象可创建、可查询、可版本化
- 对象间引用可解析

### B-4 对象校验器实现

**目标**：保证对象层不退化成自由文本块。

**任务内容**：
- 实现 schema validator
- 实现必填字段校验
- 实现枚举值校验
- 实现对象引用存在性校验
- 实现状态机校验

**交付物**：
- 校验器组件
- 错误结构定义

**验收标准**：
- 非法对象不能入库
- 错误信息明确可追踪

---

## 7.3 阶段 C：章节主流程闭环

### C-1 章目标输入节点

**目标**：把当前章的输入约束结构化。

**任务内容**：
- 读取当前 Canon Snapshot
- 读取最近章节摘要
- 读取未关闭伏笔
- 读取题材配置
- 生成 Chapter Goal Input

**交付物**：
- ChapterGoalInput schema
- 构建服务

**验收标准**：
- 任意章节能生成结构化章目标输入

### C-2 候选章蓝图节点

**目标**：实现候选章蓝图生成与存储。

**任务内容**：
- 接入 Chapter Planner Agent
- 定义章蓝图候选输出 schema
- 实现候选蓝图存储
- 实现候选蓝图列表接口

**交付物**：
- 章蓝图候选生成节点
- 候选列表接口

**验收标准**：
- 每章至少可生成 3 个候选蓝图
- 蓝图输出通过 schema 校验

### C-3 蓝图选择节点

**目标**：允许作者或编辑选定本章蓝图。

**任务内容**：
- 实现蓝图人工选择接口
- 实现选中蓝图锁定
- 实现选中蓝图状态切换

**交付物**：
- 蓝图选择 API / UI

**验收标准**：
- 一章只能有一个 active 蓝图
- 被选蓝图可进入后续流程

### C-4 场景拆解节点

**目标**：把章蓝图拆成可写作的场景序列。

**任务内容**：
- 接入 Scene Decomposer Agent
- 定义 SceneCard 输出结构
- 场景卡存储与查询
- 场景顺序控制

**交付物**：
- 场景拆解节点
- SceneCard 列表服务

**验收标准**：
- 章蓝图可稳定拆成多个场景卡
- 场景顺序与引用关系可追踪

### C-5 正文生成节点

**目标**：按场景序列生成章节草稿。

**任务内容**：
- 接入 Writer Agent
- 构建写作上下文装配器
- 按场景生成正文片段
- 拼装章节草稿
- 保存 DraftVersion

**交付物**：
- 写作上下文服务
- 正文生成节点
- 草稿版本存储

**验收标准**：
- 至少可生成一版完整草稿
- 草稿可追溯来源蓝图与场景卡

### C-6 章节摘要节点

**目标**：为后续章节与检索提供标准摘要。

**任务内容**：
- 接入 Summarizer Agent
- 定义章节摘要结构
- 保存章节摘要与状态提要

**交付物**：
- 摘要生成服务

**验收标准**：
- 每章生成结构化摘要
- 摘要可被下一章读取

## 7.4 阶段 D：Agent 接入与评审闭环

### D-1 Agent Runtime 基础接入

**目标**：建立统一 Agent 调用层。

**任务内容**：
- 定义 Agent Request / Response 协议
- 实现 Prompt 模板装配
- 实现上下文裁剪策略
- 实现结果校验与错误处理
- 实现 Agent 调用日志

**交付物**：
- Agent Runtime Service
- 统一调用 SDK / 服务层

**验收标准**：
- 任意 Agent 均通过统一入口调用
- 所有输出均可校验与记录

### D-2 Schema Gate 实现

**目标**：最先拦截结构性错误。

**任务内容**：
- 对 Agent 输出做 schema 校验
- 对章节草稿基础字段做格式校验
- 对对象引用做基础检查

**交付物**：
- Schema Gate

**验收标准**：
- 非法输出不能进入后续流程

### D-3 Canon Gate 实现

**目标**：检查设定、状态、时间线、关系冲突。

**任务内容**：
- 构建 Canon 校验输入
- 实现规则校验器
- 接入 Canon Review Agent
- 输出冲突报告与严重度分级

**交付物**：
- Canon Gate
- Canon Issue Report

**验收标准**：
- 能识别明显人物/设定/时间线冲突
- 冲突结果可结构化呈现

### D-4 Style Gate 实现

**目标**：完成基础文风与语言重复检查。

**任务内容**：
- 接入 Style Review Agent
- 实现基础重复表达检测
- 实现题材语体偏移检测
- 输出风格报告

**交付物**：
- Style Gate

**验收标准**：
- 可识别明显风格偏移与高频重复

### D-5 修订回路实现

**目标**：让闸门失败后可以修订而不是终止。

**任务内容**：
- 定义修订输入协议
- 接入 Rewriter 或 Writer 修订模式
- 保存修订版本
- 支持重新过闸

**交付物**：
- 修订节点
- DraftVersion 历史链

**验收标准**：
- 失败草稿可进入修订流程
- 修订后可重新进入闸门

### D-6 Change Proposal 节点实现

**目标**：从通过闸门的草稿中提取状态变化。

**任务内容**：
- 接入 Change Proposal Agent
- 定义状态变化抽取结构
- 形成 ChangeSet 草案

**交付物**：
- Change Proposal Service
- ChangeSet Draft 生成节点

**验收标准**：
- 通过的草稿能自动提取候选状态变化

## 7.6 阶段 F：联调、验收与样书测试

### F-1 流程联调

**目标**：打通端到端主流程。

**任务内容**：
- 端到端联调各节点
- 修复字段不一致问题
- 修复对象引用错误
- 修复 Agent 输出不稳定问题
- 修复 ChangeSet 应用错误

**交付物**：
- 可运行的端到端流程

**验收标准**：
- 能完整跑通至少一个章节闭环

### F-2 连续章节样书测试

**目标**：验证系统不是只会跑单章样例。

**任务内容**：
- 使用同一项目连续生成 5~10 章
- 记录人物一致性问题
- 记录设定冲突问题

## 12. 关键风险与应对

### 风险一：中枢未成型就先做写作 UI

后果：
- 前端能点，但系统不稳定
- 返工巨大

应对：
- 前两阶段禁止把重心放在界面美化上

### 风险二：Agent 输出结构不稳定

后果：
- 流程频繁断裂
- 数据污染

应对：
- 统一 schema gate
- 增加输出纠错层
- 严格限制输出格式

### 风险三：状态写入绕过 ChangeSet

后果：
- Canon 被污染
- 无法回滚

应对：
- 所有正史写入入口收口到服务层
- 禁止 Agent 直写数据库

### 风险四：题材配置与规则包界限模糊

后果：
- 后续难扩展
- 题材支持越来越乱

应对：
- 配置负责静态倾向
- 规则包负责动态判定

### 风险五：MVP 范围膨胀

后果：
- 长时间无法出可运行版本

应对：
- 所有新增需求按 P0 / P1 / P2 分级
- P0 外内容默认后置

## 14. V1 结论

MVP 阶段不是要把小说系统一次性做成“大而全平台”，而是要尽快验证以下关键命题：

**这个系统能否用“状态总线 + 结构化对象 + 工作流 + 闸门 + Agent 执行器”的方式，稳定支撑长篇小说的连续章节创作。**

因此，开发任务必须围绕最小闭环展开，而不是被 UI、平台化、复杂运营能力牵走。

MVP 的正确完成标志，不是功能菜单很多，而是：

- 章节闭环跑通；
- 正史写入可控；
- 多章连续创作不明显失控；
- 后续扩展有清晰地基。

这就是 V1 阶段最重要的工程目标。
