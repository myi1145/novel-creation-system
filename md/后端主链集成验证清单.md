# 后端主链集成验证清单（无前端）

## 文档目的

用于在**无前端**条件下，通过 Swagger/API/脚本验证当前后端是否满足主链闭环能力，按顺序覆盖：

- 单章闭环：目标 -> 蓝图 -> 场景 -> 草稿 -> Gate -> ChangeSet -> Publish -> Summary -> Derived Updates  
- 三章连续：continuity 注入、next_chapter_seed 承接、sequence runner 与 report  
- 安全边界：关键阻断规则与回滚最小闭环  
- Agent 稳定性：provider、生效状态、fallback、解析退化与错误定位

系统主链与 Canon 写入约束遵循 V1：Canon 只能通过 ChangeSet 写入。

---

## 验证前准备

### 1) 启动方式
- 调哪个接口/方法：启动 FastAPI 服务（建议 `uvicorn app.main:app --reload`）
- 传什么关键参数：无
- 要拿到哪些 ID：无
- 预期状态：服务可访问 `/docs` 和 `/api/v1/*`
- 失败先查哪里：`app/main.py` 路由挂载与 `api_prefix` 配置

### 2) 配置建议
- 调哪个接口/方法：准备 `.env`（尤其是 Agent 相关）
- 传什么关键参数：`AGENT_PROVIDER`, `AGENT_MODEL`, `AGENT_API_BASE_URL`, `AGENT_API_KEY`, `AGENT_FALLBACK_TO_MOCK`
- 要拿到哪些 ID：无
- 预期状态：provider 与 fallback 行为可控
- 失败先查哪里：`.env` 配置与 settings 默认值

### 3) Swagger 使用路径
- 调哪个接口/方法：优先在 Swagger 执行以下接口（统一前缀 `/api/v1`）
- 传什么关键参数：按各步骤填写
- 要拿到哪些 ID：`project_id`, `workflow_run_id`, `trace_id`, `draft_id`, `changeset_id`, `published_chapter_id`
- 预期状态：可完整串联主链
- 失败先查哪里：接口响应中的错误信息与 workflow 诊断接口

## 单章闭环验证清单

### Step 1：项目创建
- 调哪个接口/方法：`POST /api/v1/projects`
- 传什么关键参数：`project_name`, `premise`, `genre_id(可选)`
- 要拿到哪些 ID：`project_id`
- 预期状态：创建成功，可在 `GET /api/v1/projects` 查询
- 失败先查哪里：请求字段是否合法、是否缺必填

### Step 2：题材装载（建议）
- 调哪个接口/方法：`POST /api/v1/genres/load`，`GET /api/v1/genres`
- 传什么关键参数：`file_name`
- 要拿到哪些 ID：`genre_id`（用于核对）
- 预期状态：题材配置可读
- 失败先查哪里：题材文件名/数据格式

### Step 3：Canon 初始化
- 调哪个接口/方法：`POST /api/v1/canon/snapshots/init`
- 传什么关键参数：`project_id`, `title`, `initial_rules`, `initial_characters`
- 要拿到哪些 ID：`snapshot_id`, `version_no`
- 预期状态：项目存在 Canon 基线
- 失败先查哪里：`project_id` 是否存在

### Step 4：Chapter Goal
- 调哪个接口/方法：`POST /api/v1/chapters/goals`
- 传什么关键参数：`project_id`, `chapter_no`, `current_volume_goal`
- 要拿到哪些 ID：`chapter_goal_id`, `workflow_run_id`, `trace_id`
- 预期状态：章目标创建成功
- 失败先查哪里：项目是否存在、chapter_no 是否冲突

### Step 5：候选蓝图生成
- 调哪个接口/方法：`POST /api/v1/chapters/blueprints/generate`
- 传什么关键参数：`project_id`, `chapter_goal_id`, `candidate_count`
- 要拿到哪些 ID：`blueprint_id[]`
- 预期状态：返回候选蓝图集合
- 失败先查哪里：`chapter_goal_id` 是否有效

### Step 6：正式蓝图选择
- 调哪个接口/方法：`POST /api/v1/chapters/blueprints/select`
- 传什么关键参数：`project_id`, `blueprint_id`, `selected_by`, `selection_reason(可选)`
- 要拿到哪些 ID：`selected_blueprint_id`
- 预期状态：蓝图 selected；后续可拆场景
- 失败先查哪里：`selected_by` 是否为空；是否存在“已选蓝图已产出场景/草稿”冲突

### Step 7：Scene 生成
- 调哪个接口/方法：`POST /api/v1/chapters/scenes/decompose`
- 传什么关键参数：`project_id`, `blueprint_id`
- 要拿到哪些 ID：`scene_id[]`
- 预期状态：场景卡生成成功
- 失败先查哪里：该 blueprint 是否已 selected

### Step 8：Draft 生成
- 调哪个接口/方法：`POST /api/v1/chapters/drafts/generate`
- 传什么关键参数：`project_id`, `blueprint_id`, `scene_ids(可选)`
- 要拿到哪些 ID：`draft_id`
- 预期状态：草稿状态为 drafting，metadata 含 provider/call 信息
- 失败先查哪里：是否已有 SceneCard（无场景硬阻断）

### Step 9：Gate 审查
- 调哪个接口/方法：`POST /api/v1/gates/reviews`
- 传什么关键参数：`project_id`, `draft_id`, `gate_names`（建议含 schema/canon/narrative/style）
- 要拿到哪些 ID：`gate_review_id[]`, `pass_status`, `highest_severity`
- 预期状态：通过则进入审稿通过态；失败则进入 review_failed/attention
- 失败先查哪里：Gate issues、evidence_refs、draft 状态

### Step 10：ChangeSet Proposal
- 调哪个接口/方法：`POST /api/v1/chapters/drafts/{draft_id}/changeset-proposals/generate`
- 传什么关键参数：`project_id`, `rationale(可选)`, `auto_create_changeset`
- 要拿到哪些 ID：提议对象（含 patch_operations）
- 预期状态：生成有效 patch_operations
- 失败先查哪里：是否满足 proposal 前置 Gate（schema/canon/narrative）

### Step 11：ChangeSet propose / approve / apply
- 调哪个接口/方法：
  - `POST /api/v1/changesets/propose`
  - `POST /api/v1/changesets/{changeset_id}/approve`
  - `POST /api/v1/changesets/{changeset_id}/apply`
- 传什么关键参数：
  - propose：`project_id`, `source_type`, `source_ref=draft_id`, `rationale`, `patch_operations`
  - approve：`approved_by`
- 要拿到哪些 ID：`changeset_id`, `base_snapshot_id`, `result_snapshot_id`
- 预期状态：状态流转 `proposed -> approved -> applied`，并产出新 snapshot
- 失败先查哪里：changeset 当前状态、required_gate_names 是否满足、Canon 是否初始化

### Step 12：Publish
- 调哪个接口/方法：`POST /api/v1/chapters/drafts/publish`
- 传什么关键参数：`project_id`, `draft_id`, `published_by`, `publish_title(可选)`, `notes(可选)`
- 要拿到哪些 ID：`published_chapter_id`, `publish_record_id`
- 预期状态：发布成功；Draft 进入 published
- 失败先查哪里：draft 状态前置、applied changeset、publish_gate 结果

### Step 13：Chapter Summary
- 调哪个接口/方法：
  - `POST /api/v1/chapters/published/{published_chapter_id}/summary/generate`
  - `GET /api/v1/chapters/published/{published_chapter_id}/summary`
- 传什么关键参数：`project_id`（GET 必填）
- 要拿到哪些 ID：`published_chapter_id` 对应 summary；关键字段 `next_chapter_seed`
- 预期状态：可获得承接下一章的稳定摘要
- 失败先查哪里：published_chapter 是否存在、上下文对象是否完整

### Step 14：Post-Publish Derived Updates
- 调哪个接口/方法：
  - `POST /api/v1/chapters/published/{published_chapter_id}/derived-updates/run`
  - `GET /api/v1/chapters/published/{published_chapter_id}/derived-updates`
- 传什么关键参数：`project_id`, `task_names`
- 要拿到哪些 ID：`task_id[]`, 任务状态
- 预期状态：`completed` 或 `completed_with_warnings`
- 失败先查哪里：单任务 `details.error`、发布元数据

## 三章连续验证清单

### Step 1：执行三章序列
- 调哪个接口/方法：`POST /api/v1/workflows/chapter-sequence/execute`
- 传什么关键参数：`project_id`, `start_chapter_no`, `chapter_count=3`, `current_volume_goal`
- 要拿到哪些 ID：序列 `workflow_run_id`，每章 `chapter_workflow_run_id`
- 预期状态：返回每章 `stage_status/next_action`
- 失败先查哪里：sequence run 是否 paused/manual_review/completed

### Step 2：continuity pack 注入核验
- 调哪个接口/方法：
  - `GET /api/v1/chapters/projects/{project_id}/continuity-pack?target_chapter_no=N`
  - `GET /api/v1/workflows/chapter-sequence/reports/{workflow_run_id}`
- 传什么关键参数：`target_chapter_no`, `force_refresh(可选)`
- 要拿到哪些 ID：`source_snapshot_id`, `source_published_chapter_id`
- 预期状态：continuity 输入完整，report 可见 continuity 相关统计
- 失败先查哪里：上一章发布与 summary 是否存在

### Step 3：next_chapter_seed 承接核验
- 调哪个接口/方法：`GET /api/v1/chapters/published/{id}/summary` + continuity-pack
- 传什么关键参数：`project_id`
- 要拿到哪些 ID：上一章 `published_chapter_id`
- 预期状态：上一章 `next_chapter_seed` 在下一章 continuity 输入中可见
- 失败先查哪里：summary 是否缺失或需强制重算

### Step 4：sequence runner 行为核验
- 调哪个接口/方法：`POST /api/v1/workflows/chapter-sequence/execute`
- 传什么关键参数：`stop_on_attention`, `stop_on_failure`, `advance_only_on_completed`
- 要拿到哪些 ID：序列 `workflow_run_id`
- 预期状态：停机/推进与开关一致
- 失败先查哪里：返回的 `stop_reason`, `next_action`, 每章 `stage_status`

### Step 5：sequence report 解读
- 调哪个接口/方法：`GET /api/v1/workflows/chapter-sequence/reports/{workflow_run_id}`
- 传什么关键参数：`workflow_run_id`
- 要拿到哪些 ID：报告内各章 run id
- 预期状态：可读出 gate 分布、失败章、continuity 覆盖与建议项
- 失败先查哪里：workflow_run_id 是否对应 sequence workflow

## 安全边界验证清单

### 边界 1：无 SceneCard 不能生成 Draft
- 调哪个接口/方法：`POST /api/v1/chapters/drafts/generate`（在未生成 scenes 的前提下）
- 传什么关键参数：`project_id`, `blueprint_id`
- 要拿到哪些 ID：`blueprint_id`
- 预期状态：冲突错误（禁止）
- 失败先查哪里：该 blueprint 是否已存在 SceneCard

### 边界 2：Gate 未通过不能推进 ChangeSet / Publish
- 调哪个接口/方法：
  - `POST /api/v1/changesets/{id}/apply`
  - `POST /api/v1/chapters/drafts/publish`
- 传什么关键参数：`changeset_id`, `draft_id`
- 要拿到哪些 ID：`changeset_id`, `draft_id`
- 预期状态：被 Gate 前置条件或状态前置阻断
- 失败先查哪里：Gate 通过记录、draft 当前状态

### 边界 3：publish gate 前移是否生效
- 调哪个接口/方法：`POST /api/v1/workflows/chapter-cycle/execute`
- 传什么关键参数：`auto_apply_changeset=true`, `auto_publish=true`
- 要拿到哪些 ID：`workflow_run_id`, `draft_id`, `changeset_id`
- 预期状态：publish_gate 预检失败会阻止 apply，返回 `next_action=revise_draft`
- 失败先查哪里：workflow run `current_step` 与 gate 结果

### 边界 4：rollback 最小闭环可用
- 调哪个接口/方法：`POST /api/v1/changesets/{changeset_id}/rollback`
- 传什么关键参数：`rolled_back_by`, `reason`
- 要拿到哪些 ID：`rollback_snapshot_id`（响应模型中）
- 预期状态：applied changeset 可回滚为 rolled_back，并生成 rollback snapshot
- 失败先查哪里：changeset 是否已绑定发布记录（绑定则拒绝）

### 边界 5：ChangeSet 仍是 Canon 唯一写入口
- 调哪个接口/方法：检查链路动作（changeset apply + publish）
- 传什么关键参数：`changeset_id`, `draft_id`
- 要拿到哪些 ID：`result_snapshot_id`
- 预期状态：Canon snapshot 更新发生在 apply；publish 依赖 applied changeset
- 失败先查哪里：是否存在绕开 changeset 的写路径（当前代码未提供）

## Agent 稳定性验证清单

### Step 1：openai_compatible provider 生效验证
- 调哪个接口/方法：`GET /api/v1/workflows/agent-gateway/status`
- 传什么关键参数：`.env` 中 `AGENT_PROVIDER=openai_compatible` + base_url/api_key/model
- 要拿到哪些 ID：无
- 预期状态：`configured_provider` 为 openai_compatible；active_provider 与 fallback 状态可观测
- 失败先查哪里：.env 配置与 provider 初始化条件

### Step 2：fallback 关闭验证
- 调哪个接口/方法：`GET /api/v1/workflows/agent-gateway/governance` + `GET /api/v1/workflows/agent-calls`
- 传什么关键参数：`AGENT_FALLBACK_TO_MOCK=false`
- 要拿到哪些 ID：`agent_call_log_id`
- 预期状态：失败调用应表现为 `call_status=error`，不应有 fallback_success
- 失败先查哪里：governance 快照中的 `fallback_enabled`

### Step 3：parse/degraded/reask/fallback 可追踪验证
- 调哪个接口/方法：
  - `GET /api/v1/workflows/agent-calls`
  - `GET /api/v1/workflows/agent-calls/stats`
  - `GET /api/v1/workflows/agent-calls/{log_id}`
- 传什么关键参数：`project_id`, `workflow_run_id`, `limit`
- 要拿到哪些 ID：`log_id`
- 预期状态：可看到 `parse_decision/parse_issue_count/parse_reask_count/parse_degraded/fallback_used`
- 失败先查哪里：对应 log 的 `response_summary.parse_report`

### Step 4：provider 错误定位验证
- 调哪个接口/方法：
  - `GET /api/v1/workflows/diagnostics/overview`
  - `GET /api/v1/workflows/diagnostics/runs/{workflow_run_id}`
  - `GET /api/v1/workflows/agent-gateway/governance`
- 传什么关键参数：`project_id` 或 `workflow_run_id`
- 要拿到哪些 ID：`workflow_run_id`
- 预期状态：可区分 failed_agent_calls / parse_error / rate_limit / 熔断状态
- 失败先查哪里：provider_circuit_state、recent_issues、run nodes

## 最小执行顺序

1. `POST /api/v1/projects`（创建项目，拿 `project_id`）
2. `POST /api/v1/canon/snapshots/init`（初始化 Canon）
3. `POST /api/v1/workflows/chapter-cycle/execute`（先跑到蓝图选择或更后阶段）
4. 若停在蓝图选择：`POST /api/v1/chapters/blueprints/select` 后继续 `chapter-cycle/execute`，直至 changeset applied
5. `POST /api/v1/chapters/drafts/publish`，再查 summary 与 derived-updates 完成闭环

## 常见失败定位路径

1. **主链中断/卡住**
   - 调哪个接口/方法：`GET /api/v1/workflows/runs/{workflow_run_id}` + `GET /api/v1/workflows/diagnostics/runs/{workflow_run_id}`
   - 传什么关键参数：`workflow_run_id`
   - 要拿到哪些 ID：同一 run 的 trace 节点
   - 预期状态：定位 `current_step` 与 `next_action`
   - 失败先查哪里：run_metadata 与 recent_issues

2. **Draft 生成失败**
   - 调哪个接口/方法：`GET /api/v1/chapters/blueprints`、重新检查 scenes
   - 传什么关键参数：`project_id`, `chapter_goal_id`
   - 要拿到哪些 ID：`selected_blueprint_id`, `scene_id[]`
   - 预期状态：selected blueprint + scene 已齐备
   - 失败先查哪里：SceneCard 缺失阻断

3. **ChangeSet 失败**
   - 调哪个接口/方法：`GET /api/v1/workflows/diagnostics/runs/{id}` + 重新跑 Gate
   - 传什么关键参数：`workflow_run_id`, `draft_id`
   - 要拿到哪些 ID：`gate_review_id[]`, `changeset_id`
   - 预期状态：满足 proposal/apply 的 Gate 前置
   - 失败先查哪里：required gates 与 changeset 当前状态

4. **Publish 失败**
   - 调哪个接口/方法：`POST /api/v1/chapters/drafts/publish` 返回错误后查 run diagnostics
   - 传什么关键参数：`project_id`, `draft_id`
   - 要拿到哪些 ID：`changeset_id`, `publish_gate_review_id`
   - 预期状态：满足发布前置（canon_applied + applied changeset + publish_gate）
   - 失败先查哪里：draft 状态、changeset 状态、publish_gate issues

5. **Agent 调用异常**
   - 调哪个接口/方法：`GET /api/v1/workflows/agent-calls` + `.../agent-calls/stats` + `.../agent-gateway/governance`
   - 传什么关键参数：`project_id`, `workflow_run_id`
   - 要拿到哪些 ID：`log_id`
   - 预期状态：可定位 provider/fallback/parse 退化原因
   - 失败先查哪里：`error_type`, `rate_limited`, `parse_report`, 熔断状态
