## 2. 文档结论先行

本系统采用以下扩展路线：

**通用底座负责控制机制，题材配置层负责创作倾向，规则包负责硬约束与专用能力。**

这意味着：

- 题材差异不应侵入 Story Canon、工作流、ChangeSet、发布机制等核心底座
- 题材差异优先通过配置表达，而不是优先通过 Prompt 硬写死
- 只有当某种题材差异具有“强约束、强校验、强专用流程”的特征时，才进入规则包层
- 一个题材模板必须是“可组合、可校验、可继承、可版本化”的配置单元，而不是一组散落 prompt 文件

---

## 3. 设计目标

### 3.1 本层要解决的问题

题材配置层与规则包层主要解决以下问题：

1. **多题材扩展问题**
   - 修仙、娱乐圈、校园、都市、悬疑、科幻等题材不能靠复制系统代码来支持
   - 不同题材必须能在统一底座上运行

2. **题材风格控制问题**
   - 不同题材对人物关系、冲突形态、节奏曲线、语言风格、读者期待都不同
   - 这些差异必须通过显式配置承载，而不是依赖模型“自行理解”

3. **专用规则约束问题**
   - 某些题材有明显的世界规则和写作禁忌，例如修仙境界体系、悬疑信息披露顺序、娱乐圈舆论传播链
   - 这些内容需要成为可执行规则，而不能只存在于说明文字中

4. **复用与演化问题**
   - 系统应允许从“通用修仙模板”派生出“宗门群像修仙模板”“黑暗修仙模板”等子模板
   - 不同项目应支持模板版本锁定、升级与局部覆盖

### 3.2 本层不解决的问题

V1 阶段，本层不负责：

- 替代 Canon 状态模型
- 替代工作流编排
- 替代 Agent 契约
- 替代对象层 Schema 定义
- 处理所有 UI 层编辑体验问题
- 构建复杂插件市场

## 4. 总体分层与职责边界

### 4.1 三层扩展模型

系统扩展能力分为三层：

#### 第一层：通用底座（Core Foundation）

负责与题材无关的共性能力：

- Story Canon / Story State 机制
- ChangeSet 写入协议
- 结构化对象层
- 章节循环工作流
- 多层质量闸门
- Agent 基础契约
- 发布、回滚、审计、快照
- 检索索引与派生更新

这层是稳定内核，不应因新增题材而重写。

#### 第二层：题材配置层（Genre Configuration Layer）

负责描述“这个题材通常怎样写”。

包括：

- 世界观倾向
- 主线推进范式
- 常见冲突类型
- 节奏模板
- 角色关系偏好
- 叙事视角建议
- 风格与语体约束
- 常用对象模板
- 禁忌项与偏离提醒

这层主要通过 **声明式配置** 实现。

#### 第三层：规则包 / 插件层（Rule Pack / Plugin Layer）

负责描述“这个题材有哪些必须执行的约束或专用能力”。

包括：

- 专用校验器
- 专用评分器
- 专用对象扩展字段
- 专用流程节点钩子
- 专用解析器 / 提取器 / 映射器
- 专用节奏策略

这层用于承载“纯配置表达不了”的能力。

## 5. 题材配置层设计

### 5.1 定义

题材配置层是围绕某一题材或某一细分流派，对系统运行时行为进行声明式约束与偏好注入的一组结构化配置。

其核心目标不是“替模型写文”，而是：

- 限定生成空间
- 提供稳定偏好
- 为工作流和质量闸门提供上下文
- 将题材经验沉淀为可复用资产

### 5.2 题材配置单元

V1 阶段，题材配置单元统一称为：

**Genre Profile（题材画像）**

每个 Genre Profile 至少包含：

- profile_id
- profile_name
- version
- base_profile_id（可选）
- supported_modes
- description
- tags
- status
- compatibility
- config_sections

### 5.3 题材配置的标准组成

一个完整的题材配置应拆分为以下子配置节：

#### 1）世界观倾向配置（world_profile）

用于描述该题材的世界观共性与设定重心。

示例字段：

- era_style：古风 / 现代 / 未来 / 架空
- world_openness：封闭 / 半开放 / 开放世界
- power_system_required：是否强依赖力量体系
- institution_density：宗门 / 学校 / 公司 / 家族等组织密度
- realism_level：写实度
- mystery_density：秘密与未知元素密度
- supernatural_level：超自然强度

#### 2）叙事范式配置（narrative_profile）

用于描述该题材常见的推进方式。

示例字段：

- mainline_pattern：成长线 / 调查线 / 经营线 / 对抗线 / 生存线
- chapter_drive_type：事件驱动 / 关系驱动 / 谜题驱动 / 目标驱动
- reveal_pattern：前置解释少 / 中期逐步揭示 / 后期集中揭示
- suspense_expectation：低 / 中 / 高
- payoff_cycle：短周期 / 中周期 / 长周期

#### 3）角色关系配置（relationship_profile）

用于约束该题材下的人物关系组织方式。

示例字段：

- protagonist_mode：单主角 / 双主角 / 群像
- romance_policy：单女主 / 多感情线 / 无恋爱主线
- ensemble_weight：群像权重
- mentor_presence：导师角色权重
- rivalry_density：竞争关系密度
- intimacy_growth_speed：关系推进速度

#### 4）节奏模板配置（pacing_profile）

用于提供题材级节奏偏好。

示例字段：

- hook_frequency
- conflict_curve_pattern
- reveal_density_target
- action_scene_ratio
- daily_life_ratio
- emotional_peak_spacing
- arc_length_preference

#### 5）语言与风格配置（style_profile）

用于提供题材级表达约束。

示例字段：

- narration_distance：近距离 / 中距离 / 远距离
- language_density：简洁 / 中等 / 偏繁复
- metaphor_level：低 / 中 / 高
- dialogue_ratio
- exposition_tolerance
- terminology_density
- forbidden_tone_markers
- preferred_rhythm_markers

#### 6）对象模板配置（object_template_profile）

用于告诉系统：该题材通常需要哪些对象模板与默认字段。

示例字段：

- required_object_types
- optional_object_types
- default_character_archetypes
- default_location_archetypes
- default_open_loop_types
- rule_card_extensions
- scene_card_extensions

#### 7）禁忌项配置（taboo_profile）

用于声明该题材的显式禁区或偏离提醒。

示例字段：

- hard_forbidden_patterns
- soft_warning_patterns
- genre_breaking_elements
- immersion_breaking_words
- default_content_boundaries

## 6. 规则包层设计

### 6.1 定义

规则包是围绕某一题材、细分流派、写作目标或项目约束，对系统注入的一组 **可执行规则能力**。

规则包不同于普通配置的地方在于：

- 可以执行校验
- 可以执行打分
- 可以参与流程节点
- 可以扩展对象字段
- 可以提供运行时决策逻辑

### 6.2 规则包适用边界

以下能力应优先放入规则包，而不是仅写入配置：

1. 需要程序执行判断的硬约束
2. 需要对对象或文本进行结构化解析的能力
3. 需要接入工作流节点的专用逻辑
4. 需要复用的题材专用质量检查器
5. 需要影响节奏打分与候选筛选的专用评分逻辑

### 6.3 规则包标准结构

每个 Rule Pack 至少包含：

- pack_id
- pack_name
- version
- target_profiles
- dependencies
- compatible_core_versions
- enabled_by_default
- description
- validators
- scorers
- extractors
- object_extensions
- workflow_hooks
- prompt_fragments（可选）
- test_cases

### 6.4 规则包能力类型

#### 1）Validator（校验器）

用于检查对象或文本是否违反规则。

示例：

- 修仙境界跃迁校验器
- 悬疑线索泄露顺序校验器
- 娱乐圈传播链合理性校验器
- 校园人物年龄与学段一致性校验器

#### 2）Scorer（评分器）

用于对候选方案、章节蓝图、场景拆解或正文进行题材适配性打分。

示例：

- 宗门群像适配评分器
- 悬疑张力评分器
- 感情线推进合理性评分器

#### 3）Extractor（提取器）

用于从正文或对象中抽取专用结构。

示例：

- 修仙术语提取器
- 舆论事件链提取器
- 线索对象提取器

#### 4）Object Extensions（对象扩展）

用于给通用对象增加题材专用字段。

示例：

- CharacterCard 增加 cultivation_stage
- RuleCard 增加 breakthrough_constraints
- SceneCard 增加 clue_visibility_level

#### 5）Workflow Hooks（流程钩子）

用于在特定工作流节点前后注入能力。

示例：

- 在候选章蓝图过滤阶段追加“题材硬规则筛查”
- 在 Canon Gate 后追加“修仙力量体系一致性检查”
- 在发布前追加“关键伏笔暴露度校验”

## 7. 配置与规则包的职责划分标准

### 7.1 优先级原则

在设计时，必须遵循以下优先级：

**能用配置表达的，先不用规则包；必须执行逻辑的，再进入规则包。**

### 7.2 划分判断表

#### 适合放入题材配置层的内容

- 倾向性
- 偏好性
- 风格性
- 推荐性
- 缺省值
- 可枚举模板
- 可声明的节奏参数
- 禁忌提醒词

#### 适合放入规则包层的内容

- 必须满足的程序性约束
- 需要运行判断的逻辑
- 需要参与工作流的专用执行
- 需要扩展对象结构的能力
- 需要被单元测试验证的题材能力

## 8. 题材配置的继承、覆盖与组合机制

### 8.1 继承机制

题材配置允许从一个基础 Profile 继承。

示例：

- `xianxia_base`
- `xianxia_sect_ensemble`
- `xianxia_dark_revision`

继承规则：

- 子 Profile 默认继承父 Profile 的全部配置节
- 子 Profile 可对局部字段进行覆盖
- 子 Profile 不允许删除核心必填节
- 子 Profile 的覆盖必须保留可追踪来源

### 8.2 覆盖机制

覆盖类型分为：

- replace：整体替换
- merge：字段合并
- append：列表追加
- disable：显式禁用父级项

系统必须明确记录覆盖来源，避免后期出现“为什么这个项目行为异常却追不到来源”的问题。

### 8.3 组合机制

一个项目允许组合：

- 1 个主 Genre Profile
- 0~N 个附加风格包
- 0~N 个规则包

但必须遵循以下限制：

- 主 Profile 只能有一个
- 规则包之间若存在冲突，必须在项目初始化时显式解决
- 组合后的生效配置必须产出一份“Resolved Genre Config”供运行时使用

## 9. 运行时加载模型

### 9.1 加载时机

题材配置与规则包在以下阶段参与系统运行：

1. 项目初始化阶段
2. 卷规划阶段
3. 章节目标生成阶段
4. 候选蓝图生成与过滤阶段
5. 场景拆解阶段
6. 正文生成阶段
7. 多层质量闸门阶段
8. 发布前检查阶段

### 9.2 运行时生效流程

标准加载流程如下：

1. 选择主 Genre Profile
2. 选择附加风格包与规则包
3. 进行版本兼容性检查
4. 进行依赖关系解析
5. 进行冲突检查
6. 生成 Resolved Genre Config
7. 注入工作流上下文
8. 各节点按需读取配置与规则能力

### 9.3 Resolved Genre Config

运行时统一生效配置必须是一个展开后的只读对象。

该对象至少包含：

- resolved_profile_id
- source_profiles
- source_rule_packs
- effective_world_profile
- effective_narrative_profile
- effective_relationship_profile
- effective_pacing_profile
- effective_style_profile
- effective_object_template_profile
- effective_taboo_profile
- effective_extensions
- effective_hooks

Resolved Genre Config 一旦在项目或卷级上下文中锁定，未经显式变更不得随意漂移。

## 10. 与其他系统层的协作关系

### 10.1 与 Story State / Canon 的关系

题材配置层与规则包层：

- 不直接替代 Canon State
- 不直接持有正史
- 可以影响对象默认结构与状态校验规则
- 可以限制某些状态变化是否合法

### 10.2 与结构化对象层的关系

题材配置可以：

- 指定对象默认模板
- 指定对象必需字段
- 指定对象扩展字段
- 指定对象间推荐关系

规则包可以：

- 校验对象内容
- 扩展对象结构
- 从对象中提取专用特征

### 10.3 与工作流层的关系

题材配置主要为工作流提供：

- 缺省参数
- 候选方案偏好
- 节奏目标
- 风格要求

规则包主要为工作流提供：

- 前置校验
- 后置检查
- 特定节点挂钩
- 分流判断

### 10.4 与 Agent 的关系

Agent 不应自行定义题材规则。

Agent 只能：

- 读取题材配置
- 调用规则包能力
- 在输出中遵循相关约束

Agent 不能：

- 在未声明的情况下自行发明题材核心规则
- 绕开规则包进行违规写入
- 以模型主观判断替代已配置的题材约束

### 10.5 与质量闸门的关系

题材配置与规则包必须为质量闸门提供题材感知能力。

例如：

- Style Gate 读取 style_profile
- Narrative Gate 读取 narrative_profile 与 pacing_profile
- Canon Gate 调用题材专用 Validator
- Publish Gate 读取 taboo_profile 与关键硬约束

## 11. 题材模板包的目录与资源组织建议

### 11.1 标准资源单元

V1 建议每个题材模板包至少包含：

- `profile.yaml`：题材画像主配置
- `style.yaml`：风格配置
- `pacing.yaml`：节奏配置
- `objects.yaml`：对象模板配置
- `taboos.yaml`：禁忌项配置
- `rules/`：规则包声明与实现映射
- `prompts/`：仅作为辅助片段，不作为唯一真相
- `examples/`：少量高质量范例
- `tests/`：配置与规则包测试用例
- `manifest.json`：包元数据与版本声明

### 11.2 资源组织原则

必须遵循：

- 配置与 Prompt 分离
- 配置与规则实现分离
- 模板资源与项目实例分离
- 范例仅作参考，不直接等价于题材规则

## 12. 校验与测试要求

### 12.1 配置层校验

每个 Genre Profile 在启用前必须通过：

- Schema 校验
- 枚举值校验
- 必填节校验
- 继承链校验
- 覆盖来源校验
- 冲突检测

### 12.2 规则包校验

每个 Rule Pack 在启用前必须通过：

- 依赖校验
- 版本兼容性校验
- 钩子注册校验
- 对象扩展一致性校验
- 测试用例通过校验

### 12.3 运行时测试要求

每种主题材模板至少应有：

- 项目初始化测试
- 候选蓝图过滤测试
- Style Gate 适配测试
- Canon Gate 专用规则测试
- 发布前禁忌项测试

## 14. MVP 范围

### 14.1 P0 必做

V1 MVP 阶段，本层必须交付以下能力：

1. 主 Genre Profile 机制
2. 配置继承与覆盖机制
3. Resolved Genre Config 生成机制
4. 至少 1 套题材主模板（建议修仙）
5. 至少 2 个题材专用 Rule Pack
6. Style Gate / Narrative Gate / Canon Gate 对题材配置的读取能力
7. 项目初始化时的配置校验与冲突检查

### 14.2 P1 可延后

以下能力可延后：

- 多模板可视化编辑器
- 模板市场与模板分享
- 动态在线热更新
- 复杂的规则包组合推荐
- 大规模模板版本迁移工具
- 多租户模板管理后台

## 15. 关键风险与控制策略

### 15.1 风险一：题材能力退化成 Prompt 文件夹

表现：

- 配置不存在结构化语义
- 实际逻辑都写进 Prompt
- 无法校验、无法继承、无法比较

控制：

- 强制 Genre Profile 结构化
- Prompt 仅作辅助，不作唯一配置来源

### 15.2 风险二：规则包滥用，导致系统重新碎片化

表现：

- 一点小偏好也写成插件
- 系统越来越重
- 维护成本持续膨胀

控制：

- 能声明的不编程
- 能配置的不写插件
- 仅将强逻辑约束放入 Rule Pack

### 15.3 风险三：多规则包冲突，运行时行为不可预测

表现：

- 同一对象被多个规则包重复扩展
- 同一工作流节点被多个包写入冲突逻辑

控制：

- 启用前做依赖与冲突检查
- 生成唯一 Resolved Genre Config
- 记录全部来源链路

### 15.4 风险四：题材配置漂移，项目中途行为变化

表现：

- 同一个项目写到中后期，模板行为悄悄变化
- 章节前后风格和规则不一致

控制：

- 项目初始化时锁定模板版本
- 卷级或项目级配置变更必须走显式审批

## 16. V1 定版结论

本系统对多题材扩展的正式路线定为：

**以通用底座承载控制中枢，以题材配置层承载题材偏好，以规则包层承载可执行题材能力。**

其中：

- 底座负责稳定
- 配置层负责差异化表达
- 规则包负责可执行约束

任何新增题材都不得绕过这一路线，直接侵入核心控制架构。

系统未来支持多题材的关键，不在于“多写几套 Prompt”，而在于：

- 是否形成可继承的 Genre Profile
- 是否形成可测试的 Rule Pack
- 是否能在运行时稳定解析为 Resolved Genre Config
- 是否能把题材约束稳定注入工作流、Agent 与质量闸门

这一路线一旦成立，系统就能从“单题材项目工程”升级为“多题材可持续演化的创作控制平台”。
