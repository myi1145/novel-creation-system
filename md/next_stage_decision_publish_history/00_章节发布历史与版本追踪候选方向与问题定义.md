# 00_章节发布历史与版本追踪候选方向与问题定义

## 1. 文档目的

在“发布前一致性验收 / 章节发布准入”与“作者端术语中文化收口”完成并签收后，明确下一阶段候选方向，并给出当前阶段最终问题定义，避免继续在已签收方向重复扩展。

## 2. 当前已完成能力回顾

当前系统已完成并签收以下能力：
- 文本级、蓝图级、场景级人工修订闭环；
- 下游依赖失效提示与人工确认重跑；
- 发布前一致性验收 / 章节发布准入最小闭环；
- 作者端术语中文化与流程说明收口。

以上能力已构成“发布前可控”基线，但尚未形成“发布后可追踪”基线。

## 3. 当前发布后追踪断点

当前主要断点集中在发布后可见性与可解释性：
1. 作者难以直接查看某一章节的历史发布序列；
2. 单次发布对应的来源草稿、ChangeSet、发布时间等信息缺少统一追踪视图；
3. 发布版本与当前工作态之间是否已产生差异，缺少稳定提示口径；
4. 当作者准备再次发布时，缺少“当前内容是否晚于已发布版本”的低成本判断。

## 4. 候选方向对比

### 4.1 候选方向 A：发布历史列表
- 价值：快速回答“这一章发布过几次、分别是什么时候”。
- 风险：若缺少来源字段，将退化为仅时间线记录。

### 4.2 候选方向 B：发布版本详情
- 价值：回答“这次发布来自哪个 ChapterDraft、哪个 ChangeSet、哪次准入判断”。
- 风险：若字段未定义最小集，容易膨胀为全量审计系统。

### 4.3 候选方向 C：当前工作态与已发布态对比提示
- 价值：回答“当前是否已经晚于最近发布版本，是否需要重新发布”。
- 风险：若直接进入复杂 diff，会超出本阶段边界。

### 4.4 候选方向 D：发布来源追踪
- 价值：建立 Publish 与 ChapterDraft / ChangeSet / ReleaseReadiness 的引用链，提升可解释性。
- 风险：若追求全链路追溯明细，可能提前进入实现复杂化。

### 4.5 候选方向 E：复杂版本回滚
- 价值：支持历史版本回退。
- 风险：涉及高风险语义变更，不适合作为当前阶段优先方向。

## 5. 本阶段建议优先方向

建议优先组合方向为：
- **发布历史列表（A）** + **发布版本详情（B）** + **发布来源追踪（D）**；
- 并以 **当前工作态与已发布态差异提示（C）** 作为次方向的最小提示能力。

该组合可在不改 Publish 核心语义的前提下，先补齐“可见、可追踪、可解释”的最小能力定义。

## 6. 本阶段暂不优先方向

本阶段暂不优先：
- 复杂版本回滚（E）；
- 跨版本复杂 diff 对比；
- 多章节发布历史总览大屏；
- 自动发布与自动回滚策略。

## 7. 最终问题定义

本阶段最终问题定义为：

**在不改变既有主链语义与入史边界的前提下，补齐章节发布后“发布历史可查看、发布来源可追踪、当前工作态是否晚于已发布态可提示”的最小立项定义。**

重点结论：

**当前最需要解决的不是继续生成内容，而是让作者知道“发布了什么版本、从哪里来、现在是否已经落后于工作态”。**
