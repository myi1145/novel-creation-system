# 01_章节发布历史与版本追踪方向确认

## 1. 文档目的

在候选方向基础上完成方向拍板，明确主方向、次方向与暂不采用方向，作为阶段目标与 MVP 范围定义的上游依据。

## 2. 结论先行

下一阶段主方向确认为：**章节发布历史与版本追踪**。  
下一阶段次方向确认为：**工作态与发布态差异提示**。

## 3. 主方向

### 3.1 章节发布历史与版本追踪

主方向定义：
- 建立章节级发布历史查看口径；
- 建立单次发布记录的来源追踪口径；
- 明确发布版本与 ChapterDraft / ChangeSet / ReleaseReadiness 的最小关联关系。

主方向目标不是“完整版本管理平台”，而是先解决“发布后可见、可追踪、可解释”。

## 4. 次方向

### 4.1 工作态与发布态差异提示

次方向定义：
- 提供“当前工作态是否已晚于最近发布版本”的最小提示语义；
- 为作者提供“是否需要重新发布”的决策辅助。

次方向不包含复杂文本差异比对，仅做最小状态提示。

## 5. 暂不采用方向

本阶段暂不采用以下方向：
- 完整版本回滚；
- 复杂 diff 对比；
- 多章节发布历史大屏；
- 自动发布策略。

## 6. 为什么现在做这个方向

原因如下：
1. 发布前一致性验收与术语收口阶段已完成签收，继续扩展同一方向边际收益下降；
2. 当前短板已从“发布前是否可发布”转移为“发布后是否可追踪”；
3. 若缺少发布后追踪能力，作者无法稳定判断历史来源与再发布必要性，影响后续协作效率与审计透明度；
4. 本方向可复用现有 Publish / ChangeSet / ReleaseReadiness 结果，不需要重写架构。

## 7. 与现有 Publish / ChangeSet / ReleaseReadiness 的关系

- 与 Publish 的关系：以既有发布记录为基础补齐查看与关联语义，不改 Publish 核心语义；
- 与 ChangeSet 的关系：明确发布记录可追溯至对应 ChangeSet 引用，不改变 ChangeSet 审批/应用边界；
- 与 ReleaseReadiness 的关系：将发布前一致性验收结果作为发布来源解释的一部分，不回退到“重新定义准入规则”。

## 8. 一句话拍板

**下一阶段先让作者看清“这一章发布过什么版本、来自哪里、现在是否需要重新发布”。**
