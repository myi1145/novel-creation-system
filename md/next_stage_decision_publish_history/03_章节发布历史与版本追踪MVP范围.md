# 03_章节发布历史与版本追踪MVP范围

## 1. 文档目的

定义“章节发布历史与版本追踪”阶段若进入实现时的最小 MVP 范围，确保可执行且可验收。

## 2. MVP 结论先行

MVP 仅覆盖“发布后最小可追踪能力”，不扩展为完整版本管理系统。

## 3. MVP 必须覆盖

MVP 必须覆盖以下能力：
1. 发布历史列表；
2. 单次发布记录详情；
3. 发布来源信息；
4. 发布时间；
5. 发布状态；
6. 对应 draft / changeset 引用；
7. 当前工作态是否晚于最近发布版本的提示。

## 4. MVP 不包含

MVP 不包含：
- 版本回滚；
- diff 对比；
- 多章节总览；
- 发布审批流；
- 自动重发。

## 5. 最小后端要求

若进入实现，后端最小要求应为：
- 提供章节维度发布历史只读查询能力；
- 提供单次发布记录最小来源字段（PublishRecord + ChapterDraft + ChangeSet + ReleaseReadiness 引用）；
- 提供“工作态晚于已发布态”判定所需最小状态字段，不引入复杂算法。

## 6. 最小前端要求

若进入实现，前端最小要求应为：
- 提供章节发布历史入口与列表展示；
- 提供单次发布详情查看；
- 提供“当前工作态晚于已发布版本”的显式提示文案；
- 保持与既有术语中文化口径一致。

## 7. 最小测试要求

若进入实现，最小测试要求应为：
- 发布历史查询接口基础回归；
- 发布来源字段完整性校验；
- 工作态晚于发布态提示逻辑回归；
- 不引入对 Gate / ChangeSet / Publish 核心语义的回归破坏。

## 8. 验收口径

MVP 验收以以下口径为准：
- 每次发布可见；
- 每次发布来源可追踪；
- 关键时间与状态可解释；
- 作者可判断是否需要重新发布。

## 9. 一句话口径

**这一阶段的 MVP，不是做完整版本管理系统，而是先让每次发布可见、可追踪、可解释。**
