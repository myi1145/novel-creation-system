## 3. 各文档定位说明

### 3.1 《小说多 Agent 系统最终架构设计文档（V1）》

定位：
整个系统的最高层定版文档，是所有后续子文档的总纲。

主要回答：
- 系统到底是什么
- 为什么不是单 Agent + RAG
- 为什么采用 Story Canon / Story State 总线
- 为什么要有结构化对象层、候选方案层、节奏控制器、多层质量闸门
- 系统的总体分层与 MVP 总范围是什么

适用场景：
- 对外介绍系统整体方案
- 内部统一架构方向
- 新成员理解项目全貌
- 作为其他文档的总引用源

---

### 3.2 《Story State / Canon State 状态模型设计（V1）》

定位：
系统真相中心的正式状态协议文档。

主要回答：
- Story State 和 Canon State 有什么区别
- 什么内容属于正史
- 什么内容只能停留在工作态 / 推演态
- Canon 如何写入、审批、回滚
- 哪些状态域必须纳入统一管理

适用场景：
- 状态总线设计
- 数据库状态域设计
- ChangeSet 机制实现
- 回滚 / 发布 / 正史更新流程设计

---

### 3.3 《结构化创作对象 Schema 设计（V1）》

定位：
创作对象层的正式对象模型文档。

主要回答：
- 系统中有哪些核心创作对象
- 角色卡、规则卡、章蓝图、场景卡、伏笔卡等对象怎么定义
- 公共字段和对象协议如何统一
- 对象如何版本化、引用、校验、关联状态变更

适用场景：
- Schema 建模
- API 输入输出定义
- 对象持久化设计
- Workflow 节点输入输出结构设计

---

### 3.4 《Agent 契约说明（V1）》

主要回答：
- Agent 在系统中到底是什么角色
- 不同 Agent 分别做什么、不做什么
- Agent 可以读什么，不能直接写什么
- Agent 输入输出协议怎么统一
- Agent 如何与 Workflow 协同

适用场景：
- Agent 模块开发
- Prompt 设计边界确认

## 5. 文档依赖关系

## 5.1 依赖主线

文档之间的依赖主线如下：

《最终架构设计文档》
→ 定义系统总目标、总分层、总原则
→ 派生《状态模型设计》
→ 派生《结构化创作对象 Schema 设计》
→ 派生《Agent 契约说明》
→ 派生《章节循环工作流说明》
→ 派生《多层质量闸门设计》
→ 派生《题材配置层与规则包设计》
→ 最终汇总到《MVP 开发任务拆解》

---

### 5.2 关键依赖说明

#### 1）状态模型依赖架构总纲
没有《最终架构设计文档》，就无法确定状态模型的边界。

#### 2）对象 Schema 依赖状态模型
因为对象层必须明确哪些字段会参与正史写入、哪些字段只属于工作态。

#### 3）Agent 契约依赖状态模型和对象层
因为 Agent 的输入输出、可读可写边界，必须建立在“状态怎么分层、对象怎么定义”的基础上。

#### 4）工作流依赖 Agent 契约与对象层
因为每一个节点都必须知道：
- 吃什么对象
- 产出什么对象
- 谁来执行
- 如何进入下一状态

#### 5）质量闸门依赖对象层、状态模型和工作流
因为闸门检查不是抽象评论，而是要检查：
- 是否违反 Canon
- 是否偏离章蓝图
- 是否人物失真
- 是否允许发布

#### 6）题材配置层依赖前面所有文档
因为题材扩展不是单独存在的，它必须能注入：
- 状态初始化
- 对象模板
- Agent 行为约束
- 工作流参数
- 闸门判定规则

#### 7）MVP 任务拆解依赖全部前置文档
因为它本质上是把前面所有定版文档翻译成工程任务。

## 7. 推荐的开发启动顺序

虽然文档已经全部定版，但真正开始开发时，不建议平均推进。

推荐顺序如下：

### 第一优先级
1. Story State / Canon State 落库模型
2. 结构化创作对象 Schema 与存储
3. 章节级 Workflow 主链路
4. ChangeSet / 审批 / Apply / Rollback 机制

### 第二优先级
5. 章蓝图候选生成与选择
6. Scene Decomposer
7. Writer Agent
8. Canon Gate + Style Gate

### 第三优先级
9. Genre Profile 加载与解析
10. Rule Pack 接入
11. Narrative Gate / Character Voice Gate
12. 基础控制台界面

### 第四优先级
13. 图谱派生索引
14. 高级节奏控制
15. 高级审阅分析
16. 多项目 / 多用户增强能力
