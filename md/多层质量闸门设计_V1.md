## 2. 文档目标

本文件用于正式定义系统中的“多层质量闸门”机制，明确：

1. 为什么质量控制不能退化为单一审稿 Agent
2. 系统中各层闸门的职责边界
3. 每个闸门的输入、输出、判定逻辑与失败处理方式
4. 闸门与工作流、Agent、Story State、ChangeSet 的衔接关系
5. 哪些问题属于硬阻断，哪些属于软阻断，哪些只形成提示
6. MVP 阶段必须实现的最小质量闭环

本文件的目标不是描述“怎么写得更好”的笼统经验，而是把质量控制正式纳入系统协议，使质量评审成为可解释、可追踪、可复盘、可迭代的工程能力。

---

## 3. 质量闸门的系统定位

### 3.1 质量闸门不是单一审稿环节

在本系统中，质量控制不是“正文写完之后，再让一个审稿 Agent 看一遍”的附属动作，而是贯穿：

- 候选方案筛选
- 场景拆解校验
- 正文草稿审查
- 修订回路判断
- ChangeSet 准入
- 发布准入

的分层控制体系。

因此，质量闸门的正式定位为：

**对创作对象、正文草稿、状态变更与发布行为进行分阶段、分职责、分严重级别控制的多层审查机制。**

### 3.2 质量闸门在系统中的位置

质量闸门位于：

**Workflow 主线与 Canon 写入之间。**

它既服务于内容质量，也服务于状态安全。

其核心作用包括：

1. 防止明显违背 Canon、角色逻辑、世界规则的内容进入正史链路
2. 防止结构塌陷、推进失衡、风格漂移直接发布
3. 把“是否允许继续流转”从纯主观判断变成可解释规则
4. 为 Rewriter、Planner、Human Reviewer 提供明确修订依据
5. 为 ChangeSet 提供可信来源，避免错误状态进入 Canon Snapshot

### 3.3 质量闸门与 Agent 的关系

质量闸门不是 Agent 本身。

更准确地说：

- 闸门是**制度与协议层**
- 审阅 Agent 是**执行器之一**
- 规则服务、指标计算器、Schema 校验器、人工审批也都是闸门执行单元

因此，闸门可以由以下机制组合执行：

- 硬规则引擎
- Schema 校验
- 指标计算器
- 审阅 Agent
- 人工复核

## 4. 总体设计原则

### 4.1 分层而非单点评审

不同问题的性质不同：

- 设定冲突属于事实一致性问题
- 节奏塌陷属于叙事推进问题
- 角色说话失真属于人物声音问题
- 用语漂移属于风格问题

因此，必须拆成多个职责清晰的闸门，而不是用一个“大评审 Prompt”兜底。

### 4.2 硬规则优先于主观判断

只要某问题可以结构化判定，就不应完全依赖 LLM 主观评价。

优先级应为：

1. Schema/字段校验
2. Canon 冲突检测
3. 指标/计数/约束校验
4. LLM 审阅判断
5. 人工最终裁决

### 4.3 闸门输出必须结构化

每个闸门的输出都必须是结构化结果，而不是一句“建议修改”。

至少包括：

- gate_name
- pass_status
- severity
- issue_list
- evidence_refs
- suggested_actions
- retry_recommendation

### 4.4 闸门不直接改正文

闸门负责：

- 发现问题
- 标定严重级别
- 给出修订建议
- 决定是否允许流转

闸门本身不直接覆盖原文，不直接篡改 Canon，也不直接批准发布。

### 4.5 闸门失败必须可分流

闸门失败后，不能只有“退回重写”这一种处理方式。

不同失败应有不同分流：

- 自动修订
- 局部重写
- 重新规划
- 人工确认后放行
- 终止本轮流程

### 4.6 闸门必须支持复盘

每次闸门执行都必须保留：

- 输入版本
- 输出结果
- 问题证据
- 修订前后差异
- 最终处置结论

从而支持后续：

- 错误分析
- Prompt 优化
- Agent 评估
- 规则升级

## 5. 闸门分层模型

V1 采用“三阶段、六闸门”的最小可用模型。

### 5.1 三阶段

#### 第一阶段：规划与对象准入阶段

审查对象：
- 候选章蓝图
- 场景卡
- 对象引用与依赖

目标：
- 尽可能在写正文之前拦住明显错误的方案

#### 第二阶段：正文草稿审查阶段

审查对象：
- 章节草稿
- 场景级文本片段
- 修订稿

目标：
- 控制文本质量、叙事完成度、角色偏移与风格漂移

#### 第三阶段：状态变更与发布准入阶段

审查对象：
- ChangeSet
- 发布记录
- Canon 写入候选变更

目标：
- 防止错误事实进入正史
- 防止未满足发布条件的章节被放行

### 5.2 六闸门

V1 正式定义以下六个核心闸门：

1. **Schema Gate**
2. **Canon Gate**
3. **Narrative Gate**
4. **Character Voice Gate**
5. **Style Gate**
6. **Publish Gate**

其中：

- Schema Gate、Canon Gate、Publish Gate 偏硬约束
- Narrative Gate、Character Voice Gate、Style Gate 偏软约束，但可上升为阻断

## 6. 严重级别模型

所有闸门输出问题统一采用四级严重度。

### S0：提示级

特征：
- 不影响流转
- 只作为优化建议

示例：
- 描写略显重复
- 某处情绪表达可更凝练

处置：
- 可直接放行
- 可附带给 Rewriter 参考

### S1：轻微问题

特征：
- 有质量瑕疵
- 不影响事实正确性
- 原则上允许继续流转

示例：
- 个别句式重复
- 某一场景转场略生硬
- 某句台词风格略偏离

处置：
- 允许放行
- 推荐局部修订

### S2：中等问题

特征：
- 已明显影响阅读体验或结构完成度
- 需要修订后再进入下一环节

示例：
- 本章推进不足
- 场景目标未达成
- 角色行为动机支撑不足
- 风格漂移较明显

处置：
- 默认不进入发布链路
- 进入修订回路

### S3：严重问题

特征：
- 直接威胁 Canon 正确性、章节合法性或发布安全

示例：
- 角色状态与 Canon 冲突
- 世界规则被突破
- 时间线错乱
- ChangeSet 错提关键事实
- 发布条件不满足

处置：
- 强制阻断
- 必须修复或人工特批

## 7. 统一输出协议

所有闸门必须输出统一结构。

```json
{
  "gate_name": "canon_gate",
  "target_type": "chapter_draft",
  "target_ref": "chapter_012_draft_v3",
  "pass_status": "fail",
  "highest_severity": "S3",
  "issues": [
    {
      "issue_id": "CG-001",
      "severity": "S3",
      "category": "timeline_conflict",
      "summary": "角色在未离开宗门的前提下出现在边境战场",
      "evidence_refs": ["draft:para_18", "canon:event_094"],
      "suggested_actions": ["回退该场景设定", "重新生成此处场景"]
    }
  ],
  "recommended_route": "rewrite_scene",
  "can_override": true,
  "override_role": ["author", "chief_editor"],
  "generated_at": "2026-03-26T13:00:00+08:00"
}
```

统一字段说明：

- `gate_name`：闸门名称
- `target_type`：审查对象类型
- `target_ref`：审查对象引用
- `pass_status`：pass / warn / fail
- `highest_severity`：当前最高问题级别
- `issues`：问题列表
- `recommended_route`：建议流转路径
- `can_override`：是否允许人工特批
- `override_role`：可特批角色
- `generated_at`：执行时间

## 8. Schema Gate

### 8.1 定位

Schema Gate 是所有闸门中的第一道闸门，用于检查对象与结构化输出是否符合约定 Schema。

### 8.2 审查对象

- 候选章蓝图
- SceneCard
- ReviewResultCard
- ChangeProposalCard
- ChangeSet 提取结果
- 各 Agent 的结构化输出

### 8.3 核心目标

1. 检查字段是否缺失
2. 检查字段类型是否正确
3. 检查枚举值是否合法
4. 检查对象引用是否完整
5. 检查最小约束是否满足

### 8.4 输入

- 目标对象 JSON / 结构化输出
- 对应 Schema 定义
- 对象引用索引

### 8.5 输出

- pass / fail
- 缺失字段清单
- 非法值清单
- 无效引用清单

### 8.6 典型阻断条件

以下情况直接判定为 S3：

- 关键字段缺失
- 对象类型错误
- 枚举字段非法
- 引用对象不存在
- 输出格式不可解析

### 8.7 失败分流

- 可自动补齐默认值的：进入自动修复
- 无法自动修复的：退回生成节点重新生成

### 8.8 V1 实现要求

V1 必须以程序规则执行，不依赖 LLM 主观判断。

---

## 9. Canon Gate

### 9.1 定位

Canon Gate 是事实一致性闸门，用于检查内容是否与当前 Canon Snapshot 冲突。

### 9.2 审查对象

- 候选章蓝图
- 场景卡
- 正文草稿
- ChangeSet

### 9.3 核心目标

1. 检查角色状态是否冲突
2. 检查关系状态是否冲突
3. 检查时间线顺序是否冲突
4. 检查地点与移动逻辑是否冲突
5. 检查世界规则是否被违反
6. 检查术语、能力、势力设定是否漂移

### 9.4 输入

- 审查目标对象或正文
- 当前 Canon Snapshot
- 相关 CharacterCard / RuleCard / TimelineEvent / RelationshipEdge / OpenLoopCard
- 最近章节摘要

### 9.5 输出

- 冲突列表
- 冲突类型
- 冲突证据
- 建议修复策略

### 9.6 典型问题类别

- `character_state_conflict`
- `relationship_stage_conflict`
- `timeline_conflict`
- `location_conflict`
- `world_rule_violation`
- `terminology_drift`
- `open_loop_misuse`

### 9.7 阻断规则

默认以下问题判定为 S3：

- 明确违背已存在 Canon 事实
- 世界硬规则被突破
- 死亡/失踪/身份变更等重大状态冲突
- 时间线无法自洽
- ChangeSet 提议写入错误事实

以下问题通常判为 S2：

- 细节描述与 Canon 边缘偏移
- 新增术语定义不充分
- 伏笔提醒方式不合理但未冲突

### 9.8 特批规则

Canon Gate 的 S3 问题原则上不放行，但以下情况允许人工特批：

1. 作者决定正式改设定
2. 本轮章节属于 Canon 修订型章节
3. 冲突本身是“剧情反转揭示”而非系统误判

特批时必须：

- 同步生成 ChangeProposal
- 经人工审批后进入 Canon 修订链路
- 不得绕过日志记录

## 10. Narrative Gate

### 10.1 定位

Narrative Gate 是叙事结构闸门，用于判断本章是否完成应完成的推进任务。

### 10.2 审查对象

- 候选章蓝图
- SceneCard 序列
- 正文草稿
- 修订稿

### 10.3 核心目标

1. 检查本章是否完成章目标
2. 检查场景顺序是否合理
3. 检查冲突是否建立与推进
4. 检查信息揭示量是否合适
5. 检查结尾钩子是否达标
6. 检查是否存在明显灌水、跳跃、塌陷

### 10.4 输入

- 章蓝图
- 场景卡序列
- 正文草稿
- 节奏约束卡
- 上章摘要 / 当前卷目标

### 10.5 输出

- 推进完成度评分
- 场景有效性报告
- 缺失节点报告
- 冗余节点报告
- 修订建议

### 10.6 典型问题类别

- `goal_not_achieved`
- `scene_transition_break`
- `conflict_weak`
- `hook_missing`
- `reveal_density_imbalance`
- `padding_detected`
- `pace_mismatch`

### 10.7 判定逻辑

Narrative Gate 不追求文学批评式的“好不好看”，而是优先检查“是否完成既定叙事任务”。

最低通过标准：

1. 本章主任务明确
2. 本章至少完成一个有效推进
3. 场景链条连续
4. 信息投放没有严重失衡
5. 结尾具有明确承接或钩子

### 10.8 失败分流

- 轻微失衡：进入局部修订
- 目标未达成：退回 Rewriter
- 章蓝图本身有问题：退回 Planner 重做候选方案

## 11. Character Voice Gate

### 11.1 定位

Character Voice Gate 用于控制人物语言、行为、情绪反应与身份逻辑的一致性。

### 11.2 审查对象

- 对话片段
- 场景文本
- 章节草稿

### 11.3 核心目标

1. 检查人物说话风格是否偏移
2. 检查人物行为是否符合当前动机与关系阶段
3. 检查人物情绪反应是否与上下文一致
4. 检查是否出现“作者替角色说话”的失真

### 11.4 输入

- CharacterCard
- ArcCard
- RelationshipEdge
- 历史台词样本
- 正文草稿

### 11.5 输出

- 人物偏移点清单
- 台词失真清单
- 行为动机不足清单
- 建议修订方向

### 11.6 典型问题类别

- `voice_drift`
- `motivation_gap`
- `emotion_mismatch`
- `relationship_stage_mismatch`
- `authorial_override`

### 11.7 阻断原则

一般情况下，Character Voice Gate 以 S1/S2 为主。

但以下情况可升为 S3：

- 主角核心人设被明显破坏
- 关键关系阶段被无依据跳变
- 角色做出与已知动机完全相反的重大行为，且正文无解释

### 11.8 失败分流

- 台词问题：局部重写台词
- 行为动机问题：补写桥接段落
- 大面积角色偏移：退回整章修订

---

## 12. Style Gate

### 12.1 定位

Style Gate 用于检查文风、语体、表达重复、题材语感与整体稳定性。

### 12.2 审查对象

- 正文草稿
- 修订稿
- 局部重写结果

### 12.3 核心目标

1. 检查是否符合当前题材风格包
2. 检查用语是否出现明显串题材
3. 检查语言重复与模板痕迹
4. 检查叙述口吻是否稳定
5. 检查是否存在明显现代词、游戏词、网文化表达误入不适合题材的上下文

### 12.4 输入

- 题材配置
- 风格约束卡
- 历史风格样本
- 正文草稿

### 12.5 输出

- 风格偏差报告
- 重复表达报告
- 不当词汇清单
- 语体失真说明

### 12.6 典型问题类别

- `genre_style_drift`
- `repetition_high`
- `template_smell`
- `lexicon_mismatch`
- `tone_instability`

### 12.7 判定原则

Style Gate 默认不以“文学审美高低”为目标，而以“是否稳定、是否合题材、是否不出戏”为目标。

### 12.8 阻断规则

默认以下情况升级为 S2 或以上：

- 题材语体明显串线
- 重复表达密度过高
- 模板化痕迹影响阅读
- 不当词汇明显破坏沉浸感

---

## 13. Publish Gate

### 13.1 定位

Publish Gate 是发布准入闸门，用于决定章节是否允许进入“Apply ChangeSet + 写入 Canon + 标记已发布”的链路。

### 13.2 审查对象

- 最终章节稿
- 各闸门结果汇总
- ChangeSet
- 发布记录草案

### 13.3 核心目标

1. 检查前置闸门是否达到发布阈值
2. 检查是否存在未处理的高严重问题
3. 检查 ChangeSet 是否已审批
4. 检查章节元数据是否完整
5. 检查是否具备发布必要条件

### 13.4 输入

- 最终草稿版本
- Schema Gate / Canon Gate / Narrative Gate / Character Voice Gate / Style Gate 结果
- ChangeSet
- 审批状态
- 发布元数据

### 13.5 输出

- allow_publish / deny_publish
- 阻断原因清单
- 是否可人工特批

### 13.6 最低发布条件

章节必须同时满足：

1. Schema Gate 已通过
2. Canon Gate 不存在未处理 S3
3. Narrative Gate 不存在未处理 S2/S3
4. Character Voice Gate 不存在未处理 S3
5. Style Gate 不存在未处理 S2/S3 的必修项
6. ChangeSet 已生成
7. ChangeSet 已审批
8. 发布元数据完整

### 13.7 特批机制

只有以下角色可执行发布特批：

- 作者
- 主编 / 项目负责人

特批发布必须写入：

- 特批原因
- 特批人
- 特批时间
- 后续补修要求

## 14. 闸门执行顺序

### 14.1 章节主流程中的默认顺序

V1 推荐默认顺序为：

1. Schema Gate（对象/输出结构）
2. Canon Gate（事实一致性）
3. Narrative Gate（叙事完成度）
4. Character Voice Gate（人物声音）
5. Style Gate（风格稳定性）
6. Publish Gate（最终发布准入）

### 14.2 顺序原则

- 先判结构合法，再判事实合法
- 先排除硬错误，再优化软质量
- 先检查是否能成立，再检查是否够好

### 14.3 快速失败机制

若 Schema Gate 或 Canon Gate 出现明确 S3：

- 可直接中止后续闸门
- 或只保留最必要的辅助审查
- 避免无意义的重复评审消耗

---

## 15. 闸门与修订回路

### 15.1 路由类型

闸门失败后，系统必须输出建议路由：

- `auto_fix`
- `rewrite_scene`
- `rewrite_chapter`
- `replan_blueprint`
- `human_review`
- `manual_override`
- `terminate_run`

### 15.2 修订优先级

若多个闸门同时失败，按以下顺序处理：

1. Schema 问题
2. Canon 问题
3. Narrative 问题
4. Character Voice 问题
5. Style 问题

原因：
- 前者通常是后者的前提
- 先修风格后修设定容易返工

### 15.3 修订次数上限

V1 推荐：

- 单场景自动修订上限：2 次
- 单章节自动修订上限：2 次
- 超出上限后必须进入人工审查或退回重规划

---

## 16. 闸门与 ChangeSet 的关系

### 16.1 闸门先于 ChangeSet Apply

任何 ChangeSet 在写入 Canon 之前，必须至少满足：

- 来源章节已完成基础闸门
- ChangeSet 自身通过 Schema Gate
- ChangeSet 自身通过 Canon Gate

### 16.2 ChangeSet 专项审查

对于 ChangeSet，重点检查：

1. 提取的事实是否真的出现在章节中
2. 事实归类是否正确
3. 事实变更粒度是否合适
4. 是否误把推测写成事实
5. 是否误把暂态写成 Canon 永久状态

### 16.3 不允许直接从正文跳到 Canon

系统中禁止：

- 未经 ChangeSet 直接修改 Canon Snapshot
- 由 Writer Agent 直接写入正史状态
- 用“全文向量检索结果”直接替代 Canon 写入依据

---

## 17. 人工介入策略

### 17.1 必须人工可见的场景

以下情况必须进入人工可见状态：

1. Canon Gate 出现 S3
2. Publish Gate 拒绝发布
3. 多次自动修订失败
4. ChangeSet 涉及重大设定修改
5. 关键人物弧光发生重大跳变

### 17.2 可人工特批但必须留痕的场景

- 风格问题决定暂时带病发布
- 叙事节奏问题作者接受
- Canon 冲突实际上是有意埋设反转
- ChangeSet 属于作者主动改设定

### 17.3 人工权限边界

人工可以：

- 特批
- 驳回
- 改写对象
- 发起 Canon 修订

人工也必须遵守：

- 重大修改走 ChangeSet
- 不得跳过日志
- 不得静默改 Canon Snapshot

## 18. 监控与可观测性要求

V1 阶段，闸门系统至少要记录以下指标：

1. 各闸门调用次数
2. 各闸门失败率
3. 各类问题分布
4. 自动修订成功率
5. 人工特批占比
6. 每章节平均修订轮数
7. Canon 冲突高发类型
8. 风格漂移高发类型

这些指标用于：

- 判断哪个 Agent 最不稳定
- 判断哪个题材包最容易出问题
- 判断哪些规则需要补强
- 判断流程是否过度复杂

## 20. 关键风险与控制策略

### 20.1 风险一：闸门退化成大 Prompt

表现：
- 所有问题都丢给一个评审 Agent
- 输出不可解释
- 修订依据模糊

控制：
- 分层
- 统一结构化输出
- 规则与 LLM 混合执行

### 20.2 风险二：闸门过多导致流程过重

表现：
- 每章评审成本过高
- 自动修订链条过长
- 作者等待感明显

控制：
- V1 只保留核心闸门
- 快速失败
- 关键问题优先

### 20.3 风险三：风格问题挤占事实问题优先级

表现：
- 文风改得很顺，但 Canon 已经错了

控制：
- 固定修订优先级
- 硬问题先修

### 20.4 风险四：人工特批滥用

表现：
- 闸门形同虚设
- 正史逐步污染

控制：
- 特批留痕
- 特批原因强制记录
- 关键设定修改强制二次确认

---

## 21. V1 结论

本系统的质量控制正式采用：

**三阶段、六闸门、统一严重度、统一输出协议、可分流修订、可人工特批但必须留痕** 的设计方案。

其核心原则不是“尽量多审”，而是：

- 先挡住结构错误
- 再挡住事实错误
- 再控制叙事完成度
- 再控制人物与风格稳定性
- 最后再决定是否允许发布

由此，质量闸门不再是一个模糊的“审稿动作”，而成为系统控制中枢的一部分，负责确保：

- 错误内容不轻易进入正史
- 低质量章节不直接发布
- 修订有依据
- 审批有凭证
- 复盘有数据
