# 01_蓝图级人工修订方向确认

## 1. 文档目的

本文件用于对下一阶段主方向进行正式拍板确认，并作为后续 `02 / 03 / 04` 文档的统一上游约束。

本文件是方向确认件，不是执行工单。

## 2. 结论先行

下一阶段主方向正式确认为：**蓝图级人工修订立项优先。**

一句话结论：
**先在立项层完成 `ChapterBlueprint` 人工修订方向、边界与 MVP 收敛，再决定是否进入实现，不直接跳转蓝图编辑器开发。**

## 3. 为什么优先蓝图级人工修订

### 3.1 与主链结构最一致

系统主链为“目标 -> 蓝图 -> 场景 -> 草稿 -> Gate -> ChangeSet -> Publish”。

蓝图位于正文前的关键决策层，在该层建立人工修订能力，符合“先上游纠偏、后下游执行”的控制思路。

### 3.2 对返工成本控制最直接

相较在草稿后期修订，蓝图层修订可更早发现结构偏差，降低后续场景与正文重做成本。

### 3.3 与已签收阶段形成自然递进

文本级人工修订已完成后，下一步应上移到更高杠杆层，而非继续在同层重复增强。

## 4. 为什么不是先做场景级人工修订

- 场景级颗粒更细，实施复杂度、协同成本与范围管理难度更高。
- 在蓝图层边界尚未稳定前直接切入场景层，容易出现“中层缺位、底层过细”的结构性失衡。
- 当前阶段目标是立项收敛，不适合直接进入更细粒度能力。

## 5. 为什么不是先扩结构化设定卡层

- 设定卡扩展属于长期价值方向，但对“修订控制点上移”命中度不足。
- 当前优先事项是增强章节推进的人工控制链路，不是扩展对象面宽。
- 若与蓝图级立项并行推进，将提高口径漂移风险。

## 6. 方向确认后的阶段含义

- 下一阶段以“蓝图级人工修订立项”作为唯一主任务口径。
- 产出重点是边界、MVP 与轮次规划，不产出实现工单。
- 仅在立项文档链稳定后，才可进入后续是否开发的决策。

## 7. 最终确认结论

### 主方向

**蓝图级人工修订立项优先。**

### 后置方向

- 场景级人工修订后置。
- 结构化设定卡扩展后置。

### 当前不采用策略

**不采用“蓝图级 + 场景级 + 扩设定卡”并行推进策略。**
