# 03_轻量版本差异对比与重发决策MVP范围

## 1. 文档目的

定义“轻量版本差异对比与重发决策”阶段若进入实现时的最小 MVP 范围，确保可执行且可验收。

## 2. MVP 结论先行

MVP 仅覆盖“轻量差异摘要 + 重发辅助决策”，不扩展为完整版本管理系统。

## 3. MVP 必须覆盖

MVP 必须覆盖以下能力：
1. 当前工作态与最近发布版的轻量差异摘要；
2. 差异类别与影响等级的最小结构化表达；
3. 重发建议分级提示（建议重发 / 可暂缓 / 需人工判断）；
4. 重发前重点核查项提示；
5. 建议结果的解释性字段（来源依据最小可追溯）。

## 4. MVP 不包含

MVP 不包含：
- 文本逐段 diff 视图；
- 版本回滚；
- 自动重发；
- Publish / ChangeSet 主链重构；
- 新审批流。

## 5. 最小后端要求（进入实现时）

若进入实现，后端最小要求应为：
- 提供章节维度“工作态 vs 最近发布版”差异摘要只读查询；
- 返回最小差异分类与影响等级字段；
- 返回重发建议与解释性依据字段；
- 不改变 Publish / ChangeSet 写入与执行语义。

## 6. 最小前端要求（进入实现时）

若进入实现，前端最小要求应为：
- 在章节发布相关页面展示轻量差异摘要；
- 显示重发建议分级与简要解释；
- 显示重发前重点核查提示；
- 保持作者端术语中文化口径一致。

## 7. 最小测试要求（进入实现时）

若进入实现，最小测试要求应为：
- 差异摘要返回结构完整性回归；
- 重发建议分级逻辑回归；
- 解释性字段可追溯性回归；
- 不引入 Publish / ChangeSet / Gate 语义回归破坏。

## 8. 验收口径

MVP 验收以以下口径为准：
- 作者能快速理解“改了什么”；
- 作者能获得“是否建议重发”的稳定辅助；
- 建议结果有最小解释依据；
- 现有主链语义无破坏。

## 9. 一句话口径

**这一阶段的 MVP，不是做复杂版本管理，而是先把“差异可理解、重发可辅助决策”落地为最小闭环。**
