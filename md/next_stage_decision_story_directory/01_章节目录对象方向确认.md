# 01_章节目录对象方向确认

## 1. 文档目的

基于候选方向对比结果完成阶段拍板，明确当前阶段主方向、对象位置关系与暂不采用方向，避免执行偏差。

## 2. 结论先行

当前阶段主方向确认为：**建立 StoryDirectory / ChapterDirectory 对象**，并以 docs-only 方式先完成边界立项。

## 3. 主方向

主方向定义如下：

1. 在 StoryPlanning 之后增加 StoryDirectory / ChapterDirectory 对象层；
2. 目录层负责将全书意图落到章节序列与章节职责；
3. 在目录层稳定后，再进入卡槽候选生成与章节蓝图挂接实现。

## 4. 该对象的位置

StoryDirectory / ChapterDirectory 的位置必须固定为：

1. 位于 StoryPlanning 之后；
2. 位于卡槽候选生成之前；
3. 位于 ChapterGoal / ChapterBlueprint 之前。

## 5. 暂不采用方向

当前阶段明确不采用以下方向：

1. 继续直接生成卡槽候选；
2. 只做长文本目录；
3. 直接重写章节蓝图生成；
4. 自动写 Canon；
5. 自动生成 ChangeSet。

## 6. 为什么现在做这个方向

1. 现有系统已具备章节主链闭环，当前主要短板转为上游章节职责分配层缺失；
2. 继续直接推卡槽候选会放大“有种子、无章节落点”的偏差；
3. 目录层是连接全书意图与逐章执行的必要桥梁，先补目录层可显著降低后续返工。

## 7. 与 StoryPlanning 的关系

1. StoryPlanning 负责全书级目标、主线与分卷阶段意图；
2. StoryDirectory 负责将上述意图细化为可引用的章节序列职责；
3. StoryDirectory 不替代 StoryPlanning，而是将其结构化下沉到章节级执行入口。

## 8. 与卡槽候选的关系

1. 卡槽候选应由 StoryPlanning 与 StoryDirectory 共同驱动；
2. StoryDirectory 提供章节覆盖范围、优先级与落地章节上下文；
3. 无目录约束的卡槽候选应视为高风险输入，不作为下一阶段主路径。

## 9. 与 ChapterGoal / ChapterBlueprint 的关系

1. StoryDirectory 是 ChapterGoal 的章节职责上游输入；
2. StoryDirectory 是 ChapterBlueprint 的章节推进约束上游输入；
3. 目录层应先于 ChapterGoal / ChapterBlueprint 接入实现，避免孤立蓝图生成。

## 10. 与 Canon / Story State / ChangeSet 的关系

1. StoryDirectory 在本阶段仅作为 Story State 工作态对象边界定义；
2. 本阶段不直接写 Canon；
3. Canon 仍仅通过 ChangeSet 写入；
4. 当目录中的事实被确认为必须长期遵守的正史约束时，后续才可进入 ChangeSet 评审链。

## 11. 一句话拍板

**下一阶段先把全书规划落到可引用的章节目录对象，再让卡槽候选和章节蓝图基于目录推进。**
