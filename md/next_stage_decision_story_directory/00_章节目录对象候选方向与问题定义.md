# 00_章节目录对象候选方向与问题定义

## 1. 文档目的

在“全书级规划对象最小闭环”与“四类基础卡槽导入导出”已完成的前提下，重新识别上游主线缺口，明确是否应继续推进“全书规划直接生成卡槽候选”，并为下一阶段确立可执行的问题定义。

## 2. 当前系统能力回顾

基于现有阶段文档与交接口径，系统已形成以下能力基础：

1. 全书级规划对象最小闭环已具备；
2. 四类基础卡槽（角色卡、术语卡、势力卡、地点卡）已具备导入导出与最小维护能力；
3. 章节主链可执行并保持 V1 主链稳定：目标 -> 蓝图 -> 场景 -> 草稿 -> Gate -> ChangeSet -> Publish；
4. Canon 仍遵循“仅可通过 ChangeSet 写入”的中枢约束。

上述能力已为下一阶段提供工程基础，但仍未解决“章节级职责分配层”缺失问题。

## 3. 上游主线重审报告结论摘要

依据 `md/research/2026-04-14_系统级重新审查报告.md` 以及当前阶段重审结论，需吸收以下共识：

1. 直接执行“全书规划 -> 卡槽候选”应暂停或改写；
2. 必须先补“全书章节目录 / 分卷章节规划”层；
3. 章节目录应成为 ChapterGoal / ChapterBlueprint 的直接上游依据之一；
4. 卡槽候选应由“全书规划 + 章节目录”共同驱动；
5. 当前不应继续扩新卡型、复杂图谱、导入导出扩展或自动抽取主线。

## 4. 当前上游缺口

当前缺口不是“没有卡槽”，而是“缺少把全书意图落到章节序列的对象层”。具体表现为：

1. StoryPlanning 定义了全书写作方向，但尚未给出逐章职责分配；
2. ChapterGoal 缺少稳定、可引用的章节目录约束；
3. ChapterBlueprint 缺少“本章在全书中的角色”上游输入；
4. 卡槽候选缺少章节覆盖范围与章节优先级约束，易变成脱离章节职责的孤立种子。

## 5. 为什么仅有全书规划不够

仅有 StoryPlanning 仍不足以支持高质量上游驱动，核心原因如下：

1. StoryPlanning 解决“整本书写什么”，不直接回答“每章承担什么职责”；
2. 章节主链需要逐章约束输入，缺失章节目录时会退化为局部承接；
3. 卡槽候选若不绑定章节序列，会出现覆盖失衡、出场节奏失衡与设定落地点不清；
4. 章节蓝图若无目录锚点，难以稳定对齐卷阶段推进与伏笔开闭节奏。

## 6. 候选方向对比

### 6.1 继续直接生成卡槽候选

- 优点：短期推进快；
- 问题：无法回答章节级职责，卡槽候选仍可能脱离章节落地；
- 结论：本阶段不采用。

### 6.2 只在 StoryPlanning 中写长文本目录

- 优点：实现成本低；
- 问题：不可结构化引用、难与 ChapterGoal/ChapterBlueprint 建立稳定契约；
- 结论：可作为补充，不可作为主方向。

### 6.3 新增 StoryDirectory / ChapterDirectory 对象

- 优点：可将全书规划落到章节序列，形成逐章职责、推进目标与实体覆盖约束；
- 风险：需先做边界立项，防止与 Canon/ChangeSet 关系混乱；
- 结论：本阶段建议优先方向。

### 6.4 直接接入章节蓝图

- 优点：可快速影响下游；
- 问题：上游对象未定时直接接入将放大输入不稳定；
- 结论：当前不应先做。

### 6.5 重写整个章节主链

- 优点：理论上可一次性重排；
- 问题：违背 V1 增量演进原则，风险与成本不可控；
- 结论：明确不采用。

## 7. 本阶段建议优先方向

**优先方向：建立 StoryDirectory / ChapterDirectory 对象的 docs-only 立项链路。**

阶段口径：先明确对象边界、输入关系与轮次规划，再决定后续是否进入实现。

## 8. 本阶段暂不优先方向

1. 继续直接执行“全书规划生成卡槽候选”；
2. 仅以长文本目录替代对象化目录；
3. 在对象边界未定前直接接入 ChapterBlueprint；
4. 自动写 Canon；
5. 自动生成 ChangeSet；
6. 扩展新卡型、复杂图谱与自动抽取主线。

## 9. 最终问题定义

**当前最需要解决的是：把全书规划落到章节序列，明确每一章的职责、推进目标、关键出场实体和必须落地的设定点。**

为此，下一阶段应先建立 StoryDirectory / ChapterDirectory 的对象化边界，使其成为 StoryPlanning 与卡槽候选、ChapterGoal、ChapterBlueprint 之间的桥梁层。
