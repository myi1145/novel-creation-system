# 01_结构化设定卡槽体系方向确认

## 1. 文档目的

在候选方向对比基础上，完成下一阶段主方向拍板，统一团队口径，避免在“全量卡槽 / 图谱化 / 自动抽取”之间反复摇摆。

## 2. 结论先行

下一阶段主方向确认为：**结构化设定卡槽体系**。

本阶段为 docs-only 立项，不进入实现开发。

## 3. 主方向

**结构化设定卡槽体系。**

方向定义：在不改 Canon / Story State 中枢语义、不绕过 ChangeSet 的前提下，建立核心设定资产的结构化承载与审计边界。

## 4. 第一批 MVP 方向

第一批仅包含：

- 角色卡
- 术语卡

不扩展其他卡槽类型。

## 5. 暂不采用方向

当前阶段明确暂不采用：

- 一次性全量卡槽；
- 复杂知识图谱；
- 关系图谱大屏；
- 自动全局抽取入库；
- 卡片自动改写主线。

## 6. 为什么现在做这个方向

1. 作者端主路径体验已完成一轮收口，阶段主线应从“流程可用”转向“核心资产可治理”；
2. 角色与术语是跨章节一致性最常见且最高频的失真源；
3. 先做最小结构化卡槽，有助于减少后续章节生成对临时上下文的依赖；
4. docs-only 立项可先锁定边界，避免直接开发导致架构漂移。

## 7. 与 Canon / Story State / ChangeSet 的关系

- 卡槽体系属于 Story State 中“结构化创作对象治理层”的增强，不替代中枢；
- Canon 仍是正史真相中心，卡槽中的“进入 Canon”状态必须受审计；
- 任何可影响正史的卡槽变更，必须通过 ChangeSet 提案、审批、应用链路入史；
- 卡槽不应绕过主链，不应直接写 Canon，不应让 Agent 成为中枢。

## 8. 一句话拍板

**下一阶段先把角色与术语这两类最核心设定资产结构化，形成可读、可改、可审计、可复用的卡槽底座。**
