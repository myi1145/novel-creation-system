- 适用范围：小说多 Agent 创作控制系统 V1 全部后端服务、工作流编排、对象层、状态层、API 层、存储层

## 2. 文档目标

本文件用于解决以下问题：

1. 架构文档中的概念名与代码实现中的字段名不统一
2. 不同模块对同一对象使用不同命名，导致对象漂移
3. API、数据库、缓存、日志、工作流上下文的字段表达不一致
4. 中英文混用、缩写滥用、临时字段泛滥，导致后续维护成本过高

因此，V1 阶段必须形成统一的命名与映射规则：

- **业务名称统一**：一个概念在系统内只能有一个正式主名
- **代码名称统一**：正式主名必须映射到稳定的英文标识
- **分层字段统一**：对象字段在 API、服务、存储、日志层必须遵守同一命名标准
- **映射关系明确**：允许展示名、存储名、日志名不同，但必须有明确映射
- **禁止随意起名**：未经本表登记的核心对象名与核心字段名，不应进入主干代码

## 3. 总体命名原则

### 3.1 命名分层原则

系统中的命名分为四层：

1. **业务中文名**
   - 面向文档、策划、产品、作者界面
   - 例如：角色卡、章蓝图、伏笔卡、变更集

2. **领域英文主名**
   - 面向代码、Schema、API、日志、数据库
   - 例如：`character_card`、`chapter_blueprint`、`open_loop_card`、`change_set`

3. **类型标识名**
   - 用于对象类型、事件类型、任务类型、节点类型
   - 例如：`CharacterCard`、`ChapterBlueprint`、`OpenLoopCard`、`ChangeSet`

4. **展示别名**
   - 仅用于前端显示、文档行文、提示词展示
   - 不得作为代码主字段使用

### 3.2 字段命名风格

V1 统一采用以下规则：

- 数据库字段：`snake_case`
- JSON / API 字段：`snake_case`
- Python 变量：`snake_case`
- Python 类名 / Pydantic 模型名：`PascalCase`
- 枚举值：`snake_case`
- 常量：`UPPER_SNAKE_CASE`
- 事件类型：`dot.case` 或 `snake_case` 二选一，V1 推荐 `snake_case`

### 3.3 时间与标识字段统一规则

- 主键统一为：`xxx_id`
- 外键统一为：`xxx_id`
- 时间统一为：`xxx_at`
- 布尔字段统一以：`is_` / `has_` / `can_` 开头
- 计数字段统一以：`xxx_count`
- 列表字段统一使用复数名或 `_list`，V1 推荐语义复数
- 状态字段统一以：`status`
- 阶段字段统一以：`phase` 或 `stage`
- 版本字段统一以：`version`
- 引用列表统一以：`refs`

### 3.4 禁止事项

禁止使用以下命名方式进入核心代码：

- 模糊名：`data`、`info`、`item`、`content2`、`temp`、`misc`
- 无语义缩写：`obj`、`ctx2`、`cfgx`
- 同义混用：`chapter_plan` / `chapter_blueprint` 并存
- 中英混合：`chapter_章节状态`
- 一义多名：同一对象既叫 `foreshadow` 又叫 `open_loop`
- 结构字段与展示字段混用：例如把 `title` 同时当对象名、显示标题、场景类型名

### 6.1 关于 `id` 与 `xxx_id`

V1 推荐策略：

- 面向数据库主键，可统一提供 `id`
- 面向领域对象与 API，必须同时明确领域 ID，如 `character_id`
- 若只能保留一种，则领域核心对象优先使用 `xxx_id`

## 15. V1 实施要求

### 15.1 开发要求

1. 核心对象代码命名必须以本表为准
2. API 入参与返回字段必须遵守本表
3. 数据库表名与字段名优先遵守本表
4. 日志中的关键字段必须与对象主字段同名
5. 工作流上下文中的引用字段必须使用标准 ID 名称

### 15.2 文档要求

后续所有实现文档、接口文档、数据库设计文档，都必须引用本表中的正式命名。

### 15.3 演进要求

V1 之后若需要新增核心对象、核心字段、核心枚举，必须通过统一修订：

- 新增项必须补入本表
- 禁止只改代码不改命名总表
- 禁止同义项并行存在超过一个版本周期

## 17. 结论

《字段命名与对象映射总表（V1）》不是补充性附件，而是系统从“概念设计”进入“实现设计”的关键桥梁。

本文件的正式作用是：

- 给 V1 所有核心概念提供唯一正式命名
- 给对象层、状态层、工作流层、Agent 层提供统一字段语言
- 防止后续开发出现同义漂移与命名失控
- 为 API、数据库、日志、索引、测试用例提供一致基线

V1 阶段，凡属于核心对象、核心状态、核心流程、核心审核机制的命名与字段，均应以本表为准。
